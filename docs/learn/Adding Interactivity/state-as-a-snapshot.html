스냅샷처럼 상태 사용하기

상태 변수들은 JavaScript의 평범한 변수들처럼 보이기도 합니다. 읽고 쓸 수 있죠. 그치만 상태들은 조금 더 스냅샷처럼 동작합니다. 이 값을 단순히 재설정(set)하는 것은 이미 있는 상태를 변경하지는 않습니다. 대신 리렌더링을 발생시킵니다.

여러분이 배울 것들
  • 어떻게 상태를 변경하는게 리렌더링을 발생시키는지
  • 언제, 그리고 어떻게 상태를 변경하는지
  • 왜 상태의 업데이트가 바꾼 직후에 발생하지 않는지
  • 어떻게 이벤트 핸들러들이 상태의 "스냅샷"에 접근하는지

상태를 변경해 렌더링 일으키기​

아마 여러분은 클릭같은 유저 이벤트가 발생했을 때 즉시 유저 인터페이스가 변경된다고 생각할 것입니다. React에서는 이 생각과 살짝 다르게 동작합니다. 이전 페이지에서 상태를 설정하는게 React에게 리렌더링을 요청한다는 것을 배웠습니다. 이 말은 인터페이스가 이벤트에 반응하기 위해서는 상태를 업데이트 해야 합니다.

예를 들어 만약 "Send" 버튼을 클릭한 경우, setIsSent(true)는 React에게 UI를 다시 그리라고 명령합니다.

아래는 버튼을 클릭했을 때 일어나는 과정입니다.

  1. onSubmit 이벤트 핸들러가 호출됩니다.
  2. setIsSent(true)는 isSent를 true로 바꾸고 새로운 렌더링을 예약합니다.
  3. React는 업데이트된 isSent 값에 따라 리렌더링 됩니다.

이제 상태와 렌더링의 관계에 대해 더 자세히 알아봅시다!

렌더링은 그 시간에 맞춰 스냅샷을 찍습니다​

"렌더링"은 React가 컴포넌트 함수를 호출함을 의미합니다. 컴포넌트 함수가 반환하는 JSX는 어떤 시간에 촬영한 스냅샷과 같습니다. 컴포넌트에 포함된 props, 이벤트 핸들러, 지역변수들은 모두 렌더링 되는 당시의 상태에 기반해 계산됩니다.

사진이나 영화 프레임들과는 달리, UI "스냅샷"은 상호작용이 가능합니다. 스냅샷은 입력값에 따라 어떤 반응을 할지 명시하는 이벤트 핸들러같은 로직을 포함하고 있습니다. React는 이 스냅샷과 일치시키기 위해 화면을 업데이트하고 이벤트 핸들러들을 연결합니다. 결국 버튼을 누르는 것면 JSX의 클릭 핸들러를 호출하게 됩니다.

React가 컴포넌트를 리렌더링 할때는

  1. React가 함수(컴포넌트)를 다시 호출합니다.
  2. 함수(컴포넌트)는 새로운 JSX 스냅샷을 반환합니다.
  3. React는 그 새로운 스냅샷으로 화면을 업데이트합니다.

리렌더링​

컴포넌트의 메모리로 사용되는 상태는 함수가 반환되면 사라지는 일반적인 변수들과는 다릅니다. 상태는 사실 함수 바깥의 React에서 관리되고 "살아있습니다". React가 컴포넌트를 호출했을 때, 컴포넌트는 특정 렌더링을 위한 스냅샷을 제공해줍니다. 컴포넌트는 최신의 props와 이벤트 핸들러를 포함한 JSX와 함께 UI의 스냅샷을 반환합니다. 이 모든 것들은 상태값에 기반해 계산됩니다!

이 과정에 대한 이해를 돕기 위한 실험을 하나 해봅시다. 이 예시에서 여러분은 "+3" 버튼을 클릭하면 카운터가 3 올라갈 것이라고 예상할 겁니다. 이 버튼이 setNumber(number + 1)을 세 번 클릭할 것이니까요. "+3" 버튼을 클릭했을 때 어떤 일이 일어나는지 봅시다.

number는 오직 클릭당 한번만 증가했습니다!

상태를 변경하는 것은 오직 다음 렌더링을 위해서만 동작합니다. 첫 번째 렌더링 단계에서 number는 0이었습니다. 이 onClick 핸들러가 포함된 렌더링에서 number 값은 0이었기 때문에 setNumber(number + 1)이 호출된 이후에도 상태는 계속 0입니다.

<button onClick={() => {
  setNumber(number + 1);
  setNumber(number + 1);
  setNumber(number + 1);
}}>+3</button>

버튼의 클릭 핸들러가 React에게 전달하는 내용은 다음과 같습니다.

  1. setNumber(number + 1) : number는 0이므로 setNumber(0 + 1)
  • React는 다음 렌더링에서 number를 1로 바꿀 준비를 합니다.
  1. setNumber(number + 1) : number는 0이므로 setNumber(0 + 1)
  • React는 다음 렌더링에서 number를 1로 바꿀 준비를 합니다.
  1. setNumber(number + 1) : number는 0이므로 setNumber(0 + 1)
  • React는 다음 렌더링에서 number를 1로 바꿀 준비를 합니다.

setNumber(number + 1)을 세 번 호출했다 하더라도, 이번 렌더링의 이벤트 핸들러에서 number는 항상 0이므로 상태를 1로 세 번 변경한 것과 같습니다. 이것이 이벤트 핸들러의 실행이 끝난 이후에 React가 컴포넌트의 number를 3이 아닌 1로 렌더링하는 이유입니다.

상태 변수를 직접 코드에 넣고 시각화 해봅시다. number 상태 변수가 이번 렌더링에서는 0이므로 이벤트 핸들러는 다음과 같이 생겼을 것입니다.

<button onClick={() => {
  setNumber(0 + 1);
  setNumber(0 + 1);
  setNumber(0 + 1);
}}>+3</button>

다음 렌더링에서 number는 1입니다. 따라서 그때의 렌더링에서 클릭 핸들러는 다음과 같이 생겼을 것입니다.

<button onClick={() => {
  setNumber(1 + 1);
  setNumber(1 + 1);
  setNumber(1 + 1);
}}>+3</button>

버튼을 다시 클릭했을 때 카운터가 2가 되고 그 다음엔 3이 되는 이유입니다.

시간에 따른 상태​

재밌지 않나요? 아래 코드를 보고 버튼을 클릭했을 때 어떤 내용이 alert 될 지 맞춰보세요.

만약 이전 렌더링의 상태 변경 메소드를 사용했다면, alert가 "0"을 보여줄 것을 예상해볼 수 있습니다.

setNumber(0 + 5);
alert(0);

만약 alert에 타이머를 추가한다면, 컴포넌트가 리렌더링 된 후에 alert가 호출될까요? "0"과 "5" 중 alert가 보여줄 내용은 무엇일까요?

놀랍지 않나요? 상태 변경 메소드를 사용했다면, 여러분은 그 상태의 "스냅샷"이 alert에 전달됨을 확인하실 수 있습니다.

setNumber(0 + 5);
setTimeout(() => {
  alert(0);
}, 3000);

React에 저장된 상태는 alert가 실행되는 중에 변경되었을 수도 있습니다. 그치만 alert는 이전 상태, 즉 유저가 버튼을 클릭했을 때의 스냅샷을 이용해 스케줄링 되었습니다!

상태 변수의 값은 항상 렌더링과 함께 업데이트 됩니다. 만약 이벤트 핸들러의 코드가 비동기적이라두요. 이전 상태의 렌더링의 onClick에서, number는 setNumber(number + 5)가 호출된 다음이라도 0입니다. 그 값은 React가 컴포넌트가 호출되어 "스냅샷을 촬영한 시점"의 값으로 "고정되어" 있습니다.

아래 예시는 이벤트 핸들러가 타이밍 관련한 실수를 덜 하게 만드는 방법입니다. 아래 코드는 5초의 딜레이를 갖고 메시지를 보내는 form입니다. 시나리오를 생각해보면:

  1. "Send" 버튼을 클릭하면 Alice에게 "Hello"가 보내집니다.
  2. 5초의 딜레이가 끝나기 전에 여러분은 "To" 필드를 "Bob"으로 변경합니다.

alert가 보여줄 내용은 무엇일까요? "You said Hello to Alice"와 "You said Hello to Bob" 중 어떤 메시지를 보여줄까요? 한번 추측해보고 실행해보세요!

React는 상태 값을 "고정된 상태로" 이벤트 핸들러가 들고있도록 합니다. 여러분은 코드가 실행되면서 상태가 변할 것에 대해서는 걱정할 필요가 없습니다.

만약 리렌더링 되기 직전의, 즉 최신의 상태를 읽고 싶은 경우엔 어떻게 할까요? 다음 장에서 배울 상태 변경 함수를 이용하면 됩니다!

되짚어보기​

  • 상태를 변경하는 것은 새로운 렌더링을 요청합니다.
  • React는 컴포넌트 외부에 상태를 저장합니다. React에 선반이 있다고 생각하면 쉽습니다.
  • useState를 호출하면, React는 그 렌더링에 대한 상태 스냅샷을 제공합니다.
  • 변수들과 이벤트 핸들러들은 리렌더링이 일어나는 과정에서 "살아남지" 못합니다. 내용이 같더라도, 모든 렌더링에서 이벤트 핸들러들은 전부 다릅니다.
  • 모든 렌더링(그리고 함수 내부)에서는 항상 React가 그 렌더링에 대해 준 상태의 스냅샷만 볼 수 있습니다.
  • 렌더링된 JSX를 생각할 때와 비슷하게, 이벤트 핸들러 내부에서의 상태도 비슷하게 상상해볼 수 있습니다.
  • 이벤트 핸들러 내부에서 상태값을 참조하고 있다면, 그 상태는 이벤트 핸들러가 생성된 시점의 값입니다.

도전 과제​

1. 신호등 구현하기​

아래는 버튼이 클릭되면 신호등 불빛이 바뀌는 컴포넌트입니다.

클릭 핸들러에 alert를 추가해보세요. 신호등이 녹색이고 "Walk"일 때 버튼을 클릭하면 "Stop is next"를 출력하고, 신호등이 적색이고 "Stop"일 때 버튼을 클릭하면 "Walk is next"를 출력하도록 해보세요.

alert를 setWalk의 앞에 두었을 때와 뒤에 두었을 때 차이가 있나요?

정답​

여러분의 alert는 아래와 같을 것입니다.

alert를 setWalk의 앞에 놓는것과 뒤에 놓는 것은 아무 차이도 없을 것입니다. walk의 값은 그 때의 렌더링에 고정되어 있기 때문입니다. setWalk를 호출하는 것은 다음 렌더링에서의 상태를 변경할 뿐입니다. 이전 상태로부터 파생된 렌더링에서의 이벤트 핸들러에는 아무 영향을 끼칠 수 없습니다.

아래 코드는 처음에는 직관적이지 않게 보일수 있습니다.

alert(walk ? 'Stop is next' : 'Walk is next');

하지만 "신호등이 'Walk now'일 경우 'Stop it next' 메시지를 보여준다"라고 읽으면 말이 됩니다. 이벤트 핸들러 안에 있는 walk 변수는 현재 렌더링의 walk 값과 일치하고, 변하지 않습니다.

이게 진짜 맞는지는 상태 변경 메소드를 바꿔봄으로써 검증할 수 있습니다. walk가 true일 경우 코드는 아래와 같을 것입니다.

<button onClick={() => {
  setWalk(false);
  alert('Stop is next');
}}>
  Change to Stop
</button>
<h1 style={{color: 'darkgreen'}}>
  Walk
</h1>

결국, "Change to Stop"을 클릭하면 walk가 false로 세팅된 상태로 렌더링을 요청합니다. 그리고 "Stop is next" 메시지를 띄울 것입니다.
