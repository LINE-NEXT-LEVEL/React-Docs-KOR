이벤트에 반응하기

React에서는 JSX에 이벤트 핸들러를 부착할 수 있습니다. 이벤트 핸들러는 여러분이 직접 만든 함수들로, 클릭(click), 호버(hover), 입력창에 포커스(focus)같은 이벤트가 발생했을 때 반응하는 함수를 말합니다.

You will Learn
  • 이벤트 핸들러를 작성할 수 있는 여러 가지 방법들
  • 부모 컴포넌트로부터 이벤트 핸들링 로직을 전달하는 방법
  • 어떻게 이벤트가 전파되는지, 그리고 이벤트 전파를 막는 방법

이벤트 핸들러 추가하기​

이벤트 핸들러를 추가하기 위해서는 우선 함수를 선언하고 적절한 JSX 태그에 prop으로 전달해야 합니다. 예를 들어, 아래 예시에는 아직 아무것도 하지 않는 버튼이 있습니다.

아래 세 단계를 통해 유저가 버튼을 클릭한 경우 메시지를 보여주도록 할 수 있습니다.

  1. Button 컴포넌트 내부에 handleClick 함수를 선언합니다.
  2. 함수 내부에 로직을 구현합니다(alert를 이용해 메시지를 보여주도록 합니다).
  3. <button> JSX에 onClick={handleClick}을 추가합니다.

여러분은 handleClick 함수를 정의해 <button>에 prop으로 전달했습니다. handleClick은 이벤트 핸들러입니다. 이밴트 핸들러는

  • 보통 컴포넌트 내부에 정의됩니다.
  • handle 로 시작하는 이름을 갖습니다. 그 뒤에는 이벤트 이름이 붙습니다.
정보

관습적으로, 이벤트 핸들러의 이름은 handle 뒤에 이벤트 이름이 붙습니다. 여러분은 onClick={handleClick}이나 onMouseEnter={handleMouseEnter}같은 이름들을 종종 보게 될겁니다.

JSX에 인라인으로 이벤트 핸들러를 넣는 것도 가능합니다.

<button onClick={function handleClick() {
  alert('You clicked me!');
}}>

혹은 더 간단하게 화살표 함수를 사용할 수도 있습니다.

<button onClick={() => {
  alert('You clicked me!');
}}>

위는 전부 같은 코드입니다. 인라인 이벤트 핸들러는 짧은 함수일때 유용합니다.

위험

이벤드 핸들러로 전달되는 함수들은 전달만 되어야 하고, 호출되어서는 안됩니다. 예를 들어

함수를 전달(O)함수를 호출(X)
<button onClick={handleClick}><button onClick={handleClick()}>

둘의 차이는 미묘합니다. 첫 번째 예시에서 handleClick 함수는 onClick 이벤트 핸들러로써 전달되었습니다. 이는 React에게 함수를 기억하게 하고 버튼이 클릭되었을 때 그 함수를 호출하라고 명령하는 것입니다.

두 번째 예시에서 handleClick() 뒤에 붙은 ()는 클릭 이벤트가 발생하지 않더라도 렌더링 도중에 즉시 함수가 실행되도록 합니다. JSX의 {와}가 내부에 있는 JavaScript 코드를 즉시 실행시키기 때문입니다.

코드를 인라인으로 작성한다면, 같은 위험에 처할 수 있습니다.

함수를 전달(O)함수를 호출(X)
<button onClick={() => alert('...')}><button onClick={alert('...'}>

이런 방식으로 인라인 코드를 작성하면 클릭 이벤트 발생 시에 함수가 호출되지 않습니다. - 컴포넌트가 렌더링 될때마다 함수가 호출됩니다.

// 이 alert는 컴포넌트가 렌더링 될 때 호출됩니다. 클릭 이벤트 발생 시가 아닙니다.
<button onClick={alert('You clicked me!')}>

만약 이벤트 핸들러를 인라인으로 작성하고자 한다면, 아래처럼 익명 함수에 이를 감싸야 합니다.

<button onClick={() => alert('You clicked me!')}>

이렇게 하면 렌더링 될때마다 코드가 실행되는 것이 아니라, 후에 호출될 함수를 생성하게 됩니다.

두가지 경우에서 JSX에 전달해야 하는 것은 함수입니다.

  • <button onClick={handleClick} 은 handleClick 함수를 전달합니다.
  • <button onClick={() => alert('...')}>은 () => alert('...') 함수를 전달합니다.
JavaScript Refresher에서 화살표 함수에 대한 정보를 더 확인해보세요.

이벤트 핸들러에서 props 읽기​

이벤트 핸들러들은 컴포넌트 내부에 선언되기 때문에, 핸들러들은 컴포넌트의 props에 접근할 수 있습니다. 래 버튼은 클릭되었을 때 message prop을 이용해 alert를 띄워줍니다.

두 개의 버튼은 서로 다른 메시지를 보여주고 있습니다. 이 버튼들에 전달되는 메시지를 한번 바꿔보세요.

이벤트 핸들러를 props로써 전달하기​

가끔 자식 컴포넌트에서는 부모 컴포넌트로부터 정의된 이벤트 핸들러를 내려받고 싶을 때가 있습니다. Button 컴포넌트가 어디서 사용되는지에 따라 다른 함수를 실행시켜야 할 때를 생각해보세요.

  • 어떤 버튼은 영화를 재생하고 또 어떤 버튼은 이미지를 업로드 하는 동작을 해야 할 수 있습니다.

이를 위해서는 컴포넌트가 부모 컴포넌트로부터 받은 props를 다음과 같이 이벤트 핸들러로 전달합니다.

이 Toolbar 컴포넌트는 PlayButton과 UploadButton을 렌더링합니다.

  • PlayButton은 Button의 내부에 handlePlayClick을 onClick prop으로 전달합니다.
  • UploadButton은 Button의 내부에 () => alert('Uploading!')을 onClick prop으로 전달합니다.

결국, 여러분의 Button 컴포넌트는 onClick을 통해 prop을 전달받습니다. 이는 onClick={onClick}을 통해 브라우저의 내장 객체 <button>에게 바로 전달됩니다. 그리고 React는 클릭 이벤트가 발생했을때 이를 호출하라는 명령을 받습니다.

만약 여러분이 디자인 시스템을 사용한다면, 컴포넌트가 어떤 스타일만 포함하도록 하고 동작은 특정짓지 않도록 하는 방식이 자연스럽게 느껴질겁니다. 대신, PlayButton이나 UploadButon과 같은 컴포넌트들은 이벤트 핸들러를 자식 컴포넌트로 넘겨줄 것입니다.

이벤트 핸들러 props에 이름 붙이기​

내장 컴포넌트인 <button>이나 <div>는 onClick과 같은 브라우저의 이벤트 이름들만 지원합니다. 그치만 만약 여러분이 직접 컴포넌트를 만들어야 한다면, 이벤트 핸들러 props의 이름은 여러분이 원하는대로 지어도 괜찮습니다.

관습적으로 이벤트 핸들러 props의 이름은 `on`으로 시작해야 합니다. 그 뒤는 대문자로 된 단어가 붙습니다.

예를 들어, Button 컴포넌트의 onClick prop은 onSmash로 이름이 붙여질 수 있습니다.

이 예제에서 <button onClick={onSmash}>는 브라우저의 <button>(소문자)이 여전히 onClick이라는 prop을 받아야 함을 보여주고 있습니다. 하지만 직접 만든 Button 컴포넌트로부터 내려받은 prop의 이름은 온전히 여러분이 어떻게 이름을 붙이냐에 달려있습니다!

컴포넌트가 여러 동작을 해야 한다면, 앱에 관련된 개념을 차용해 이름을 붙일 수도 있습니다. 예를 들어 이 Toolbar 컴포넌트는 onPlayMovie와 onUploadImage 이벤트 핸들러를 prop으로 받습니다.

App 컴포넌트는 Toolbar가 onPlayMovie와 onUploadImage를 가지고 무엇을 하는지 전혀 모른다는 점에 주목해보세요. 그건 Toolbar가 자세하게 구현할 겁니다. 지금 Toolbar는 이 prop들을 onClick 핸들러를 통해 Button들에게 내려주지만, 나중엔 클릭이 아닌 키보드 단축키 이벤트를 통해서도 이벤트 핸들러를 호출해야 할 수도 있습니다. onPlayMovie와 같은 앱과 관련된 동작으로 이름을 붙인다면 미래의 변경을 대비한 유연성을 챙겨갈 수 있습니다.

이벤트 전파​

이벤트 핸들러는 자식 컴포넌트에서 발생한 이벤트에 대해 반응할 수도 있습니다. 우리는 이런걸 트리 위로 "버블링(bubble)" 혹은 "전파(propagate)"된다고 합니다. 이벤트는 발생한 지점부터 시작해 트리를 타고 올라갑니다.

이 <div>는 두개의 버튼을 자식으로 갖고 있습니다. <div>와 버튼들은 각각 onClick 핸들러를 갖고 있습니다. 버튼을 클릭하면 어떤 이벤트 핸들러가 반응할까요?

어떤 버튼이든 클릭했을 때 그 버튼이 갖고있는 onClick이 먼저 실행됩니다. 그리고 바로 이어서 부모인 <div>의 onClick이 호출됩니다. 즉 두개의 메시지가 나타나게 되죠. 만약 버튼이 아닌 툴바 자체를 클릭하면 부모인 <div>의 onClick만 실행됩니다.

위험

React에서 onScroll을 제외한 모든 이벤트는 전파됩니다. onScroll은 이벤트를 붙인 JSX 태그에서만 동작합니다.

전파 막기​

이벤트 핸들러는 이벤트 객체를 인자로 전달받습니다. 관습적으로 "event"를 줄여 e라고 부릅니다. 이 객체를 통해 이벤트에 대한 정보를 읽을 수 있습니다.

또한 이벤트 객체를 통해 전파를 막는것도 가능합니다. 만약 부모 컴포넌트에 도달하기 전에 이벤트의 전파를 막으려 한다면, 아래 예시처럼 e.stopPropagation()을 호출하면 됩니다.

버튼이 클릭되면

  1. React는 <button>에 넘겨진 onClick 핸들러를 호출합니다.
  2. 그 핸들러는 Button에 정의되어 있으며, 아래와 같은 동작이 일어납니다.
    • e.stopPropagation()을 호출해 이후 이벤트 버블링이 일어나지 않도록 막습니다.
    • Toolbar 컴포넌트의 prop으로 전달받은 onClick 함수를 호출합니다.
  3. Toolbar에 정의된 함수는 버튼만의 alert를 띄우도록 합니다.
  4. 전파가 멈췄기 때문에 부모인 <div>의 onClick은 실행되지 않습니다.

e.stopPropagation() 때문에 버튼을 클릭했을때 이제는 두개(<button>과 부모 툴바인 <div>)가 아닌 한개(<button>)의 alert만 나타나게 됩니다. 버튼을 클릭하는 것은 이를 둘러싼 툴바를 클릭하는 것과 동일하지 않으므로, 전파를 막는게 이 상황에서는 더 적절한 UI입니다.

Deep Dive

이벤트의 캡쳐링(capture) 단계​

자식 요소의 이벤트들이 전파를 막았다 하더라도 자식 요소에서 발생한 모든 이벤트들을 캐치해야 하는 경우가 드물게 발생합니다. 예를 들어, 전파 로직과 관계 없이 분석을 위해 모든 클릭 이벤트를 기록(log)해야 하는 경우가 있습니다. 이럴 때는 이벤트 이름 뒤에 Capture를 추가해 간단히 해결할 수 있습니다.

<div onClickCapture={() => { /* 여기가 먼저 실행됩니다. */ }}>
  <button onClick={e => e.stopPropagation()} />
  <button onClick={e => e.stopPropagation()} />
</div>

각 이벤트는 아래와 같은 3단계를 거쳐 전파가 일어납니다.

  1. 이벤트가 트리의 루트로부터 아래로 전파됩니다. 이때 만나는 onClickCapture 핸들러를 모두 실행시킵니다.
  2. 요소의 onClick 핸들러를 실행시킵니다.
  3. 이벤트가 다시 요소로부터 위로 전파됩니다. 이때 만나는 모든 onClick 핸들러를 실행시킵니다.

캡쳐 이벤트는 라우터나 분석을 위한 코드에 유용하게 사용됩니다. 하지만 그 외엔 아마 잘 쓰이지 않을겁니다.

이벤트 전파 대신 핸들러를 전달하기​

이 클릭 핸들러는 몇줄의 코드를 실행한 후에 부모로부터 prop으로 내려받은 onClick을 실행합니다.

function Button({ onClick, children }) {
  return (
    <button onClick={e => {
      e.stopPropagation();
      onClick();
    }}>
      {children}
    </button>
  );
}

부모로부터 내려받은 onClick을 호출하기 전에 더 많은 코드를 작성할 수도 있습니다. 이런 패턴은 이벤트 전파의 대안이 될 수 있습니다. 이는 자식 컴포넌트가 부모가 특정한 몇몇 동작을 추가하면서도 이벤트를 제어할 수 있습니다. 이벤트 전파와 다르게, 자동으로 일어나지 않습니다. 하지만 그렇기 때문에 여러분이 직접 이벤트의 흐름을 추적하고 몇몇 이벤트 핸들러의 실행 결과를 따라갈 수 있다는 장점이 있습니다.

만약 여러분이 이벤트 전파에 의존한다면, 어떤 핸들러가 왜 실행되었는지 추적하기 어려울 수 있습니다. 대신 방금 제안한 방법을 이용해보세요.

기본 동작 막기​

어떤 브라우저 이벤트는 기본 동작을 갖고 있습니다. 예를 들어, <form>의 제출(submit) 이벤트는 <form> 내부에 있는 버튼이 클릭되면 발생하고 전체 페이지를 새로고침 하는게 기본 동작입니다.

이벤트 객체를 통해 e.preventDefault()를 호출하면 이를 막을 수 있습니다.

e.stopPropagation()과 e.preventDefault()를 혼동하지 마세요. 둘 모두 유용하지만, 서로 연관은 없습니다.

  • e.stopPropagation()은 상위 태그의 이벤트 핸들러가 실행되는 것을 막아줍니다.
  • e.preventDefault()는 브라우저에 정의된 몇몇 기본 동작을 막아줍니다.

이벤트 핸들러가 부작용(side dffect)를 일으킬 수 있나요?​

물론입니다! 이벤트 핸들러가 부작용을 일으킬 수 있는 최적의 장소입니다.

렌더링 함수들과는 다르게 이벤트 핸들러는 꼭 순수할 필요가 없습니다. 그래서 이벤트 핸들러는 무언가를 바꾸는데 최적의 장소입니다. 예를 들어, 타이핑에 반응해 어떤 입력창(input)의 value를 바꾸거나 버튼을 누름에 따라 리스트의 내용을 바꿀 수 있습니다. 하지만, 어떤 정보를 수정하기 위해서는 먼저 정보를 저장할 곳이 필요합니다. React에서는 컴포넌트의 메모리인 state에 저장하곤 합니다. 다음 페이지에서는 state에 대한 모든 것을 다룹니다.

정리​

  • <button>과 같은 요소의 prop에 함수를 전달함으로써 이벤트를 제어할 수 있습니다.
  • 이벤트 핸들러는 반드시 전달만 되어야 하고, 호출되어서는 안됩니다! onClick={handleClick}같이 전달되어야 하고 onClick={handleClick()}은 안됩니다.
  • 이벤트 핸들러는 인라인으로 선언될 수도 있고, JSX와 분리되어 선언될 수도 있습니다.
  • 이벤트 핸들러는 컴포넌트 안에 정의되므로 props에 접근하는 것이 가능합니다.
  • 이벤트 핸들러를 부모 컴포넌트에서 정의한 후 자식에게 prop으로 넘겨줄 수 있습니다.
  • 앱에 연관된 이름으로 이벤트 핸들러의 이름을 짓는것도 가능합니다.
  • 이벤트는 위로 전파됩니다. 핸들러의 첫번째 인자를 통해 e.stopPropagation()을 호출하는 것으로 이를 막을 수 있습니다.
  • 이벤트는 브라우저의 기본 동작에 의해 원하지 않는 동작을 발생시킬 수 있습니다. e.preventDefault()를 통해 이를 막을 수 있습니다.
  • 이벤트 핸들러를 prop으로 받아 자식의 핸들러에서 호출하는 것은 전파의 좋은 대안입니다.
문제에 도전해보세요.

도전 1 : 이벤트 핸들러 고치기​

이 버튼을 클릭하면 페이지의 배경색이 검게 또는 하얗게 변해야 합니다. 그런데 버튼을 클릭해도 아무런 일도 일어나지 않네요. 이 문제를 고쳐보세요. (handleClick의 로직은 신경쓰지 마세요. 여기는 아무런 문제가 없습니다.)

정답​

문제는 <button onClick={handleClick()}>에서 handleClick을 전달하지 않고 호출했기 때문입니다. ()를 없애 <button onClick={handleClick}>으로 바꾸면 문제가 해결됩니다.

문제에 도전해보세요.

도전 2 : 이벤트 묶기​

이 ColorSwitch 컴포넌트는 페이지의 색상을 바꾸는 버튼을 렌더링합니다. 부모로부터 prop으로 전달받은 이벤트 핸들러인 onChangeColor를 통해 버튼을 클릭했을 때 색상이 바뀌도록 해보세요.

그 다음엔 버튼을 클릭했을 때 페이지 클릭 카운터가 올라가는지 확인해보세요. 부모 컴포넌트를 만든 여러분의 동료는 onChangeColor가 페이지 카운터를 증가시키지 않아야 한다고 합니다. 버튼을 클릭했을 때 카운터를 증가시키지 않고 오직 색상만 변경하도록 해보세요.

정답​

우선, 이벤트 핸들러를 부착해야 합니다. <button onClick={onChangeColor}> 이렇게요.

그치만 이는 카운터가 증가하는 문제를 만들고 말았습니다. 만약 여러분의 동료의 말처럼 onChangeColor가 카운터를 증가시키지 않아야 한다면, 이벤트가 전파되어 상위 계층에 있는 어떤 다른 핸들러가 카운터를 증가시켰을 것입니다. 이 문제를 해결하기 위해서는 이벤트 전파를 멈춰야 합니다. 그치만 onColorChange의 동작은 여전히 유지되어야 함을 잊지 마세요.
