상태: 컴포넌트의 기억

컴포넌트들은 종종 인터랙션의 결과에 따라 무엇이 스크린에 올지를 변경해야 합니다. form에 적어내는 것은 입력 필드를 업데이트해야하고, 이미지 회전의 "다음"을 클릭하면 보여지는 이미지들을 변경해야 하고, "구매"를 클릭하면 물건을 쇼핑 카트에 넣어야 합니다. 컴포넌트들은 현재의 입력 값, 현재 이미지, 쇼핑카트를 "기억"해야 합니다. React에서 이러한 종류의 컴포넌트 특정 기억을 상태라고 부릅니다.

보통의 변수가 충분하지 않을 떄​

여기 조각 이미지를 렌더링하는 컴포넌트가 있습니다. "다음"버튼을 클릭하면 인덱스를 1에서 2로, 그리고 그 다음으로 바꾸면서 다음 조각을 보여줘야 합니다. 하지만, 이 코드는 동작하지 않습니다. (해볼 수 있습니다!):

handleClick 이벤트 핸들러는 지역변수를 업데이트하고 있습니다. 하지만 두 가지의 요인이 변화가 눈에 보이게 하는 것을 막고 있습니다:

  1. 지역변수는 렌더링이 일어날때 마다 다시 생성됩니다. React가 컴포넌트를 두 번째로 렌더링할때, 처음부터 다시 렌더링합니다 - 지역변수의 어떤 변화도 고려하지 않습니다.
  2. 지역변수의 변화는 렌더링을 일으키지 않습니다. React는 새로운 데이터로 그 컴포넌트를 다시 렌더링할 필요가 있다는 것을 인지하지 못합니다.

컴포넌트를 새 데이터로 업데이트하기 위해서는, 두 가지 일이 일어나야 합니다:

  1. 렌더링 사이에 데이터를 유지합니다.
  2. React가 새로운 데이터를 가진 컴포넌트를 렌더링하도록 유도합니다.(리렌더링)

useState 훅이 그 두 개를 제공합니다:

  1. 렌더링 사이에 데이터를 유지하는 상태변수
  2. 변수를 업데이트하고 React가 컴포넌트를 다시 렌더링하도록 유도하는 상태 설정 함수

상태 변수를 추가하기​

상태변수를 추가하기 위해서는 파일 상단에 React에서 useState를 import하세요.

import { useState } from 'react';
let index = 0;

그리고 이 라인을

const [index, setIndex] = useState(0);

으로 교체하세요.

index는 상태변수이고 setIndex는 상태 설정 함수입니다.

노트

[와 ] 문법은 구조분해할당이라고 불리고 배열에서 값을 읽어냅니다. useState가 돌려준 배열은 항상 정화히 두 개의 아이템을 갖고 있습니다.

handleClick은 상태변수와 함께 이렇게 쓰입니다:

function handleClick() {
  setIndex(index + 1);
}

여러분의 첫 번째 훅을 만나보세요​

React에서, "use"로 시작하는 모든 함수들, useState도 마찬가지로, 은 훅이라고 불립니다. 훅은 리액트가 렌더링( 다음 페이지에서 더 자세하게 다룰 것입니다. )하는 동안만 사용가능한 특별한 함수입니다. 훅은 React의 다른 특징들에 "연결"되도록 해줄 것입니다.

상태는 단지 그 중 한가지 특징에 불과하지만, 다른 훅들도 나중에 만나게 될 것입니다.

주의할 점

use로 시작하는 함수인 훅은, 컴포넌트의 최상단에서만 불리거나 직접 만든 훅에서만 불릴 수 있습니다. 조건문, 반복분, 혹은 다른 중첩된 함수 안에서는 훅을 부를 수 없습니다. 훅은 함수지만, 컴포넌트가 필요로 하는 무조건적인 선언으로 생각하는 것이 도움이 됩니다. 파일 상단에 모듈을 "import"하는 것과 비슷하게 컴포넌트의 상단에 React의 특성을 "사용"하는 것입니다.

useState 해부​

useState를 부를 때는 React에게 이 컴포넌트의 무언가를 기억하라고 말할 때입니다:

const [index, setIndex] = useState(0);

이 경우는 React가 index를 기억하기를 원하는 경우입니다.

노트

const [something, setSomething]처럼 짝으로 이름짓는 것이 관례입니다. 좋아하는 무언가로 이름을 지어도 되지만, 관례를 따르는 것이 프로젝트 전반에 걸쳐서 이해하는 데 도움이 됩니다.

useState의 유일한 인자는 상태변수의 초기값입니다. 이 예시에서, index의 초기값은 useState(0)에 0으로 설정됩니다.

컴포넌트가 렌더링 될 때마다, `useState은 두 값을 갖고 있는 배열을 줍니다:

  1. 저장한 값을 갖고 있은 상태변수(index)
  2. 상태값을 업데이트할 수 있고 React가 컴포넌트를 다시 렌더링하도록 하는 상태 설정 함수(setIndex) 실제로 일어나는 방식은 이렇습니다:
const [index, setIndex] = useState(0);
  1. 컴포넌트가 첫 번째 렌더링을 합니다. useState에 index의 최초값으로 0를 넘겨줬기 때문에, useState는 [0,setIndex]를 되돌려 줄 것입니다. React는 0이 가장 최신의 상태값인 것을 기억합니다.
  2. 상태를 업데이트합니다 사용자가 버튼을 누를 때, setIndex(index+1)함수를 호출합니다. index는 0이기 때문에 setIndex(1)을 부릅니다. 이 함수를 부르면 React는 index가 이제 1이라는 것을 기억하고 또 다른 렌더링을 유발합니다.
  3. 컴포넌트의 두 번째 렌더링입니다. React는 여전히 useState(0)을 보지만, index가 1이고, 그 함수가 대신 [1,setIndex]을 반환했다는 것을 기억합니다.
  4. 그리고 반복됩니다!

컴포넌트에 여러 개의 상태변수를 주기​

한 컴포넌트에 원하는 만큼 많은 타입의 상태변수를 가질 수 있습니다. 이 컴포넌트는 두 개의 상태변수를 갖고, 숫자인 index와 "자세히 보기"를 클릭하면 전환되는 불린값인 showMore이 있습니다:

서로 연관되어 있지 않으면 이 예시의 index, showMore처럼 여러 상태변수를 갖는 것은 좋은 생각입니다. 하지만 두 상태변수를 같이 종종 바꾼다는 것을 인지한다면, 두 개를 하나의 것으로 합치는게 좋을 수도 있습니다. 예를 들어, 많은 필드를 가진 양식을 갖고 있다면, 각 필드마다 상태변수를 가지는 것보다 객체 하나를 갖는 하나의 상태변수를 사용하는 것이 더 편리합니다. 상태 구조를 선택하는 것에서 이에 대한 더 많은 팁을 얻을 수 있습니다.

Deep Dive

React는 어떤 상태를 반환해 줘야 하는 지 어떻게 아나요?​

여러분은 useState가 어떤 상태변수를 가르키는지에 대한 아무 정보도 받지 않는다는 사실을 눈치채셨을 수도 있습니다. useState에 전달되는 것에는 아무 "식별자"도 없습니다. 그래서 React는 상태변수 중 어떤 것을 반환해야 하는지 어떻게 아는 걸까요? 함수를 쪼개면서 어떤 마법에라도 의존하는 걸까요? 답은 "아니다"입니다.

대신, 정확한 문법을 가능하게 하기 위해, 훅은 같은 컴포넌트의 모든 렌더링에서 부르는 안정된 호출 순서에 의존합니다. 이렇게 하는 것은 위의 규칙("최상단에서만 훅을 부른다")을 따른다면 꽤 잘 작동합니다. 훅은 항상 같은 순서로 호출될 것입니다. 게다가, linter plugin이 대부분의 실수를 잡아냅니다.

내부적으로, React는 각 컴포넌트의 상태 쌍 배열을 갖고있습니다. 그 배열은 렌더링 전에 0으로 설정된 현재의 index 쌍도 유지합니다.useState를 부를 때마다, React는 다음 상태 쌍을 주고 index를 증가시킵니다. React 훅: 마법이 아닌 단순한 배열에서 동작방식을 더 읽어볼 수 있습니다.

상태는 고립되고 사적입니다​

상태는 화면에 나오는 컴포넌트 객체에 지역적입니다. 다른 말로, 같은 컴포넌트를 두 번 렌더링 한다면, 각각의 복사본은 완전히 분리된 상태를 가질 것입니다! 그 중 하나를 바꾸는 것은 나머지 하나에 영향을 주지 않습니다.

이 예시에서, 아까의 Gallery 컴포넌트는 논리 구조에 변화 없이 두 번 렌더링 되었습니다. 각 갤러리의 버튼을 눌러보세요. 상태가 각각 독립적이라는 것을 확인하실 수 있습니다:

이 부분이 모듈의 상단에 선언되곤 하는 일반적인 변수와 다른 점입니다. 상태는 특정 함수 호출이나 코드의 특정 위치에 묶여있지 않고, 화면의 특정 위치에 "지역적"입니다. 두 Gallery 컴포넌트를 렌더링했기 때문에 각 상태는 분리되서 저장됩니다.

또한 Page컴포넌트가 Gallery에 대해 아무것도 "모르고" 어떤 상태를 가졌는지조차 모른다는 사실을 확인하세요. prop과 다르게, "상태는 상태를 선언한 컴포넌트 안에 완전히 사적인 것입니다". 부모 컴포넌트는 상태를 바꿀 수 없습니다. 이러한 사실은 나머지 컴포넌트에 아무 영향도 주지 않고 상태를 추가하거나 제거할 수 있도록 합니다.

만약 두 갤러리가 상태를 동기화하고자 한다면 어떻게 해야할까요? React에서 사용되는 옳바른 방법으로는 자식 컴포넌트로부터 상태를 제거하고 가장 가까운 공통의 부모 컴포넌트에 그 상태를 추가하는 것입니다. 다음 몇 페이지는 한 컴포넌트에 상태를 정리하는 법에 초점을 맞출 것이지만, 컴포넌트 간에 상태를 공유하는 것에서 이 주제로 돌아올 것입니다.

요약​

  • 렌더링 사이에 컴포넌트가 어떤 정보를 "기억"해야 한다면 상태변수를 사용하십시오.
  • 상태변수는 useState 훅을 불러서 선언될 수 있습니다.
  • 훅은 use로 시작하는 특별한 함수입니다. 상태와 같은 React의 특성들을 사용하도록 도와줍니다.
  • 훅은 무조건적으로 호출되어야 한다는 점에서 import가 떠오를 수도 있습니다. useState를 포함한 훅을 부른다면, 한 컴포넌트의 최상단이나 다른 훅에서 사용하는 것이 타당합니다.
  • useState 훅은 한쌍의 값을 돌려줍니다: 현재 상태와 상태를 업데이트하는 함수입니다.
  • 한 개 이상의 변수를 사용할 수 있습니다. 내부적으로 React가 순서대로 상태를 파악해나갑니다.
  • 상태는 한 컴포넌트에 사적입니다. 두 위치에 렌더링하면, 각각의 복사본은 각각 상태값을 갖습니다. 308 docs/learn/Describing the UI/rendering-lists.mdx Viewed @@ -0,0 +1,308 @@

리스트 렌더링하기

종종 수집된 데이터를 바탕으로 다양한 비슷한 컴포넌트들을 보여주고 싶을 때가 있습니다. 배열 형태의 데이터를 조작하고 싶으면 JavaScript array methods를 사용하면 됩니다. 이 페이지에서는 배열 형태의 데이터를 컴포넌트 배열로 필터링하고 변형시키기 위해 filter()과map()를 사용해 볼 것입니다.

여러분이 배울 것들
  • JavaScript의 map()을 사용하여 배열을 컴포넌트로 렌더링하는 방법
  • JavaScript의 filter()을 사용하여 특정한 컴포넌트만 렌더링하는 방법
  • React key를 언제 사용하고 왜 사용해야 하는지

배열로 데이터 렌더링하기​

여기 목록 형태의 내용이 있습니다.

<ul>
  <li>Creola Katherine Johnson: mathematician</li>
  <li>Mario José Molina-Pasquel Henríquez: chemist</li>
  <li>Mohammad Abdus Salam: physicist</li>
  <li>Percy Lavon Julian: chemist</li>
  <li>Subrahmanyan Chandrasekhar: astrophysicist</li>
</ul>

이 리스트 아이템 간의 유일한 차이는 내용, 즉 데이터입니다. 인터페이스를 구축할 때, 한 컴포넌트를 여러 인스턴스에 사용하면서 다른 데이터들을 사용하곤 합니다: 댓글 목록부터 프로필 이미지의 갤러리까지 다양한 예시가 있습니다. 이런 상황에서는 JavaScript객체와 배열 안에 데이터를 저장하고 map()이나filter()같은 메소드를 사용해서 데이터를 가진 컴포넌트 목록을 렌더링할 수 있습니다.

배열로 아이템 목록을 생성하는 방법의 짧은 예시입니다:

  1. 배열로 데이터 옮기기:
const people = [
  'Creola Katherine Johnson: mathematician',
  'Mario José Molina-Pasquel Henríquez: chemist',
  'Mohammad Abdus Salam: physicist',
  'Percy Lavon Julian: chemist',
  'Subrahmanyan Chandrasekhar: astrophysicist'
];
  1. people의 멤버들을 새로운 JSX node 배열인 listItems로 매핑하기:
const listItems = people.map(person => <li>{person}</li>);
  1. <ul>로 listItems를 감싸서 컴포넌트에서 리턴시켜주기:
return <ul>{listItems}</ul>;

  name: 'Mohammad Abdus Salam',
  profession: 'physicist',
}, {
  name: 'Percy Lavon Julian',
  profession: 'chemist',
}, {
  name: 'Subrahmanyan Chandrasekhar',
  profession: 'astrophysicist',
}];

직업이 chemist인 사람들만 보여주고 싶다고 가정해 봅시다. 그 사람들만 리턴해주기 위해서 JavaScript의 filter()메소드를 사용할 수 있습니다. 이 메소드는 아이템 배열들을 받아들이고, 테스트(true또는false값을 리턴해주는 함수)로 전달합니다. 그리고 나서, 테스트를 통과(true를 리턴)한 아이템들로만 새로운 배열을 만들어서 다시 리턴해줍니다.

여러분은 직업(profession)이 화학자(chemist)인 아이템만 원합니다. 이를 검증하기 위한 "테스트" 함수는 (person) => person.profession === 'chemist'이렇게 생겼습니다. 같이 합쳐봅시다:

  1. person.profession === 'chemist'로 걸러져야 할 사람들에 filter()를 불러서 "화확자"인 사람들만 있는 새 배열인 chemists을 생성합니다.
const chemists = people.filter(person =>
  person.profession === 'chemist'
);

2.이제 화학자로 매핑합니다.

const listItems = chemists.map(person =>
  <li>
     <img
       src={getImageUrl(person)}
       alt={person.name}
     />
     <p>
       <b>{person.name}:</b>
       {' ' + person.profession + ' '}
       known for {person.accomplishment}
     </p>
  </li>
);

3.마지막으로 컴포넌트에서 listItems를 리턴해줍니다.

return <ul>{listItems}</ul>;
//App.js
import { people } from './data.js';
import { getImageUrl } from './utils.js';
export default function List() {
  const chemists = people.filter(person =>
    person.profession === 'chemist'
  );
  const listItems = chemists.map(person =>
    <li>
      <img
        src={getImageUrl(person)}
        alt={person.name}
      />
      <p>
        <b>{person.name}:</b>
        {' ' + person.profession + ' '}
        known for {person.accomplishment}
      </p>
    </li>
  );
  return <ul>{listItems}</ul>;
}
주의할 점

화살표 함수는 => 바로 다음의 표현을 내부적으로 리턴해줍니다. 그래서 따로 return을 명시해줄 필요가 없습니다:

const listItems = chemists.map(person =>
  <li>...</li> // Implicit return!
);

하지만 =>뒤에 중괄호({})가 따라오면 명시적으로 return을 써줘야 합니다!

const listItems = chemists.map(person => { // Curly brace
  return <li>...</li>;
});

=> {을 갖고 있는 화살표 함수는 "block body"를 갖고 있다고 합니다. block body는 한 줄 이상의 코드를 쓰게 해 주지만, return을 직접 써줘야 합니다. 잊어버리면 아무것도 리턴되지 않습니다!

key로 순서대로 있는 아이템 목록을 유지하기​

새 탭에서 위에 있는 샌드박스 아무거나 열어본다면, 콘솔창에 에러를 볼 수 있을 것입니다.

주의: 리스트의 각각의 자식은 prop으로 고유의 "키"를 갖고 있어야 합니다.

아이템 배열에는 key를 주어야 합니다 — 그 배열에 있는 다른 아이템들과 구별해줄 수 있는 string이나 숫자입니다.

<li key={person.id}>...</li>

노트​

map() 바로 안에 있는 JSX element들은 항상 key가 필요합니다! key들은 나중에 짝을 지을 수 있도록 아이템 배열의 어떤 아이템이 React의 컴포넌트에 상응하는지 알려줍니다. key는 배열 아이템들이 움직이거나(분류 등), 삽입이 되거나, 제거될 때 중요해집니다. 잘 선택된 key는 React에게 정확히 무엇이 일어났는지 알려주고 DOM 트리를 적절하게 업데이트하게 해줍니다. 즉석으로 키를 생성하는 것보다, 데이터 안에 키를 포함시켜야 합니다:

Deep Dive​

각 리스트 아이템에 여러 DOM 노드들을 보여주기​

각 아이템이 보여주어야 할 게 하나가 아니라 여러개의 DOM 노드일때는 어떻게 해야할까요?

짧은 <>``</> 구문으로는 키를 전달할 수 없기 때문에 <div>로 그룹화하거나 약간 더 길고 더 명시적인 <Fragment>구문을 사용해줄 수 있습니다.

import { Fragment } from 'react';
// ...
const listItems = people.map(person =>
  <Fragment key={person.id}>
    <h1>{person.name}</h1>
    <p>{person.bio}</p>
  </Fragment>
);

Fragment는 DOM에서 사라지고 있는 그대로의 <h1>,<p>,<h1>,<p> 만 말들어줄 것 입니다.

키를 어디서 얻을 것인가​

데이터 소스에 따라 키 값이 달라집니다:

  • 데이터베이스에서 가져온 데이터: 데이터가 데이터베이스에서 온 것이라면,원래 유일한 값인 데이터베이스의 key/ID를 사용하면 됩니다.
  • 현재 환경에서 만들어진 데이터: 데이터가 현재 환경에서 생성되고 유지되는 경우(예시. 노트 필기앱의 노트), 아이템을 만들 때 [crypto.randomUUID()](https://developer.mozilla.org/en-US/docs/Web/API/Crypto/randomUUID)나[uuid](https://www.npmjs.com/package/uuid)같은 패키지를 이용해서 증가하는 숫자를 사용하세요.

키 규칙들​

  • 키는 형재/자매들 간에 특별한 값이어야 합니다. 다른 배열 상의 JSX노드에서 같은 값을 사용하는 것은 괜찮습니다.
  • 키는 불변성을 갖고 있어야 합니다. 변하면 key의 목적이 훼손됩니다. 렌더링하는 동안 key를 생성하지 마세요.

React는 키가 왜 필요할까?​

데스크탑의 파일들이 이름이 없다고 상상해보세요. 대신, 파일을 확인하려면 순서대로 확인해야 합니다. — 첫 번째 파일, 두 번째 파일 같은 식이 될 것입니다. 익숙해질 수야 있지만, 한 번 파일을 삭제하면 혼란스러울 것입니다. 두 번째가 첫 파일이 되고, 세 번째 파일이 두번째 파일이 되기 때문입니다. 익숙해질 수는 있지만, 한번 파일을 삭제하면 혼란스워질 것입니다. 두 번째 파일이 첫 번째 파일이 되고, 세 번째 파일이 두 번째 파일이 되고, 계속 반복될 것이기 때문입니다.

폴더의 파일 이름들과 배열의 JSX 키는 비슷한 목적을 갖고 있습니다. 여러분은 파일 이름과 키로 형제 관계에 있는 아이템을 식별할 수 있습니다. 잘 선택된 키는 배열에 놓인 위치보다 더 많은 정보를 전달합니다. 재배열되면서 위치가 변한다 하더라도, React는 키를 통해 아이템이 살아있는 동안 그 아이템을 식별할 수 있습니다.

주의할 점

배열의 아이템 인덱스를 키로 사용하고 싶을 수도 있습니다. 사실, 만약 키를 명시하지 않으면 React는 인덱스를 키로 사용할 것입니다. 하지만 아이템이 삽입되고, 제거되거나 재배열된다면, 렌더링 순서도 시간이 지나면서 달라질 것입니다. 인덱스를 키로 사용하는 것은 혼란스러운 버그를 일으킬 수 있습니다.

비슷하게, 키를 즉석으로 생성하지 마세요. key={Math.random()}를 예로 들 수 있습니다. 이렇게 키를 생성하는 방식은 렌더링 될 때마다 다른 키를 생성할 것이고, 매 번 모든 컴포넌트와 DOM이 재생성되도록 할 것입니다. 이렇게 하면 느릴 뿐 아니라, 리스트 아이템의 사용자 인풋 또한 잃어버릴 수 있습니다. 대신에, 데이터에 기반한 안정적인 ID를 사용하세요.

여러분의 컴포넌트는 키를 prop으로 받지 않을 것입니다. 키는 React 내부에서 힌트로만 사용됩니다. 만약 컴포넌트가 ID를 필요로 한다면, 다른 <Profile key={id} userId={id} />처럼 분리된 prop으로 넘겨줘야 합니다.
