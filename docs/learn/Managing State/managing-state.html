상태 관리하기 (중급)

여러분의 앱이 성장할수록, 여러분의 상태를 조직하고 컴포넌트간의 데이터 흐름을 더 의도적으로 구성하는 것은 도움이 될 것입니다. 불필요하거나 중복되는 버그의 흔한 이유입니다. 이 챕터에서, 여러분은 상태를 어떻게 잘 구조화하고, 상태 업데이트 로직을 유지보수 가능하게 하며, 거리가 먼 컴포넌트 사이에 상태를 공유하는 법을 배울 것 입니다.

이 챕터에서는
  • 상태가 변경함에 따라 UI변화를 생각하는 법
  • 상태를 잘 구조화하는 법
  • 컴포넌트 간에 공유하기 위해 "상태를 올리는 법"
  • 상태가 보존될지, 리셋될지를 조절하는 법
  • 함수에서 복잡한 상태 로직을 통합하는 법
  • "prop drilling"없이 정보를 넘기는 법
  • 여러분의 앱이 성장함에 따라 상태 관리를 확장하는 법

상태로 입력에 반응하기​

React와 함께라면, 여러분은 UI를 코드에서 직접 수정하지 않을 것 입니다. 예를들어 "버튼 비활성화", "버튼 활성화", "성공 메시지 출력" 등의 명령을 작성하지 않을 것 입니다. 대신에 여러분은 UI에게 컴포넌트의 다른 시각적 상태에 대해 설명할 것이고 ("초기 상태", "타이핑 상태", "성공 상태), 유저 입력에 반응하여 상태 변화를 트리거할 것 입니다. 이 것은 디자이너가 UI에 대해 생각하는 방식과 유사합니다.

여기 React로 구성된 퀴즈 폼이 있습니다. status 상태 값을 제출버튼을 활성화, 혹은 비활성화 하는 것과 성공 메시지를 보여줄지를 결정하는데 어떻게 활용하는지 기록해보세요.

상태 구조 선택하기​

상태를 잘 구조화하는 것은 수정과 디버깅하기 쉽게하고, 거듭되는 버그의 원인이 되는 컴포넌트의 다른 점을 만들어냅니다. 가장 중요한 원칙은 상태가 불필요하거나 중복되는 정보를 가지게해서는 안된다는 것 입니다. 불필요한 상태가 있다면, 업데이트하는 것을 잊기 쉬워 버그를 만들 것 입니다.

예를 들어, 이 폼은 중복되는 fullName 상태값을 가집니다.

여러분은 그것을 지우고 컴포넌트 렌더링동안 fullName을 계산하여 코드를 간단히 할 수 있습니다.

이는 작은 변화같지만, React 앱의 많은 버그들이 이 방법으로 해결됩니다.

컴포넌트 간의 상태 공유​

가끔, 여러분은 두 컴포넌트의 상태가 동시에 변경되길 원할 수 있습니다. 그렇게 하기위해서 두 컴포넌트에서 상태를 제거하고, 가장 가까운 부모 컴포넌트에 옮겨 props로 상태를 넘겨줍니다. 이는 "상태 올리기"로 알려져 있고, 여러분이 React 코드를 작성하면서 가장 흔하게하는 것중 하나입니다.

이 예시에서 한번에 하나의 패널만이 활성화 되어야합니다. 이를 위해 각 패널에 활성화 여부를 저장하지 않고, 부모 컴포넌트가 상태를 갖고 자식들에게 prop으로 명시해주면 됩니다.

상태를 보존하거나 리셋하는 법​

여러분이 한 컴포넌트를 재랜더링할때, React는 트리의 어떤 파트를 유지할지(그리고 업데이트할지), 처음부터 버리거나 재생성할지를 결정해야합니다. 대부분의 경우에 React의 자동 동작이 충분히 잘 작동합니다. 기본적으로, React는 이전에 렌더링된 컴포넌트 트리와 "잘 조화되는" 트리의 일부를 보존합니다.

하지만 가끔 이는 여러분이 원하는 것이 아닐 수 있습니다. 예를들어, 이 앱에서 메시지를 작성하고 받는 사람을 변경하는 것이 입력을 초기화 해주지 않습니다. 이는 유저가 뜻하지않게 다른사람에게 메시지를 보내게 할 수 있습니다:

React는 이 기본 동작을 오버라이딩 할 수 있게 해주고 <Chat key={email} /> 처럼 컴포넌트가 다른 key를 넘겨 상태를 초기화하도록 강요합니다. 이는 React에게 받는 사람이 다르다고 알려주는 것이고, 다른 Chat 컴포넌트로 인식되므로 처음부터 새로운 데이터(그리고 입력과 같은 UI)와 함께 컴포넌트가 재생성되어야합니다. 이제 받는 사람을 변경하는 것은 같은 컴포넌트를 렌더링하더라도 항상 입력창을 초기화 합니다.

리듀서로 상태 로직을 추출하는 법​

많은 이벤트 핸들러에 퍼져 상태 업데이트가 많은 컴포넌트는 과부하에 걸릴 수 있습니다. 이 경우를 위해 여러분은 모든 상태 업데이트 로직을 "리듀서"라고 불리는 컴포넌트 밖에 있는 하나의 함수에 통합시킬 수 있습니다. 여러분의 이벤트 핸들러는 유저의 "행동"만을 명시하기 때문에 더 간결해집니다. 파일의 아래에서, 리듀서 함수는 각 행동에 반응하여 상태가 어떻게 업데이트 되어야하는지 명시합니다.

컨텍스트로 데이터를 깊이 전달하기​

일반적으로, 여러분은 부모 컴포넌트에서 자식 컴포넌트로 props를 통해 정보를 넘겨주게 됩니다. 하지만 props를 넘기는 것은 만약 여러분이 많은 컴포넌트에 prop을 넘기거나, 많은 컴포넌트가 같은 정보를 필요로 할 때 불편할 수 있습니다. 컨텍스트는 부모 컴포넌트가 얼마나 깊든지 간에, prop으로 명시적으로 전달하지 않고도 같은 정보를 그 트리 아래의 아무 컴포넌트에게나 사용할 수 있도록 해줍니다.

여기서, Heading 컴포넌트는 그것의 Heading 레벨을 가장 가까운 Section에 레벨을 "물어서" 결정할 수 있습니다. 각 Section은 부모 Section에 묻고 그것을 추가하여 레벨을 트래킹합니다. 모든 Section은 컨텍스트를 통해 그 아래의 모든 컴포넌트들에게 props를 전달하지 않고도 정보를 제공합니다.

리듀서와 컨텍스트로 확장하기​

리듀서는 여러분이 컴포넌트의 업데이트 로직을 통합할 수 있도록 합니다. 컨텍스트는 다른 컴포넌트로 정보를 깊이 전달할 수 있도록합니다. 여러분은 리듀서와 컨텍스트를 합쳐 복잡한 화면의 상태를 관리할 수 있습니다.

이러한 접근으로, 복잡한 상태를 가진 부모 컴포넌트는 리듀서로 그것을 관리합니다. 그 트리의 모든 다른 컴포넌트는 컨텍스트를 통해 그것의 상태를 읽을 수 있습니다. 그들은 또한 상태를 업데이트하기 위해 행동들을 전달할수도 있습니다.
