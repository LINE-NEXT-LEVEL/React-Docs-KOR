컴포넌트 간 상태 공유

가끔, 여러분은 두 컴포넌트의 상태가 항상 함께 변경되기를 원할 것입니다. 이를 위해서는, 두 컴포넌트에서 상태를 제거하고 공통된 부모 컴포넌트로 옮겨준 후 상태를 props로 내려주어야합니다. 이는 상태 올리기로 알려져 있으며 여러분이 React 코드를 작성할때 가장 자주하게 될 것중 하나입니다.

여러분이 배울 것
  • 상태 올리기로 컴포넌트들 간에 상태를 공유하는 법
  • 제어 컴포넌트와 비제어 컴포넌트가 무엇인지

예시로 상태 올리기​

이 예제에서, 부모 Accordion 컴포넌트는 두 분리된 Panel들을 렌더링합니다:

  • Accordion
    • Panel
    • Panel

각 Panel 컴포넌트는 내용이 보일지 말지를 결정하는 boolean isActive상태를 가집니다.

두 panel들에서 Show 버튼을 누르세요.

하나의 panel의 버튼을 누르는 것이 다른 panel에 영향을 주지않음을 알 수 있습니다 - 둘은 독립적입니다.

하지만 여러분이 한번에 하나의 panel만 확장되도록 변경하고 싶다고 해보겠습니다. 이 디자인에서는, 두번째 panel을 확장하면 첫 panel을 접어야합니다. 여러분은 어떻게 할 수 있을까요?

두 panel들을 조정하기위해, 여러분은 세 단계를 거쳐 부모 컴포넌트로 “상태들을 올려”야 합니다.

  1. 자식 컴포넌트들에서 상태 제거하기
  2. 공통 부모에서 하드코딩된 데이터 전달하기
  3. 공통 부모에 상태를 추가하고 이벤트핸들러와 함께 내려주기

이는 Accordion 컴포넌트가 두 Panel들을 조작하여 한번에 하나만 확장되도록 합니다.

단계 1. 자식 컴포넌트에서 상태제거하기​

여러분은 Panel의 isActive의 조작을 부모 컴포넌트에 넘깁니다. 이는 부모 컴포넌트가 isActive를 Panel로 prop형태로 넘김을 의미합니다. Panel컴포넌트로부터 이 라인을 지우며 시작해보세요.

const [isActive, setIsActive] = useState(false);

그리고 대신, isActive를 Panel의 prop 리스트에 추가하세요.

function Panel({ title, children, isActive }) {

이제 Panel의 부모 컴포넌트는 isActive를 prop으로 넘겨주기로 조작할 수 있습니다. 반대로 Panel컴포넌트는 이제isActive를 조작 불가합니다. - 이제 부모 컴포넌트가 할 수 있습니다.

단계 2. 하드코딩된 데이터를 공통 부모에서 넘겨주기

상태를 올리기위해, 여러분이 조정하고 싶은 두자식 컴포넌트의 가장 가까운 공통 부모를 찾아야합니다:

  • Accordion(가장 가까운 공통 부모)
    • Panel
    • Panel

이 예시에서는 Accordion 컴포넌트입니다. 두 panel들의 상위에 있고 그들의 prop들을 조작할 수 있기때문에, 어떤 panel이 현재 활성화 상태인지에 대한 “사실 원천”이 됩니다. Accordion 컴포넌트가 두 panel들에 isActive의 하드코딩된 값을 넘겨주도록 해보세요. (예를 들면 true와 같은 것을요)

Accordion컴포넌트의 하드코딩된 isActive값 수정을 시도하고 화면에서 결과를 보세요.

단계 3. 공통부모에 상태 추가하기

상태 올리기는 종종 여러분이 상태로 저장하던 것의 본질을 바꿉니다.

이 케이스에서는, 한번에 하나의 panel만 활성되어야합니다. 이는 Accordion 공통 부모 컴포넌트는 어떤 panel이 활성화된 상태인지 계속 알고 있어야할 필요가 있다는 의미입니다. 상태 값으로 boolean 값 대신에, 숫자를 활성 Panel의 인덱스를 사용할 수 있습니다.

const [activeIndex, setActiveIndex] = useState(0);

activeIndex가 0이면, 첫번째 panel이 활성화 상태이고, 값이 1이면 두번째가 활성화됩니다.

각 Panel의 “보기” 버튼을 클릭하는 것은 Accordion의 활성 인덱스를 변경해야합니다. Panel은 상태가 Accordion내부에 있기 때문에 activeIndex 상태를 직접 설정할 수 없습니다. Accordion 컴포넌트는 prop으로 이벤트핸들러 넘겨주기를 통해 Panel 컴포넌트가 상태들을 변경할 수 있도록 명시적으로 허가해야하 합니다.

<>
  <Panel
    isActive={activeIndex === 0}
    onShow={() => setActiveIndex(0)}
  >
    ...
  </Panel>
  <Panel
    isActive={activeIndex === 1}
    onShow={() => setActiveIndex(1)}
  >
    …
  </Panel>
</>

Panel내부의 <button>은 이제 onShow prop을 클릭 이벤트 리스너로 사용합니다.

이로서 상태 올리기를 마무리합니다! 공통 부모 컴포넌트로 상태를 옮기는 것은 여러분이 두 panel들을 조정할 수 있도록 해주었습니다. 두 “보여짐” 플래그 대신에 활성 인덱스를 사용하는 것으로 한번에 하나의 panel만 활성되도록 보장합니다. 그리고 자식에게 이벤트 핸들러를 내려주는 것으로 자식이 부모의 상태를 변경할 수 있습니다.

:::Info Deep Dive

제어 그리고 비제어 컴포넌트​

지역 변수 "비제어"와 함께 컴포넌트를 호출하는 것은 일반적입니다. 예를 들어 isActive 상태 변수를 가진 원래의 Panel 컴포넌트는 그의 부모가 panel의 활성 여부에 영향을 줄 수 없기때문에 비제어입니다.

반대로, 여러분은 컴포넌트의 중요한 정보가 지역변수 대신 prop들로 전달될때 "제어"된다고 표현합니다. 이는 부모 컴포넌트가 동작을 완전히 구체화하도록 합니다. isActive prop을 가진 마지막 Panel 컴포넌트는 Accordion 컴포넌트에 의해 제어됩니다.

비제어 컴포넌트들은 구성이 덜 필요하기 때문에 부모들과 함께 사용되기 쉽습니다. 하지만 그들을 함께 조정하고 싶을때는 덜 유연하게됩니다. 제어 컴포넌트들은 매우 유연하지만 부모 컴포넌트들이 prop들로 완전히 조작해야합니다.

실제로 "제어"와 "비제어"는 엄격한 기술용어가 아닙니다-각 컴포넌트는 대개 지역 변수와 prop들을 함께 가집니다. 하지만 이는 컴포넌트들이 어떻게 디자인되었고 어떤 기능을 제공하는지에 대해서는 유용한 방식입니다.

컴포넌트를 작성할때, 어떤 정보를 제어(prop으로)할지, 어떤 정보를 비제어할지(state로)를 고려합니다. 하지만 여러분은 이후에 마음을 바꾸고 리팩토링을 진행하게 될 것 입니다.

각 상태를 위한 하나의 사실 원천​

React 앱에서, 많은 컴포넌트들은 자신의 상태를 가집니다. 어떤 상태는 입력값들과 같이 리프 컴포넌트(트리 구조의 바닥에 존재하는 컴포넌트) 근처에 "살게"될 것 입니다. 다른 상태는 앱의 상단에 "살게"될 것입니다. 예를들어 클라이언트쪽의 라우팅 라이브러리들은 대개 React 상태 내부의 현재 라우트에 저장되어 구현되고, prop들로 내려주게 됩니다!

여러분은 각각의 유니크한 상태 조각을 "소유할" 컴포넌트를 선택할 것입니다. 이 원리는 또한 "하나의 사실 원천"을 가진다고도 알려져 있습니다. 이는 모든 상태가 하나의 장소에 위치한다는 의미가 아닙니다-하지만 각각의 상태 조각들은, 그 정보 조각을 가지는 특정 컴포넌트에 있습니다. 컴포넌트간에 중복된 상태 공유를 대신하여 여러분은 공통 부모로 올리고 필요로 하는 자식들에게 내려주게 됩니다.

여러분의 앱은 여러분이 작업하는 대로 변합니다. 여러분은 상태의 일부가 "살아 있는" 곳을 파악하는 동안 상태를 내리거나 다시 올리는 것은 일반적입니다. 모두 과정의 일부입니다!

좀 더 많은 컴포넌트들에서 실제로 어떤 느낌인지 보고싶다면 React로 생각하기를 읽어보세요.

되짚어 보기

  • 두 컴포넌트를 조정하고 싶다면, 그들의 상태를 공통 부모로 옮기세요.
  • 그 후 공통부모로부터 prop으로 정보를 내려주세요.
  • 마지막으로, 이벤트 핸들러들을 넘겨주어서 자식들이 부모의 상태를 변경할 수 있도록하세요.
  • 컴포넌트를 "제어"(prop으로) 또는 "비제어"(상태로) 생각해보는 것도 유용합니다.
문제에 도전해보세요

1. 동기화된 입력​

이 두 입력값은 독립적입니다. 두 값이 동기화되도록 만들어보세요: 하나의 입력값을 수정하는 것은 다른 입력을 같은 문자로 업데이트해야합니다. 반대의 경우도 같습니다.

힌트​

공통 부모로 상태를 올려보세요.

정답​

text 상태를 handleChange 핸들러와 함께 부모 컴포넌트로 옮기세요. 그리고 Input 컴포넌트로 prop으로 내려주세요. 이렇게 하면 두 값을 동기화할 수 있습니다.

2. 리스트 필터링하기​

이 예제에서, SearchBar는 문자 입력값을 조작하는 query 상태를 가집니다. 그의 부모 FilterableList 컴포넌트는 아이템들의 List를 표현하지만, 검색 쿼리를 고려하지 않습니다.

filterItems(foods, query) 함수를 사용하여 검색 쿼리에 따라 리스트를 필터합니다. 여러분의 변경을 테스트하기위해 입력에 "s"를 적으면 리스트를 "Sushi", "Shish kebab", "Dim sum"으로 필터하는지 확인하세요.

filterItem은 이미 구현되었고 import되었으므로 직접 작성할 필요는 없습니다.

힌트​

SearchBar에서 query 상태와 handleChange 핸들러를 제거하고 FilterableList로 옮기고 싶을 것입니다. 그 후 SearchBar로 query와 onChange prop을 내려주세요.
