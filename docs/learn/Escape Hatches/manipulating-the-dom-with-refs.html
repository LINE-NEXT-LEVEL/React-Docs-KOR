manipulating-the-dom-with-refs

Deep Dive

명령형 핸들로 API 집합 노출하기​

위의 예시에서 MyInput은 원래의 DOM input 요소를 노출합니다. 이는 부모 컴포넌트가 MyInput에 focus()를 호출하도록 합니다. 하지만 이는 또한 부모 컴포넌트가 다른 것을 할 수 있게 해줍니다 - 예를 들면 CSS 스타일을 변경하는 것이 있습니다. 흔하지 않은 경우들에서, 여러분은 노출된 기능을 제한하고 싶을 수 있습니다. 이 경우, 여러분은 useImperativeHandle로 제한할 수 있습니다.

여기서 MyInput내부의 realInputRef는 실제 input DOM 노드를 갖고 있습니다. 하지만, useImperativeHandle는 React가 여러분만의 특별한 객체를 ref의 값으로 부모에게 제공하도록 지도합니다. 따라서 Form 컴포넌트 내부의 inputRef.current는 focus 메소드만을 가집니다. 이 경우에 ref "handle"은 DOM 노드가 아니고, 여러분이 useImperativeHandle 내부에서 만든 커스텀객체입니다.

React가 refs를 부착하는 경우​

React에서 모든 업데이트는 두 방면로 나눠집니다.

  • 렌더링 동안, React는 스크린이 어떻게 구성되어야 하는지를 알아내기 위해 컴포넌트들을 부릅니다.
  • commit 동안, React는 변경사항을 DOM에 반영합니다.

일반적으로 여러분은 렌더링동안 refs에 접근 하고싶지 않을 것 입니다. 그것은 refs가 DOM 노드들을 갖고 있도록 합니다. 첫 렌더링동안, DOM 노드들은 아직 만들어지지 않았기 때문에, ref.current는 null입니다. 그리고 업데이트들을 렌더링 하는동안, DOM은 아직 업데이트 되지 않았습니다. 따라서 읽기에 너무 이른 시간입니다.

React는 ref.current를 commit 중에 설정합니다. DOM을 업데이트하기 전에, React는 영향받는 ref.current값들을 null로 설정합니다. DOM 업데이트 이후, React는 값들을 즉시 알맞은 DOM 노드들로 설정합니다.

일반적으로, 여러분은 refs에 이벤트 핸들러로 접근할 것 입니다. 만약 여러분이 ref로 무언가를 하고싶지만, 활용할 특정 이벤트가 없다면 Effect가 필요할 것입니다. 다음 페이지에서 effects에 대해 다룹니다.

Deep Dive

플러싱 상태는 flushSync로 동기적 업데이트됩니다.​

새로운 todo를 추가하고, 리스트의 마지막 자식으로 스크롤되는 코드를 생각해봅시다. 어떤 이유로, 코드는 항상 마지막에 추가된 todo 직전의 todo로 스크롤 되는지 알아봅시다.

이슈는 이 두줄입니다:

setTodos([...todos, newTodo]);
listRef.current.lastChild.scrollIntoView();

React에서, 상태 업데이트는 저장됩니다. 대체로, 이는 여러분이 원하는 것입니다. 하지만 setTodos가 즉시 DOM에 업데이트 되지않기 때문에 문제가 생깁니다. 따라서 여러분이 리스트 마지막 요소로 스크롤할때, todo는 아직 추가되지 않았습니다. 이것이 스크롤이 항상 "한발 느린"이유입니다.

이 문제를 해결하기 위해, React에게 DOM을 동기적으로 업데이트("플러쉬")하게 강제할 수 있습니다. 이를 위해서 react-dom에서 flushSync를 import하고 flushSync 호출로 상태 업데이트를 감싸면됩니다.

flushSync(() => {
  setTodos([...todos, newTodo]);
});
listRef.current.lastChild.scrollIntoView();

이는 flushSync로 감싸진 코드가 실행된 후 React가 동기적으로 DOM을 업데이트하도록 합니다. 결과적으로 마지막 TODO는 여러분이 스크롤을 시도할 때 DOM에 존재하게 됩니다.

refs로 DOM 조작하는 모범사례​

Refs는 탈출구입니다. 여러분은 반드시 "React 외부로 이동"해야할 때만 써야합니다. 흔한 예시들은 포커스, 위치 스크롤, React가 제공하지않는 브라우저 API호출 등이 있습니다.

만약 포커싱과 스크롤링같은 비구조화 하지않는 행동들을 고수한다면, 여러분은 문제가 없을 것입니다. 하지만 수동으로 DOM을 수정하려고하면, React가 만드는 변경들과 충돌할 위험이 있습니다.

이 문제를 묘사하기위해서, 아래 예제는 환영메시지와 두 버튼을 가집니다. 첫 버튼은 React가 일반적으로 하는 것 처럼 조건부 렌더링과 상태를 이용하여 토글합니다. 두번째 버튼은 remove() DOM API를 이용하여 React 제어 밖인 DOM으로부터 환영 메시지를 강제로 제거합니다. 마지막으로, "setState로 토글"을 누르세요:

여러분이 수동으로 DOM 요소를 제거한 뒤, setState를 이용해 다시 보이게 하려하면 충돌로 이어집니다. 이는 여러분이 DOM을 변경했고, React가 그 것을 어떻게 관리해야할지 알지 못하기 때문입니다.

React에 의해 관리되는 DOM 노드들의 변경을 피하세요. 수정하고, 자식에 추가하고, React에 의해 관리되는 요소들을 수정하고, 자식에 추가하거나 자식들을 제거하는 것은 일관되지않은 시각적 결과들이나 위와 같은 충돌로 이어집니다.

하지만, 여러분이 이 것을 전혀 하지못한다는 의미는 아닙니다. 조심성이 필요합니다. React가 업데이트할 이유가 없는 DOM의 일부를 안전하게 수정할 수 있습니다. 예를 들어 어떤 <div>가 JSX에서 항상 비어 있다면, React는 그 자식 리스트를 건드릴 이유가 없습니다. 따라서 그 곳에 수동으로 자식을 추가하거나 제거하는 것은 안전합니다.

요약​

  • Refs는 일반적 개념이지만, 여러분은 대체로 DOM 요소들을 갖는데 사용합니다.
  • 여러분은 React에게 <div ref={myReff}>를 넘겨 myRef.current에 DOM 노드를 넣도록 안내합니다.
  • 대체로 비구조화 하지않는 포커싱, 스크롤링, DOM요소 측정하기 등의 동작에 refs를 사용할 것 입니다.
  • 컴포넌트는 DOM 노드들을 기본적으로 노출하지 않습니다. forwardRef를 사용하거나 두번째 ref 인자를 특정 노드에게 내려주어 DOM 노드의 노출을 선택할 수 있습니다.
  • React에 의해 관리되는 DOM 노드들의 변경을 피하세요.
  • 만약 React에 의해 관리되는 DOM 노드들을 변경한다면, React가 업데이트 할일이 없는 부분을 변경하세요.
문제에 도전해보세요.

도전 1 : 비디오 재생과 정지​

이 예제에서, 버튼은 재생과 정지 상태를 전환하기위해 상태변수를 토글합니다. 하지만, 실제 재생과 정지를 위해서 토글은 충분하지 않습니다. 여러분은 또한 <video>의 DOM 요소의 play()와 pause()를 호출해야합니다. ref를 추가해서 버튼이 동작하도록 해보세요.

추가적인 도전으로는, "재생" 버튼을 유저가 비디오 우클릭이나 브라우저 미디어 제어로 재생할때도 동기화되도록 유지해보세요. 비디오가 그렇게 동작하게 하려면 여러분은 onPlay와 onPause을 구독 해야할 것 입니다.

정답​

ref를 선언하여 <video> 요소에 넣으세요. 그리고 다음 상태에 의존하는 이벤트핸들러에서 ref.current.play()와 ref.current.pause()를 호출하세요.

내장 브라우저 컨트롤들을 쓰기 위해서, 여러분은 onPlay와 onPause 핸들러들을 <video> 요소에 추가해서 setIsPlaying을 호출할 수 있습니다. 이 방법으로 유저가 브라우저 컨트롤들로 비디오를 재생하면, 상태는 알맞춰 일치됩니다.

도전 2 : 검색 영역 포커스​

"검색"버튼 클릭이 포커스를 영역으로 놓도록 만들어보세요.

정답​

ref를 input에 추가하고, DOM에서 focus()를 호출해 input을 포커싱하도록 합니다.

도전 3 : 이미지 카라셀로 스크롤하기​

이 이미지 카라셀은 활성된 이미지를 변경하는 "다음" 버튼이 있습니다. 갤러리가 클릭시에 활성된 이미지로 수평 스크롤링 하도록 만들어보세요. 이미지를 활성화하기 위해서 scrollIntoView()를 호출할 것 입니다.

node.scrollIntoView({
  behavior: 'smooth',
  block: 'nearest',
  inline: 'center',
});

힌트​

여러분은 이 예제에서 각 이미지에 ref를 가질 필요가 없습니다. 현재 활성화된 이미지나 리스트 자체의 ref가 있으면 됩니다. flushSync를 사용하여 DOM이 스크롤 이전에 업데이트 되는지를 보증하세요.

정답​

여러분은 selectedRef를 선언하고, 현재 이미지로 조건부로 전달할 수 있습니다:

<li ref={index === i ? selectedRef : null}>

이미지가 선택되었다는 의미인 index===i일 때, <li>는 selectedRef를 받습니다. React는 selectedRef.current가 항상 알맞은 DOM 노드를 겨냥한다는 것을 보증합니다.

flushSync 호출은 스크롤 전에 DOM을 업데이트하도록 React에게 강제하기위해 필수입니다. 그렇지 않으면 selectedRef.current는 항상 선택된 것 이전의 것을 겨냥할 것 입니다.

도전 4 : 분리된 컴포넌트들로 검색 영역 포커싱하기​

"검색" 버튼 클릭이 해당 영역으로 포커스를 두도록 만들어보세요. 각 컴포넌트는 다른 파일에 정의되어있고 밖으로 옮겨지면 안됩니다. 여러분은 어떻게 그들을 연결할 건가요?

힌트​

여러분은 SearchInput과 같은 고유 컴포넌트에서 DOM 노드를 노출하는 것을 선택하기 위해 forwardRef가 필요합니다.

정답​

SearchButton으로 onClick prop 추가가 필요할 것이고, SearchButton이 onClick을 브라우저 <button>으로 넘겨주도록 합니다. 또한 실제 <input>으로 넘겨주고 이동시키는 <SearchInput>으로도 ref를 내려줍니다. 마지막으로 클릭 핸들러에서, ref내부에 저장된 DOM노드에서 focus를 호출합니다.
