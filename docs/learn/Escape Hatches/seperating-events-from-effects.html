Effects에서 이벤트 분리하기

이벤트 핸들러는 여러분이 같은 상호작용을 다시 행할때만 재실행됩니다. 이벤트 핸들러와 다르게, Effects는 직전 렌더링과는 다른 값을 가지는 prop이나 상태 변수같은, Effects가 읽는 어떤 값들에 의해 재동기화합니다. 때때로, 여러분은 이 두 동작을 섞고 싶을 것입니다: 특정 값들에만 반응하여 재랜더링되는 Effect. 이 페이지에서 여러분이 어떻게 할 수 있는지를 알려드립니다.

여러분이 배울 것
  • 이벤트핸들러와 Effect간에 선택하는 법
  • Effect가 반응적이고 이벤트 핸들러는 그렇지 않은 이유
  • Effect의 코드 일부가 반응하지 않기를 원할 때 해야할 것
  • Effect 이벤트가 무엇인지와 Effects에서 그것을 분리하는 법
  • Effect 이벤트를 사용하여 최신 props와 상태를 읽는 법 :::

이벤트핸들러와 Effect간에 선택하는 법​

먼저, 이벤트 핸들러와 Effects의 차이점을 요약합니다. 여러분이 채팅방 컴포넌트를 만들고 있다고 가정합니다. 여러분의 요구사항은 다음과 같습니다:

  1. 여러분의 컴포넌트는 자동으로 선택된 채팅방에 연결됩니다.
  2. "전송" 버튼을 누르면, 채팅에 메시지를 보내야합니다.

여러분이 이미 이를 위한 코드를 만들어 두었지만, 어디에 넣어야할지 모른다고 해봅시다. 여러분이 이벤트 핸들러나 Effects를 써야할까요? 이 질문에 답을 해야할 때마다, 왜 코드가 실행되어야하는가를 고려하세요.

특정 상호작용에 반응하여 실행되는 이벤트 핸들러​

유저의 관점에서, 메시지를 보내는 것은 유저가 특정 "전송" 버튼이 클릭되어야 발생됩니다. 유저는 만약 그들의 메시지가 어떤 이유로, 어떤 다른 시간에 보내진다면 다소 짜증날 것입니다. 이는 메시지 전송이 이벤트 핸들러여야하는 이유입니다. 이벤트 핸들러는 특정 상호작용을 처리하도록 합니다.

function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');
  // ...
  function handleSendClick() {
    sendMessage(message);
  }
  // ...
  return (
    <>
      <input value={message} onChange={(e) => setMessage(e.target.value)} />
      <button onClick={handleSendClick}>Send</button>;
    </>
  );
}

이벤트 핸들러를 사용했을 때, 여러분은 sendMessage(message)가 오직 유저가 버튼을 클릭했을 때만 동작하도록 합니다.

Effects는 동기화가 필요할 때 항상 실행됩니다​

여러분이 컴포넌트와 채팅방의 연결을 유지해야 한다는 것도 상기해보세요. 코드가 어떻게 될까요?

이 코드의 실행 이유는 특정 상호작용을 위한 것이 아닙니다. 왜 혹은 어떻게 유저가 채팅방 화면으로 향하게 되었는지는 중요하지 않습니다. 지금 사용자는 채팅방 화면을 보고있고 상호작용하고 있어야 하며, 컴포넌트는 선택된 채팅서버와 연결되어 있어야합니다. 채팅방 컴포넌트가 여러분의 앱의 첫 화면이었더라도, 유저가 어떤 상호작용도 행하지 않았다해도, 여러분은 여전히 연결이 필요합니다. 이는 그것이 Effect로 수행되는 이유입니다.

function ChatRoom({ roomId }) {
  // ...
  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.connect();
    return () => {
      connection.disconnect();
    };
  }, [roomId]);
  // ...
}

이 코드로, 유저가 특정 상호작용을 수행하였는지와 상관없이 항상 현재 선택된 채팅 서버에 활성화된 연결이 있음에 확신을 가질 수 있습니다. 유저가 여러분의 앱을 열기만 했더라도, 다른 방을 선택했더라도, 다른 스크린으로 향하고 돌아왔더라도, 여러분의 Effect는 컴포넌트가 현재 선택된 채팅방과 동기화된 상태로 남는 것을 보장하고, 필요할 때 재연결할 것 입니다.

반응형 값들과 반응형 로직​

직관적으로, 여러분은 이벤트 핸들러는 항상 버튼을 클릭하는 예와 같이 "수동적으로" 트리거된다고 할 수 있습니다. Effects는 반대로 "자동으로": 이는 동기화 상태로 남아야할 필요가 있을 때마다 실행 및 재실행됩니다.

이에 대해 생각할 수 있는 더 정확한 방법이 있습니다.

여러분의 컴포넌트의 본문 내에 선언된 Props, 상태, 그리고 변수들은 반응형 값으로 불립니다. 예를들어 serverUrl은 반응형 값이 아니지만, roomId와 message는 맞습니다. 이들은 렌더링 데이터 흐름에 관여합니다.

const serverUrl = 'https://localhost:1234';

function ChatRoom({ roomId }) {
  const [message, setMessage] = useState('');

  // ...
}

이와 같은 반응형 값들은 리렌더링으로 변경될 수 있습니다. 예를 들어 유저가 message를 변경하거나 드롭다운에서 다른 roomId를 선택하는 것입니다. 이벤트핸들러와 Effects는 변경들에 다르게 반응합니다.

  • 이벤트 핸들러 내부의 로직은 비반응형입니다 그것은 유저가 같은 상호작용(클릭과 같은)을 수행하지 않는한 다시 실행되지 않습니다. 이벤트 핸들러는 반응형 값의 변경에 "반응하지" 않고 반응형 값을 읽을 수 있습니다
  • Effects 내부의 로직은 반응형입니다. 만약 여러분의 Effect가 반응형 값을 읽는다면, 종속성으로 표시하세요. 그러면, 리렌더링이 값을 변경하면, React는 Effect의 로직을 새로운 값으로 재실행합니다.

이 차이점을 묘사하기 위해 이전 예시를 다시 보겠습니다.

이벤트 핸들러 내부의 로직은 비반응형​

이 한 줄의 코드를 보세요. 이 로직이 반응형이어야할까요?

// ...
sendMessage(message);
// ...

유저의 관점에서 message의 변경이 그들이 메시지를 보내기를 원한다는 의미가 아닙니다. 이는 그저 유저가 타이핑중이라는 의미입니다. 다르게 말하면, 메시지를 보내는 것은 반응형이면 안됩니다. 이 로직은 반응형 값이 변경되었다고해서 재실행되면 안됩니다. 이 것이 이벤트 핸들러에 속하는 이유입니다.

function handleSendClick() {
  sendMessage(message);
}

이벤트 핸들러는 반응형이 아니므로, sendMessage(message)는 유저가 전송 버튼을 클릭했을 때만 실행됩니다.

Effects 내부의 로직은 반응형​

아래 코드들에 돌아와보세요:

// ...
const connection = createConnection(serverUrl, roomId);
connection.connect();
// ...

유저의 관점에서, roomId 변경은 유저가 다른 채팅방에 연결되기를 원한다는 의미입니다. 다르게 말하면, 방에 연결하는 로직은 반응형이어야합니다. 여러분은 이 코드 몇 줄로 반응형 값에 "동기화"하고 값이 달라지면 재실행하기를 원합니다. 이것이 Effect에 속하는 이유입니다:

useEffect(() => {
  const connection = createConnection(serverUrl, roomId);
  connection.connect();
  return () => {
    connection.disconnect();
  };
}, [roomId]);

Effects는 반응형이므로, createConnection(serverUrl, roomId)와 connection.connect()는 매 별개의 roomId값에 실행됩니다. 여러분의 Effect는 채팅 연결이 현재 선택된 방으로 동기화되도록 유지합니다.

비반응형 로직을 Effect에서 분리하기​

여러분이 반응형 로직과 비반응형 로직을 섞기를 원한다면 일이 더 복잡해집니다.

예를들어, 유저가 채팅에 연결되었을때 알림을 보여주고싶다고 가정해 봅시다. 여러분이 props에서 현재 테마(다크 혹은 라이트)를 읽어 알맞은 색깔로 알림을 보여줄 수 있습니다.

function ChatRoom({ roomId, theme }) {
  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.on('connected', () => {
      showNotification('Connected!', theme);
    });
    connection.connect();
    // ...

이에 반해 theme는 반응형 값(리렌더링의 결과로 변경될 수 있음)이고, 모든 Effect에 의해 읽히는 반응형 값은 의존성에 선언되어야합니다. 이제 여러분은 Effect 의존성에 theme을 작성해야 합니다.

function ChatRoom({ roomId, theme }) {
  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.on('connected', () => {
      showNotification('Connected!', theme);
    });
    connection.connect();
    return () => {
      connection.disconnect()
    };
  }, [roomId, theme]); // ✅ 모든 의존성이 선언됨
  // ...

이 예제를 확인하고, 유저 경험에 문제를 발견해보세요.

roomId가 변경될 때, 채팅은 여러분이 예상한대로 재연결됩니다. 하지만 theme 또한 의존성이 있는 값이기 때문에, 채팅 또한 매 번 다크, 라이트로 테마를 변경할 때마다 재연결됩니다.

다르게 말하면, 아래 줄이 Effect(반응형인)안에 있어도 반응형이길 원하지 않는다면:

// ...
showNotification('Connected!', theme);
// ...

이 비반응형 로직을 반응형 Effect에서 분리해야합니다.

Effect 이벤트 선언하기​

이 섹션은 아직 정식 React 버전에 릴리즈되지 않은 실험적인 API를 설명합니다. :::

useEffectEvent라 볼리는 특별한 훅을 사용하여 여러분의 Effect에서 비반응형 로직을 추출해보세요.

import { useEffect, useEffectEvent } from 'react';

function ChatRoom({ roomId, theme }) {
  const onConnected = useEffectEvent(() => {
    showNotification('Connected!', theme);
  });
  // ...

여기 onConnected는 Effect 이벤트라고 불립니다. 여러분의 Effect 로직의 일부이지만, 이벤트핸들러에 가깝게 동작합니다. 내부 로직은 비반응형이고, 항상 최신의 props와 상태값을 "관찰"합니다.

이제 여러분은 Effect 내부에서 onConnected Effect 이벤트를 호출할 수 있습니다.

function ChatRoom({ roomId, theme }) {
  const onConnected = useEffectEvent(() => {
    showNotification('Connected!', theme);
  });

  useEffect(() => {
    const connection = createConnection(serverUrl, roomId);
    connection.on('connected', () => {
      onConnected();
    });
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]); // ✅ 모든 의존성이 선언됨
  // ...

이것이 문제를 해결합니다. 여러분이 onConnected를 Effect 종속성 리스트에서 제거함을 확인하세요. Effect 이벤트들은 비반응형이고 종속성에서 생략되어야합니다.

새 동작이 여러분이 예상한대로인지 확인해보세요.

Effect 이벤트들이 이벤트 핸들러와 아주 유사하다고 생각할 수 있습니다. 주된 차이점은 이벤트 핸들러는 유저 상호작용에 반응하여 실행되는 것임에 반해, Effect 이벤트는 Effects에서 여러분에 의해 실행된다는 것입니다. Effect 이벤트는 여러분이 Effects의 반응성과 반응형이면 안되는 코드 사이의 "체인을 부수도록" 해줍니다.

Effect 이벤트로 최신의 props와 상태를 읽기​

Under Construction

이 섹션은 안정 React 버전에서 릴리즈되지 않은 실험적인 API를 설명합니다.

Effect 이벤트는 여러분이 종속성 linter를 억제하기 위해 시도했던 많은 패턴들을 수정할 수 있도록 합니다.

예를들어 Effect가 페이지 방문들을 기록한다고 해봅시다:

function Page() {
  useEffect(() => {
    logVisit();
  }, []);
  // ...
}

후에, 여러분의 사이트에 여러 라우트를 추가하게 되었습니다. 이제 여러분의 Page 컴포넌트는 url prop에 현재 경로를 받습니다. 여러분은 url을 logVisit호출의 부분으로 넘기고 싶지만, 종속성 linter가 불만을 표시합니다:

function Page({ url }) {
  useEffect(() => {
    logVisit(url);
  }, []); // 🔴 React Hook useEffect가 'url' 종속성을 갖고 있지 않습니다.
  // ...
}

여러분이 코드가 어떻게 동작했으면 하는지 생각해보세요. 여러분은 다른 페이지를 의미하는 다른 URL들로 방문을 구분하여 기록하고 싶습니다. 다르게 말하면, logVisit 호출은 반드시 url에 반응하여야합니다. 이 경우에, 종속성 linter를 따르는 것과 url을 종속성에 추가하는 것이 합당한 이유입니다.

function Page({ url }) {
  useEffect(() => {
    logVisit(url);
  }, [url]); // ✅ 모든 의존성이 선언됨
  // ...
}

이제 여러분이 모든 페이지 방문에서 쇼핑카트에 담은 물품들을 포함하고 싶다고 해봅시다:

function Page({ url }) {
  const { items } = useContext(ShoppingCartContext);
  const numberOfItems = items.length;

  useEffect(() => {
    logVisit(url, numberOfItems);
  }, [url]); // 🔴 React Hook useEffect가 'numberOfItems' 종속성을 갖고 있지 않습니다.
  // ...
}

여러분은 numberOfItems를 Effect 내부에 사용하였으므로 linter는 종속성에 추가하라고 요구합니다. 하지만, 여러분은 logVisit이 numberOfItems에 반응하여 호출되기를 원하지 않습니다. 만양 유저가 쇼핑카트에 무언가를 추가하여 numberOfItems가 변경되면, 이는 유저가 그 페이지를 재방문 했다는 의미가 아닙니다. 다르게 말하면, 페이지를 방문하는 것은 "이벤트"에 가깝습니다. 특정한 시간에 발생하게 됩니다.

코드를 두 파트로 분리합니다:

function Page({ url }) {
  const { items } = useContext(ShoppingCartContext);
  const numberOfItems = items.length;

  const onVisit = useEffectEvent((visitedUrl) => {
    logVisit(visitedUrl, numberOfItems);
  });

  useEffect(() => {
    onVisit(url);
  }, [url]); // ✅ 모든 의존성이 선언됨
  // ...
}

여기서 onVisit는 Effect 이벤트입니다. 내부의 코드는 반응형이 아닙니다. 이것이 numberOfItems(또는 다른 반응형 값)를 주위를 둘러싼 코드가 변경에 재실행될 걱정없이 사용할 수 있는 이유입니다.

반면에, Effect는 그 자체가 반응형으로 납습니다. Effect 내부 코드는 url prop을 사용하고, Effect는 매번 다른 url으로 리렌더링되면 재실행됩니다. 결과로, onVisit Effect 이벤트를 호출하게 도비니다.

결과적으로, 여러분은 logVisit를 url변경마다 부르고, 최신의 numberOfItems를 읽게됩니다. 하지만 만약 numberOfItems가 변경된다면 코드의 재실행이 발생하지 않습니다.

Note

onVisit()를 인자들 없이 호출하고 내부에서 url을 읽을 수 있는지 궁금할 것 입니다.

const onVisit = useEffectEvent(() => {
  logVisit(url, numberOfItems);
});

useEffect(() => {
  onVisit();
}, [url]);

이는 동작하지만, Effect 이벤트에 정확히 url을 전달하는 것이 낫습니다. url을 Effect 이벤트의 인자로 전달하여서, 여러분은 다른 url로 페이지를 방문하는 것이, 유저 관점에서 별도의 "이벤트"를 구성한다는 의미입니다. visitedUrl은 발생한 "이벤트"의 부분입니다.

const onVisit = useEffectEvent((visitedUrl) => {
  logVisit(visitedUrl, numberOfItems);
});

useEffect(() => {
  onVisit(url);
}, [url]);

여러분의 Effect 이벤트가 정확히 visitedUrl에 대해 "묻기"때문에, 이제 Effect의 의존성들에서 url을 임의로 제거할 수 없습니다. 만약 url을 종속성에서 제거한다면(별도의 페이지 방문이 하나로 측정될 것), linter는 여러분에게 그것을 경고할 것입니다. url에 따라 onVisit가 반응하길 원한다면, 내부에서 url을 읽는 것 대신에 Effect에서 넘겨주면됩니다.

Effect 내부에 비동기적인 로직이 있다면 이 내용이 중요해집니다.

const onVisit = useEffectEvent((visitedUrl) => {
  logVisit(visitedUrl, numberOfItems);
});

useEffect(() => {
  setTimeout(() => {
    onVisit(url);
  }, 5000); // 방문 여부의 기록을 지연시킵니다.
}, [url]);

여기서, onVisit 내부의 url은 최신의 url(이미 변경된)과 일치하지만, visitedUrl은 이 Effect를 발생시킨(그리고 이 onVisit를 호출한) 원래의 url과 일치합니다.

DEEP DIVE

의존성 lint를 무시해도 괜찮을까요?​

여러분은 아래 코드에서처럼 일부 코드에서 lint 규칙이 무시되고 있는걸 확인할 수 있습니다.

function Page({ url }) {
  const { items } = useContext(ShoppingCartContext);
  const numberOfItems = items.length;

  useEffect(() => {
    logVisit(url, numberOfItems);
    // 🔴 아래와 같이 lint가 무시됩니다.
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, [url]);
  // ...
}

useEffectEvent가 React에서 안정적인 부분이 된 이후로부터 우리는 절대 lint를 무시하지 않는 것을 권장하고 있습니다.

Lint 규칙을 무시함으로써 발생하는 첫번째 부작용은, 여러분이 새로 작성한 의존성에 대해서 "반응"해야 한다고 React가 경고하지 않는다는 것입니다. 이전의 예시에서 여러분은 url을 의존성 배열에 추가해 React가 수정사항을 알려주길 원했습니다. 만약 lint를 끈다면 미래에 생길 수정사항에 대해 React가 더 이상 알려주지 않습니다. 그리고 이는 버그로 이어질 것입니다.

Lint를 껐을 때 생기는 또 다른 혼란스러운 버그도 있습니다. 예를 들어 handleMove 함수는 현재의 canMove 상태 변수를 읽고 커서를 따라갈지 말지를 결정하게끔 작성되어 있습니다. 하지만 canMove는 handleMove 안에서 항상 true일 것입니다.

왜 그런지 이해하셨나요?

이 코드의 문제점은 의존성을 확인하는 lint가 꺼져있다는 것입니다. 만약 이 억제를 제거한다면 이 Effect는 handleMove 함수에 의존하게 됩니다. 이건 말이 되죠. handleMove는 컴포넌트 내부에 선언되어 있기 때문에 반응형 변수라고 볼 수 있습니다. 모든 반응형 변수는 항상 의존성으로 관리되어야 합니다. 그렇지 않다면 언젠가 상한(stale) 상태가 될 것입니다!

이 코드를 작성한 사람은 React에게 Effect는 어떤 반응형 변수에도 의존하지 않는다고 "거짓말"을 했습니다. 이로 인해 React는 canMove 함수(그리고 handleMove 함수)가 변경된 후에도 다시 싱크를 맞추지 않았던 것입니다. React가 Effect와 다시 싱크를 맞추지 않는다면, 리스너에 부착된 handleMove 함수는 최초에 렌더링되었을 때 생성된 handleMove일 것입니다. 그리고 이 handleMove 함수는 첫 렌더링시에 canMove가 true인 상태일때 만들어진 함수이므로, 영원히 canMove가 true인 채로 기억하고 있을 것입니다.

lint를 무시하지 않는다면, 여러분은 상한(stale) 변수로 인해 발생하는 문제점들을 볼 일이 없을 것입니다.

useEffectEvent와 함께라면, lint에게 "거짓말"할 필요가 없고, 코드는 여러분이 예상한대로 동작할 것입니다.

useEffectEvent가 항상 최고의 해답임을 의미하는 것은 아닙니다. lint를 끄는 것은 리액티브하게 동작하지 않는 코드에만 해야 합니다. 그리고 위 코드에서는 canMove에 대해 Effect가 리액티브 하게 동작했으면 합니다. 이것이 Effect 이벤트를 추출한 이유입니다.

Effect의 의존성 제거하기를 통해 lint를 억제하는 다른 올바른 대안에 대해서도 알아보세요.

Effect 이벤트의 제한사항​

작성중!

아래 섹션은 실제 안정된 React 버전으로 배포되지 않은 실험적 API에 대해 다루고 있습니다.

Effect 이벤트는 여러분이 어떻게 사용하느냐에 따라 매우 제한적입니다.

  • Effect 내부에서만 호출해야 합니다.
  • 다른 컴포넌트나 hook에 전달해서는 안됩니다.

예를 들어, 아래처럼 Effect 이벤트를 선언하거나 전달해서는 안됩니다.

function Timer() {
  const [count, setCount] = useState(0);

  const onTick = useEffectEvent(() => {
    setCount(count + 1);
  });

  useTimer(onTick, 1000); // 🔴 주의: Effect 이벤트를 전달하지 마세요!

  return <h1>{count}</h1>
}

function useTimer(callback, delay) {
  useEffect(() => {
    const id = setInterval(() => {
      callback();
    }, delay);
    return () => {
      clearInterval(id);
    };
  }, [delay, callback]); // 의존성 배열에 "callback"이 무엇인지 구체적으로 적어야 합니다.
}

이 대신, Effect 이벤트를 항상 이를 사용하는 Effect의 옆에 선언하세요.

function Timer() {
  const [count, setCount] = useState(0);
  useTimer(() => {
    setCount(count + 1);
  }, 1000);
  return <h1>{count}</h1>
}

function useTimer(callback, delay) {
  const onTick = useEffectEvent(() => {
    callback();
  });

  useEffect(() => {
    const id = setInterval(() => {
      onTick(); // ✅ Effect 내부에서 지역적으로만 호출하세요.
    }, delay);
    return () => {
      clearInterval(id);
    };
  }, [delay]); // "onTick" (Effect Event)가 무엇인지 의존성 배열에 명시할 필요가 없습니다.
}

Effect 이벤트는 리액티브 하지 않은 Effect 코드의 "조각"입니다. 이들은 항상 이를 사용하는 Effect의 옆에 있어야 합니다.

되짚어보기​

  • 이벤트 핸들러는 특정한 상호작용에 반응함으로써 동작합니다.
  • Effect는 싱크가 필요한 경우에 동작합니다.
  • 이벤트 핸들러 내부에 있는 로직은 리액티브 하지 않습니다.
  • Effect 내부에 있는 로직은 리액티브 합니다.
  • 리액티브하지 않은 로직을 Effect에서 Effect 이벤트로 옮길 수 있습니다.
  • Effect 이벤트는 오직 Effect 내부에서만 호출되어야 합니다.
  • Effect 이벤트를 다른 컴포넌트나 훅에 전달하면 안됩니다.
문제에 도전해 보세요.

1. 업데이트 되지 않는 변수 고치기​

이 Timer 컴포넌트는 매 초마다 꾸준히 증가하는 count 상태 변수를 가지고 있습니다. 이 변수의 증가하는 만큼은 increment 상태 변수에 저장되고 있습니다. increment 상태 변수는 '+' 혹은 '-' 버튼을 통해 조절할 수 있습니다.

하지만 '+' 버튼을 아무리 많이 클릭해도 카운터는 매 초마다 하나씩만 증가할 뿐입니다. 이 코드는 무엇이 잘못되었을까요? 왜 increment는 항상 Effect 코드 안에서 1일까요? 실수를 찾아 고쳐보세요.

힌트 이 코드를 고치기 위해서는 규칙을 따르는 것으로도 충분합니다.

정답 평소처럼 여러분은 Effect 안에서 버그를 찾기 시작할겁니다. linter 억제가 잘못되진 않았는지부터 확인해보세요.

만약 lint를 억제하는 주석을 제거하면, React는 여러분에게 이 Effect 코드의 의존성이 increment에 있지만 여러분이 React에게 이 코드가 어떤 리액티브한 변수에도 의존하지 않고 있다고([]) 거짓말 하고 있음을 알려줄 것입니다. increment 변수를 의존성 배열에 추가해주세요.

이제 increment가 변하면 React는 다시 Effect와 싱크를 맞추고, interval을 초기화 시킬 것입니다.

2. 멈춰버린 카운터 고치기​

이 Timer 컴포넌트는 매 초마다 증가하는 count 상태 변수를 가지고 있습니다. 이 변수의 증가하는 만큼은 increment 상태 변수에 저장되고 있습니다. increment 상태 변수는 '+' 혹은 '-' 버튼을 통해 조절할 수 있습니다. 예를 들어, '+' 버튼을 아홉번 누르면 count는 매 초마다 1이 아닌 10만큼 증가됩니다.

하지만 작은 이슈가 있습니다. '+' 버튼 혹은 '-' 버튼을 초당 1회 이상 누르게 되면 타이머는 멈춘 것처럼 보입니다. 이는 오직 여러분이 어떤 버튼을 건드리든 1초 이후에 다시 재개됩니다. 왜 이런일이 일어나는지 알아보고, 타이머가 방해 없이 정확히 매 초 1씩 증가할 수 있도록 고쳐보세요.

힌트 Effect가 increment에만 "반응"하도록 작성되어 있는것 같습니다. setCount를 호출하기 위해 필요한 현재의 increment를 사용하는 라인은 정말 리액티브 해야 할까요?

정답 이 이슈는 Effect 내부의 코드가 increment 상태 변수를 사용하기 때문에 발생합니다. increment가 Effect의 의존성 배열에 있는 한 increment의 모든 업데이트는 Effect의 싱크를 다시 맞추게 됩니다. 이는 interval을 초기화하도록 만듭니다. 만약 이 이벤트가 발생하기 전에 계속 interval을 초기화 한다면, 타이머가 멈춘 것처럼 보일 것입니다.

이 문제를 해결하기 위해서는 onTick Effect 이벤트를 Effect 밖으로 추출해야 합니다.

onTick이 Effect 이벤트이므로, 그 내부는 리액티브 하지 않습니다. increment의 변화는 어떤 Effect도 트리거하지 않습니다.

3. 조절 불가능한 딜레이를 고쳐보세요.​

이 예시에서 여러분은 interval의 딜레이를 조절할 수 있습니다. 이는 delay 상태 변수에 저장되고 두개의 버튼에 의해 업데이트 됩니다. 하지만 만약 delay가 1000 밀리세컨드(1초입니다)가 되도록 "plus 100 ms" 버튼을 누른다면 여전히 타이머가 매우 빠르게(매 100ms마다) 증가됨을 확인할 수 있습니다. 이는 여러분이 delay를 바꾸지 않은 것처럼 보입니다. 버그를 찾고 해결해보세요.

힌트 Effect 이벤트 내부의 코드는 리액티브 하지 않습니다. setInterval을 다시 호출하길 원하는 하는 다른 경우들이 있었나요?

정답 위 예시의 문제점은 onMount 코드가 실제로 무엇을 하는지 고려하지 않고 Effect 이벤트로 추출했다는 것입니다. Effect 이벤트는 코드의 일부를 리액티브 하지 않게 만들려 하는 등의 특별한 이유가 있을때만 추출되어야 합니다. 하지만 setInterval의 호출은 delay 상태 변수의 관점에서는 리액티브 해야 합니다. delay가 변경되면 interval을 다시 시작하고 싶어하기 때문입니다. 이를 해결하기 위해서는 리액티브한 코드를 다시 Effect의 내부로 이동시켜야 합니다.

일반적으로 onMount와 같이 타이밍을 코드의 목적보다 더 집중하는 함수들에 대해 의심해야 합니다. 처음에는 "더 설명적"이라고 느껴질 수도 있지만 의도를 흐리게 합니다. 경험상 Effect 이벤트들은 유저의 관점에서 반응하는 함수들입니다. 예를 들어, onMessage, onTick, onVisit이나 onConnect 같은 것들은 좋은 Effect 이벤트의 이름들입니다. 이들의 내부 코드들은 대부분 리액티브할 필요가 없습니다. 반면에 onMount, onUpdate, onUnmount나 onAfterRender 같은 함수들은 너무 일반적이라 실수로 리액티브 해야만 하는 코드들을 그 안에 넣을 수도 있습니다. 코드가 실행되었을 때의 동작이 아닌 이벤트가 발생했을때 유저가 어떻게 생각할지를 고려해 이름을 붙어야 하는 이유입니다.

4. 늦은 알림 고치기​

여러분이 채팅방에 입장했을 때 컴포넌트는 알림을 하나 보여줘야 합니다. 하지만 컴포넌트는 알림을 즉시 보여주지 않습니다. 대신 유저가 UI를 둘러볼 수 있도록 알림은 의도적으로 2초정도 딜레이 되어 있습니다.

이는 거의 동작하지만, 버그가 하나 있습니다. 드롭다운을 "general"에서 "travel"로 바꾸고, 또 다시 "music"으로 매우 빠르게 바꿔보세요. 충분히 빠르게 조작했다면 여러분은 (예상과 같이) 두개의 알림을 보게 되지만, 두 알림은 모두 "Welcome to music"이라고 나타납니다.

"general"에서 "travel"로, 그리고 "music"으로 매우 빠르게 변경했을 때 두개의 알림이 보이되 첫번째 알림은 "Welcome to travel"로, 두번째는 "Welcome to music"으로 보이도록 수정해보세요. (추가 도전 과제로 이미 알림에 올바른 채팅방이 표시되었다고 가정하고 코드를 변경하여 마지막 알림만 표시되도록 해보세요.)

힌트 여러분의 Effect는 어떤 채팅방과 연결되었는지 이미 알고 있습니다. Effect 이벤트에 전달하고 싶은 추가적인 정보가 더 있나요?

정답 여러분의 Effect 이벤트 내부에서는 roomId가 시간 Effect 이벤트가 호출되었을 때의 값으로 남아 있습니다.

Effect 이벤트는 2초의 딜레이를 두고 호출됩니다. 만약 travel에서 music 채팅방으로 빠르게 이동하면, 그때까지 travel 채팅방의 알림이 보여지고, roomId는 이미 "music"입니다. 이것이 두 알림이 모두 "Welcome to music"을 보여주는 이유입니다.

이 이슈를 수정하기 위해서는, 최신의 roomId를 Effect 이벤트 내부에서 읽도록 하는 것이 아니라, 아래의 connectedRoomId처럼 Effect 이벤트의 파라미터로 만들어야 합니다. 그리고 roomId를 onConnect(roomId)를 호출함으로써 전달합니다.

roomId를 "travel"로 변경한 Effect는("travel" 채팅방에 연결되었습니다) "travel"의 알림을 보여줄 것입니다. roomId를 "music"으로 변경한 Effect는("music" 채팅방에 연결되었습니다) "music"의 알림을 보여줄 것입니다. 다른 말로 하면 Effect로부터 추출한(리액티브한) connectedRoomId는 항상 최신의 theme 변수를 사용하게 됩니다.

추가 과제를 해결하기 위해서는 알림의 timeout ID를 저장하고 Effect의 cleanup function을 통해 초기화해줍니다.

이렇게 하면 채팅방을 변경할때 이미 예약된(아직 노출되지 않은) 알림이 취소됩니다.
