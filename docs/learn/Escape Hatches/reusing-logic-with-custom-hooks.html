커스텀 훅으로 로직 재사용하기

React는 useState, useContext, useEffect와 같은 내장 훅들을 갖고 있습니다. 때때로, 여러분은 더 구체적인 목적을 가진 훅이 있었으면 할 것 입니다: 예를 들어 데이터를 가져오기 위한 것, 유저가 온라인 상태인지를 지속적으로 확인하는 것, 채팅방에 연결하는 것 입니다. 이 훅들을 React에서 찾을 순 없지만, 어플리케이션의 요구사항에 따라 여러분의 훅을 작성할 수 있습니다.

여러분이 배울 것
  • 커스텀 훅이 무엇인지와 어떻게 작성할 수 있는지
  • 컴포넌트간의 로직을 재사용 하는 법
  • 커스텀 훅의 이름과 구조를 정하는 법
  • 커스텀 훅을 왜, 언제 추출하는지

여러분이 네트워크에 크게 의존하는 앱(대부분의 앱이 그렇듯이)을 개발하고 있다고 상상해봅시다. 여러분이 유저들이 앱을 사용하는 중에 그들의 네트워크 연결이 우연히 끊기면 경고를 보내고 싶습니다. 어떻게 하시겠습니까? 여러분의 컴포넌트에는 두 가지가 필요할 것으로 보입니다.

  1. 네트워크 온라인 여부를 추적하는 상태 하나

  2. 전역적으로 online과 offline이벤트들 구독하고 상태를 업데이트하는 Effect

    이걸로 여러분의 컴포넌트를 네트워크 상태와 동기화 되도록할 수 있습니다. 아래와 같이 시작할 수 있습니다.

네트워크를 켜고 꺼보시고, 이 StatusBar가 여러분의 행동에 반응하여 어떻게 업데이트 되는지 살펴보세요.

또한 여러분이 다른 컴포넌트에서 같은 로직을 사용하고 싶다고 상상해봅시다. 네트워크가 꺼진 동안 “저장”대신 “연결중…”을 보이고 비활성화되는 저장버튼을 구현하고 싶습니다.

시작을 위해, isOnline상태와 Effect를 SaveButton에 복사 & 붙여넣기하세요.

확인을 위해, 네트워크를 끄면 버튼은 형태를 바꿀 것입니다. 이 두 컴포넌트들은 잘 동작하지만, 그들 사이의 로직의 중복은 별로입니다. 비록 다른 생김새를 가졌더라도 여러분은 로직을 재사용하고 싶을 것입니다.

컴포넌트로부터 여러분만의 커스텀 훅을 추출하기​

[useState](https://react.dev/reference/react/useState)와 [useEffect](https://react.dev/reference/react/useEffect)와 유사하게 useOnlineStatus 내장 Hook이 있다고 상상해 봅시다. 여러분은 두 컴포넌트들을 간소화하고 둘 사이의 중복을 제거할 수 있습니다.

function StatusBar() {
  const isOnline = useOnlineStatus();
  return <h1>{isOnline ? '✅ Online' : '❌ Disconnected'}</h1>;
}

function SaveButton() {
  const isOnline = useOnlineStatus();

  function handleSaveClick() {
    console.log('✅ Progress saved');
  }

  return (
    <button disabled={!isOnline} onClick={handleSaveClick}>
      {isOnline ? 'Save progress' : 'Reconnecting...'}
    </button>
  );
}

비록 그런 내장 훅은 없었지만, 여러분이 직접 작성할 수 있습니다. useOnlineStatus라는 함수를 선언하고, 여러분이 이전에 작성한 컴포넌트에서 모든 중복된 코드를 옮겨보세요.

function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(true);
  useEffect(() => {
    function handleOnline() {
      setIsOnline(true);
    }
    function handleOffline() {
      setIsOnline(false);
    }
    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);
    return () => {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);
  return isOnline;
}

함수의 마지막에는 isOnline을 리턴하세요. 이는 여러분의 컴포넌트가 그 값을 읽도록 합니다:

네트워크를 켜고 끄는 것이 두 컴포넌트들을 업데이트한다는걸 확인해보세요. 이제 여러분이 컴포넌트는 중복되는 로직이 별로 없습니다. 더 중요한 것은, 내부의 코드가 수행 방식(브라우저 이벤트 구독)보다 수행하려는 작업(온라인 상태 사용!)을 묘사한다는 것입니다.

여러분이 커스텀 Hooks로 로직을 추출하면, 여러분이 외부 시스템 혹은 브라우저 API를 다루는 방식에 대한 별로라 느껴지는 세부사항들을 숨길 수 있습니다. 여러분의 컴포넌트의 코드는 여러분의 의도를 표현하지 구현을 표현하지 않습니다.

Hook 이름은 언제나 use로 시작합니다.​

React 어플리케이션은 컴포넌트로 이루어져있습니다. 컴포넌트는 내장, 커스텀에 상관없이 Hooks로 작성됩니다. 여러분은 대개 다른사람들이 만든 커스텀 Hooks를 사용할 것이지만, 때로는 직접 작성하기도 할 것 입니다.

여러분은 다음의 네이밍 컨벤션들을 따라야합니다:

  1. React 컴포넌트 이름은 StatusBar, SaveButton과 같이 대문자로 시작되어야합니다. React 컴포넌트는 또한 JSX처럼 React가 표현할줄 아는 것들을 리턴해야합니다.
  2. Hook 이름은[useState](https://react.dev/reference/react/useState)(내장) 또는 useOnlineStatus(커스텀, 이 페이지에서처럼) 와 같이 use로 시작하고 대문자가 따라와야합니다. Hook은 임의의 값을 반환합니다.

이 컨벤션은 여러분이 컴포넌트를 볼 때, 상태, Effects 그리고 “숨겨진” React 기능들이 어디에 있는지 알 수 있게 합니다. 예를 들어 여러분이 getColor() 함수를 컴포넌트 내부에서 본다면, 그 것이 use로 시작되지 않기 때문에 React 상태를 포함하고 있을 가능성이 없다는 것을 확신할 수 있습니다. 하지만 useOnlineStatus()와 같이 불리는 함수는 대부분 내부에 다른 Hooks에 대한 호출을 포함할 것 입니다.

Note

여러분의 린터가 React를 위해 구성되었다면, 이 네이밍 컨벤션을 강조할 것 입니다. 위의 샌드박스로 스크롤을 올려보고 useOnlineStatus를 getOnlineStatus로 변경해보세요. 린터가 그 내부에 useState나 useEffect를 더 이상 호출하지 못하도록함을 확인하세요. 오직 Hooks와 컴포넌트들만이 다른 Hooks를 호출할 수 있습니다!

Deep Dive

렌더링에 호출되는 모든 함수들은 use 접두사로 시작되어야 하나요?​

아닙니다. Hook을 호출하지 않는 함수들은 Hooks가 될 필요가 없습니다.

여러분의 함수가 Hooks를 호출하지 않는다면, use 접두사를 피하세요. 대신에 use 접두사 없이 일반적인 접두사로 작성하세요. 예를들어 아래의 useSorted는 Hooks를 호출하지 않으므로, 대신 getSorted라고 명명하세요:

// 🔴 Avoid: Hooks를 사용하지 않는 Hook
function useSorted(items) {
  return items.slice().sort();
}

// ✅ Good: Hooks를 사용하지 않는 일반 함수
function getSorted(items) {
  return items.slice().sort();
}

이로서 여러분은 이 일반 함수를 조건들을 포함하여 어디서나 호출할 수 있음을 보장할 수 있습니다:

function List({ items, shouldSort }) {
  let displayedItems = items;
  if (shouldSort) {
    // ✅ Hook이 아니기 때문에 getSorted()를 조건부로 호출할 수 있습니다
    displayedItems = getSorted(items);
  }
  // ...
}

내부에 최소 하나의 Hook을 사용하는 함수에는 use 접두사를 부여하세요(그렇게 Hook을 만드세요):

// ✅ Good: 다른 Hooks를 사용하는 Hook
function useAuth() {
  return useContext(Auth);
}

기술적으로, React에 의해 강제되는 사항은 아닙니다. 원리적으로 다른 Hook을 부르지 않는 Hook을 만들 수 있습니다. 이는 종종 혼란스럽거나 제한적이기 때문에 그 패턴을 피하는 것이 최선입니다. 하지만 도움이 되는 흔하지 않은 케이스들이 있기도 합니다. 예를 들어 여러분의 함수가 어떠한 Hook을 사용하지 않지만, 미래에 Hook 호출들을 추가할 계획이 있는 경우입니다. 그런 경우, use 접두사를 붙이는 것이 합리적입니다:

// ✅ Good: 추후에 다른 Hooks를 사용할 것으로 예상되는 Hook
function useAuth() {
  // TODO: 인증이 구현되면 이 코드로 교체할 것:
  // return useContext(Auth);
  return TEST_USER;
}

따라서 컴포넌트들은 조건부로 호출될 수 없어지게 됩니다. 이는 내부에 Hook 호출들을 추가하면 중요해질 것 입니다. 나중에 Hooks를 내부에 사용할 계획이 아니라면(지금 혹은 나중에), Hook으로 만들지 마세요.

커스텀 Hooks는 상태가 아니라 상태와 관련된 로직을 공유하도록 합니다.​

이전의 예시에서, 여러분이 네트워크를 켜고 껐을 때, 두 컴포넌트는 함께 업데이트 되었습니다. 하지만 그 사이에 하나의 isOnline상태가 공유되는 것은 잘못된 것입니다. 이 코드를 보세요:

function StatusBar() {
  const isOnline = useOnlineStatus();
  // ...
}

function SaveButton() {
  const isOnline = useOnlineStatus();
  // ...
}

이는 여러분이 중복을 제거하기 전과 같이 동작합니다:

function StatusBar() {
  const [isOnline, setIsOnline] = useState(true);
  useEffect(() => {
    // ...
  }, []);
  // ...
}

function SaveButton() {
  const [isOnline, setIsOnline] = useState(true);
  useEffect(() => {
    // ...
  }, []);
  // ...
}

이는 완전히 독립된 상태 변수와 Effects입니다! 여러분이 같은 외부 변수(네트워크가 켜졌는지 여부)로 동기화하였기 때문에 같은 시간에 같은 값을 가져 발생하였습니다.

이를 더 잘 묘사하기 위해서, 우리는 다른 예시가 필요합니다. Form 컴포넌트를 고려해보세요:

각 폼 필드에 일부 중복된 로직이 있습니다.

  1. 상태가 몇 개 있습니다(firstName 과 lastName)
  2. 변경 핸들러가 있습니다(handleFirstNameChange와 handleLastNameChange)
  3. 입력 값에 대해 value와 onChange 속성 값을 묘사하는 JSX가 있습니다.

여러분은 useFormInput 커스텀 Hook에 반복적인 로직을 추출할 수 있습니다.

그것이 오직 value라는 하나의 상태 변수만을 선언한다는 것을 확인하세요.

하지만 Form컴포넌트는 useFormInput을 두번 호출합니다.

function Form() {
  const firstNameProps = useFormInput('Mary');
  const lastNameProps = useFormInput('Poppins');
  // ...

이것이 두가지 별개의 상태 변수들을 선언하는 것과 같이 동작하는 이유입니다.

커스텀 Hooks는 상태 자체가 아니라 상태와 관련된 로직을 공유하도록 합니다. Hook으로의 각 호출은 완전히 같은 Hook에서의 다른 호출과 독립적입니다. 이것이 위 두 샌드박스들이 완전히 같은 이유입니다. 원하신다면, 다시 스크롤해 올라가 비교해보세요. 커스텀 Hook을 호출하기 전후 동작은 동일합니다.

상태 자체를 다수의 컴포넌트들간에 공유해야한다면, 대신에 그것을 위로 올리고 아래로 전달해주세요

Hooks 간에 반응형 변수들을 전달하기​

여러분의 커스텀 Hooks 안의 코드는 컴포넌트가 리렌더링 될 때마다 재실행됩니다. 컴포넌트와 유사하게 커스텀 Hooks가 순수해야하기때문입니다. 커스텀 Hooks의 코드를 컴포넌트 본문의 일부라고 생각하세요.

커스텀 Hooks가 컴포넌트와 동시에 리렌더링되기 때문에, 항상 최신의 props와 상태를 받습니다. 이것이 의미하는 바를 보려면, 이 채팅방 예시를 생각해보세요. 서버 URL이나 채팅방을 변경해보세요.

여러분이 serverUrl이나 roomId를 변경하면, Effect는 여러분의 변경들에 “반응”하고 다시 동기화됩니다. 여러분의 Effect의 종속 값들을 변경할때 마다 채팅이 재연결된다는 걸 콘솔 메시지로 확인할 수 있습니다.

이제, Effect의 코드를 커스텀 Hook으로 옮겨봅시다

export function useChatRoom({ serverUrl, roomId }) {
  useEffect(() => {
    const options = {
      serverUrl: serverUrl,
      roomId: roomId,
    };
    const connection = createConnection(options);
    connection.connect();
    connection.on('message', (msg) => {
      showNotification('New message: ' + msg);
    });
    return () => connection.disconnect();
  }, [roomId, serverUrl]);
}

이걸로 여러분의 ChatRoom 컴포넌트가 내부 동작이 어떤지에 대한 걱정없이 여러분의 커스텀 Hook을 호출하게 합니다:

export default function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  useChatRoom({
    roomId: roomId,
    serverUrl: serverUrl,
  });

  return (
    <>
      <label>
        Server URL:
        <input value={serverUrl} onChange={(e) => setServerUrl(e.target.value)} />
      </label>
      <h1>Welcome to the {roomId} room!</h1>
    </>
  );
}

이게 훨씬 간단해 보이네요!(같은 것이지만요) 로직이 prop과 상태 변경들에 여전히 반응하는 것을 확인하세요. 서버 URL을 변경하거나 선택된 방을 수정해보세요:

하나의 Hook의 반환 값을 어떻게 취하는지 확인하세요.

export default function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  useChatRoom({
    roomId: roomId,
    serverUrl: serverUrl
  });
  // ...

그리고 다른 Hook의 입력 값으로 넘겨주세요.

export default function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  useChatRoom({
    roomId: roomId,
    serverUrl: serverUrl
  });
  // ...

ChatRoom 컴포넌트가 재렌더링 될 때마다, 그것은 최신의 roomId와 serverUrl을 여러분의 Hook에 넘겨줍니다. 여러분의 Effect가 리렌더링 후에 값들이 다를 때 재연결하는 이유입니다. (만약 음향이나 비디오 프로세싱 소프트웨어로 작업해보았다면, 이와 같이 Hooks를 연결하는 것은 시각 또는 음향 효과들을 연결하는 걸 떠오르게 할 것입니다. useState의 결과 값이 useChatRoom`의 입력 값으로 “먹여 주는 것”과 같습니다.)

작업중

이 영역은 안정된 React 버전에 배포되지 않은 실험적인 API를 설명합니다.

useChatRoom을 더 많은 컴포넌트들에서 사용하기 시작할 때, 컴포넌트들이 그 동작은 커스텀하기를 원할 것 입니다. 예를 들어, 지금은 메시지가 도착했을 때 어떻게 동작해야 하는지 Hook내부에 하드코딩되어 있습니다.

export function useChatRoom({ serverUrl, roomId }) {
  useEffect(() => {
    const options = {
      serverUrl: serverUrl,
      roomId: roomId,
    };
    const connection = createConnection(options);
    connection.connect();
    connection.on('message', (msg) => {
      showNotification('New message: ' + msg);
    });
    return () => connection.disconnect();
  }, [roomId, serverUrl]);
}

여러분이 컴포넌트에 로직을 되돌리길 원하다고 가정합시다.

export default function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  useChatRoom({
    roomId: roomId,
    serverUrl: serverUrl,
    onReceiveMessage(msg) {
      showNotification('New message: ' + msg);
    }
  });
  // ...

동작하게 하기 위해서는, 여러분의 커스텀 Hook이 명명된 옵션중 하나로 onReceiveMessage를 받도록 변경하세요.

export function useChatRoom({ serverUrl, roomId, onReceiveMessage }) {
  useEffect(() => {
    const options = {
      serverUrl: serverUrl,
      roomId: roomId,
    };
    const connection = createConnection(options);
    connection.connect();
    connection.on('message', (msg) => {
      onReceiveMessage(msg);
    });
    return () => connection.disconnect();
  }, [roomId, serverUrl, onReceiveMessage]); // ✅ 모든 종속성이 선언됨
}

동작은 하겠지만, 여러분의 커스텀 Hook이 이벤트 핸들러들을 받을 때 할 수 있는 하나의 개선이 더 있습니다.

onReceiveMessage 종속성을 추가하는 것은 컴포넌트를 다시 렌더링 할때마다 채팅 연결을 다시 하기 때문에 이상적이지 않습니다. 이 이벤트 핸들러를 Effect Event로 감싸서 종속성에서 제거하세요

import { useEffect, useEffectEvent } from 'react';
// ...

export function useChatRoom({ serverUrl, roomId, onReceiveMessage }) {
  const onMessage = useEffectEvent(onReceiveMessage);

  useEffect(() => {
    const options = {
      serverUrl: serverUrl,
      roomId: roomId,
    };
    const connection = createConnection(options);
    connection.connect();
    connection.on('message', (msg) => {
      onMessage(msg);
    });
    return () => connection.disconnect();
  }, [roomId, serverUrl]); // ✅ 모든 종속성이 선언됨
}

이제 채팅은 ChatRoom 컴포넌트가 리랜더링될 때마다 재연결되지 않습니다. 여러분이 갖고 놀 수 있는 커스텀 Hook으로 이벤트 핸들러를 넘기는 완전한 데모가 있습니다.

이제 useChatRoom을 사용하기 위해 어떻게 동작하는지 알 필요가 없음을 확인하세요. 다른 어떤 컴포넌트에도 추가할 수 있고, 어떤 다른 옵션들도 넘겨도 같은 방식으로 동작할 것 입니다. 이 것이 커스텀 Hook의 힘입니다.

커스텀 Hook을 사용 해야할 때​

작은 중복 코드마다 커스텀 Hook으로 추출할 필요는 없습니다. 일부 중복은 괜찮습니다. 예를 들어, 이전처럼 하나의 useState 호출을 감싸기 위해 useFormInput Hook을 추출하는 것은 아마도 불필요할 것입니다.

하지만, Effect를 작성할때마다, 커스텀 Hook으로 감싸는 것이 더 명확한지를 고려하세요. Effects를 매우 자주 필요로 하면 안되기 때문에, 여러분이 하나를 작성할 때, 어떤 외부 시스템이나 React가 내장 API로 지원하지 않는 어떤 것과 동기화 하기위해 "React 외부로 나가야"할 필요가 있다는 의미입니다. 커스텀 Hooks로 감싸는 것은 여러분의 의도와 어떤 데이터 흐름을 가지는지 명확히 소통하도록 해줍니다.

예를 들어 두 드롭다운을 표현하는 shippingForm 컴포넌트를 생각해보세요. 하나는 도시들의 목록을 보여주고, 다른 하나는 선택된 도시의 지역들의 목록을 보여줍니다. 아래와 같은 코드로 시작하게 될 것입니다.

function ShippingForm({ country }) {
  const [cities, setCities] = useState(null);
  // This Effect fetches cities for a country
  useEffect(() => {
    let ignore = false;
    fetch(`/api/cities?country=${country}`)
      .then(response => response.json())
      .then(json => {
        if (!ignore) {
          setCities(json);
        }
      });
    return () => {
      ignore = true;
    };
  }, [country]);

  const [city, setCity] = useState(null);
  const [areas, setAreas] = useState(null);
  // This Effect fetches areas for the selected city
  useEffect(() => {
    if (city) {
      let ignore = false;
      fetch(`/api/areas?city=${city}`)
        .then(response => response.json())
        .then(json => {
          if (!ignore) {
            setAreas(json);
          }
        });
      return () => {
        ignore = true;
      };
    }
  }, [city]);

  // ...

비록 이 코드들은 매우 반복적이지만, Effects를 각각 분리해두는 게 좋습니다. 다른 두가지를 동기화하므로, 하나의 Effect로 합치면 안됩니다. 대신에 여러분은 shippingForm 컴포넌트를 공통되는 로직을 useData Hook으로 추출해서 간단하게 만들 수 있습니다.

function useData(url) {
  const [data, setData] = useState(null);
  useEffect(() => {
    if (url) {
      let ignore = false;
      fetch(url)
        .then((response) => response.json())
        .then((json) => {
          if (!ignore) {
            setData(json);
          }
        });
      return () => {
        ignore = true;
      };
    }
  }, [url]);
  return data;
}

이제 shippingForm 컴포넌트들의 두 Effects를 useData를 호출하는 것으로 변경하세요.

function ShippingForm({ country }) {
  const cities = useData(`/api/cities?country=${country}`);
  const [city, setCity] = useState(null);
  const areas = useData(city ? `/api/areas?city=${city}` : null);
  // ...

커스텀 Hook을 추출하는 것은 데이터 흐름을 명확하게 합니다. url을 넣고 data를 얻는 식입니다. useData 내부로 Effect를 "숨김"으로서, 여러분은 shippingForm에 작업하는 다른 누군가가 불필요한 종속성들을 추가하는 것을 방지할 수 있습니다. 시간이 지나면, 여러분의 앱의 Effects의 대부분은 커스텀 Hooks에 있을 것입니다.

Deep Dive

Custom hook을 높은 수준에서 사용하세요

여러분의 custom hook의 이름을 정하는것부터 시작해봅시다. 만약 명확한 이름을 고르는데 어려움을 느낀다면, effect가 컴포넌트의 로직과 너무 깊은 관계를 맺고 있기 때문일 수도 있습니다. 그리고 아직 추출될 준비가 되지 않은 것이죠.

이상적으로 여러분의 custom hook의 이름은 코드를 한 줄도 쓸줄 모르는 사람이 보더라도 custom hook이 무엇을 하고 무엇을 받고, 무엇을 리턴하는지 추측할 수 있을 정도로 명확해야 합니다.

  • ✅ useData(url)
  • ✅ useImpressionLog(eventName, extraData)
  • ✅ useChatRoom(options)

외부의 시스템과 싱크를 맞추려 할 때 여러분의 custom hook의 이름은 아마 조금 더 기술적이거나 그 시스템에 대한 전문용어가 들어갈 수 있습니다. 그 시스템에 친숙한 사람이 봤을때는 충분히 좋은 이름입니다.

  • ✅ useMediaQuery(query)
  • ✅ useSocket(url)
  • ✅ useIntersectionObserver(ref, options)

Custom hook을 높은 수준에서 사용하세요. Custom hook이 단순히 useEffect API를 편의를 위해 감싸거나 이의 대안 정도밖에 안되는 "생명주기" hook을 만들거나 사용하는 것을 피해야 합니다.

  • 🔴 useMount(fn)
  • 🔴 useEffectOnce(fn)
  • 🔴 useUpdateEffect(fn)

예를 들어, 이 useMount hook은 코드가 "마운트 시점에" 돌아갈 수 있도록 해줍니다.

function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  // 🔴 피해야 할 것 : 커스텀된 "생명주기" hook의 사용
  useMount(() => {
    const connection = createConnection({ roomId, serverUrl });
    connection.connect();

    post('/analytics/event', { eventName: 'visit_chat' });
  });
  // ...
}

// 🔴 피해야 할 것 : 커스텀된 "생명주기" hook의 생성
function useMount(fn) {
  useEffect(() => {
    fn();
  }, []); // 🔴 React Hook useEffect has a missing dependency: 'fn'
}

useMount같은 커스텀한 "생명주기" hook은 React의 패러다임에 맞아 떨어지지 않습니다. 예를 들어, 이 예시 코드는 잘못된 부분이 있습니다 (roomId나 severUrl의 변화에 대해 "반응"하지 않습니다). 하지만 linter는 반응하지 않습니다. linter는 useEffect의 직접적인 호출에 대해서만 검사하기 때문입니다. linter는 여러분의 hook에 대해 알지 못합니다.

만약 여러분이 Effect를 작성하고 있다면, React API를 직접 사용하는것부터 시작해보세요.

function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  // ✅ 두개의 effect가 목적에 따라 구분되고 있기 때문에 괜찮은 코드입니다.

  useEffect(() => {
    const connection = createConnection({ serverUrl, roomId });
    connection.connect();
    return () => connection.disconnect();
  }, [serverUrl, roomId]);

  useEffect(() => {
    post('/analytics/event', { eventName: 'visit_chat', roomId });
  }, [roomId]);

  // ...
}

이어서, 서로 다른 고수준 용례에 대해 custom hook을 추출할 수 있습니다(안해도 괜찮습니다).

function ChatRoom({ roomId }) {
  const [serverUrl, setServerUrl] = useState('https://localhost:1234');

  // ✅ custom hook들이 목적을 드러내는 이름을 갖고 있기 때문에 훌륭한 코드입니다.
  useChatRoom({ serverUrl, roomId });
  useImpressionLog('visit_chat', { roomId });
  // ...
}

좋은 custom hook은 코드의 호출을 선언적으로 만들고 스스로 무엇을 하는지 명확하게 알고 있습니다. 예를 들어, useChatRoom(options)는 오직 채팅방과 연결만 할 수 있고, useImpressionLog(eventName, extraData)는 오직 분석 툴에 로그를 보내는 일만 할 수 있습니다. 만약 여러분의 custom hook API가 사용되는 상황을 제한되지 않고 매우 추상적이라면, 장기적으로는 풀기 어려운 문제들을 많이 만들 것입니다.

Custom hook은 코드가 더 나은 패턴으로 작성되도록 돕습니다.​

Effect는 일종의 "부화기"입니다. 용례에 대한 충분히 훌륭한 내장 솔루션이 없을때 "React 외부와 연결"하기 위해 effect를 이용합니다. 한때 React 팀의 목표는 더 구체적인 문제들에 대해 더 구체적인 해결방법을 제시함으로써 여러분의 app 내에서 effect의 사용을 최대한 줄이는 것이었습니다. Effect를 custom hook으로 감싸는 것은 이런 해결 방법들이 유효할때 코드를 쉽게 업그레이드할 수 있도록 도왔습니다.

아래 예시로 돌아가봅시다.

위 예시에서 useOnlineStatus는 한 쌍의 useState와 useEffect로 작성되어 있습니다. 하지만 이건 최선의 해결방법이 아닙니다. 이 hook이 고려하지 않고 있는 몇개의 edge case들이 있습니다. 예를 들어, 컴포넌트가 마운트 되었을 때 isOnline은 이미 true이지만, 네트워크가 오프라인으로 바뀌어버린 상황이라면 잘못될 가능성이 있습니다. 브라우저의 navigator.onLine API를 이용해 체크할 수도 있지만, 초기 HTML을 만드는 서버에서는 이 코드를 직접 이용할 수 없습니다. 간단히 말해서, 이 코드는 충분히 개선될 수 있습니다.

다행히도, React 18은 useSyncExternalStore라는 특수 API를 포함하고 있습니다. 이 API는 여러분이 이런 유형의 문제들을 해결할 수 있도록 도울 수 있습니다. 아래는 이 새로운 API를 이용해 다시 작성된 useOnlineStatus hook입니다.

새 hook의 적용을 위해 컴포넌트의 어떤 내용도 수정할 필요가 없는 점에 주목해 보세요.

function StatusBar() {
  const isOnline = useOnlineStatus();
  // ...
}

function SaveButton() {
  const isOnline = useOnlineStatus();
  // ...
}

Effect를 감싸는 custom hook이 가끔 효과적인 또 다른 이유는 아래와 같습니다.

  1. Effect에서의 데이터 흐름을 매우 명백하게 만듭니다.
  2. 컴포넌트가 정확한 effect의 구현보다 의도에 주목할 수 있도록 합니다.
  3. React에서 새로운 기능을 추가했을 때 컴포넌트 수정 없이 effect를 제거할 수 있습니다.

App에서 사용되는 일반적이고 관용적인 표현들을 추출해 컴포넌트에서 custom hook으로 옮기는 과정은 디자인 시스템과 유사하기도 합니다. 이는 컴포넌트의 코드가 그 의도에 집중할 수 있도록 돕고, 순수한 effect를 자주 작성하는 것으로부터 피하게 해줍니다. 많은 훌륭한 custom hook들은 React 커뮤니티에 의해 관리되고 있습니다.

Deep dive

React가 데이터를 가져오기 위한 내장 솔루션을 제공할까요?​

React 팀은 아직 세부 사항을 검토하고 있지만, 미래엔 데이터를 이런 식으로 가져오길 기대하고 있습니다.

import { use } from 'react'; // 아직 사용은 안됩니다!

function ShippingForm({ country }) {
  const cities = use(fetch(`/api/cities?country=${country}`));
  const [city, setCity] = useState(null);
  const areas = city ? use(fetch(`/api/areas?city=${city}`)) : null;
  // ...

만약 여러분이 useData같은 custom hook을 app에서 사용하고 있다면, 순수한 effect를 통해 모든 컴포넌트에 각각 작성된 코드보다 훨씬 적은 수정만으로 우리가 원하는 방향으로 마이그레이션 할 수 있을 것입니다. 하지만 오래된 접근법도 여전히 잘 동작합니다. 순수한 effect를 적는 코드가 충분히 괜찮다면 굳이 바꿀 필요는 없습니다.

한가지 방법만 존재하지 않습니다.​

브라우저의 requestAnimationFrame API를 이용해 fade-in 애니메이션을 처음부터 만든다고 가정해봅시다. 아마 애니메이션 루프를 설정하는 effect부터 만들 것입니다. 각 애니메이션의 프레임에서는 ref에 할당된 DOM 노드의 opacity를 1까지 변경할 것입니다. 코드는 이렇게 시작하겠죠.

컴포넌트를 더 읽기 쉽게 만들기 위해 useFadeIn custom hook을 만들오 로직을 추출할 수도 있습니다.

useFadeIn 코드를 유지해도 괜찮지만, 조금 더 리팩토링 하길 원할 수도 있습니다. 예를 들어, useFadeIn에서 애니메이션 루프를 세팅하는 로직을 꺼내 useAnimationLoop custom hook으로 옮길 수도 있습니다.

마찬가지로 이렇게 할 필요는 없습니다. 일반 함수를 작성하듯, 궁극적으로는 코드의 서로 다른 부분에 경계선을 어떻게 그릴지를 결정하게 됩니다. 혹은 전혀 다른 접근을 할 수도 있습니다. Effect 안에 로직을 두지 않고 로직의 가장 명령적인 부분들을 JavaScript class의 내부로 옮길 수도 있습니다.

Effect는 React의 외부 시스템과 연결할 수 있도록 돕습니다. Effect들끼리의 관계가 더 필요할수록 (예를 들어, 여러 애니메이션들을 연결하는 것) effect와 hook의 외부로 로직을 완전히 추출하는것은 더 좋습니다. 바로 위의 sandbox 코드처럼 말이죠. 이제 여러분이 추출한 코드는 "외부 시스템"이 되었습니다. 이는 여러분의 effect가 간단한 형태로 남아있을 수 있도록 해줍니다. Effect는 여러분이 React 밖으로 옮긴 시스템과 통신하는 것만 하면 되기 때문입니다.

위의 예시는 fade-in 로직이 JavaScript로만 작성된다는 점을 가정한 것입니다. 하지만 이 fade-in 애니메이션은 순수한 CSS Animaition을 이용하면 훨씬 간단하고 효과적으로 구현할 수 있습니다.

가끔씩은 hook이 아예 필요 없을 수도 있습니다!

되짚어보기​

  • Custom hook은 컴포넌트들 간에 로직을 공유할 수 있도록 해줍니다.
  • Custom hook은 꼭 use로 시작하는 이름을 붙여야 하고, 뒤에 오는 글자는 영문 대문자여야 합니다.
  • Custom hook은 상태를 다루는 로직만 공유할 수 있습니다. 상태 그 자체는 공유되지 않습니다.
  • Reactive한 값을 한 hook에서 다른 hook으로 전달할 수 있으며, 이 값은 최신 상태를 유지합니다.
  • 모든 Hook은 컴포넌트가 다시 렌더링되면 항상 다시 동작합니다.
  • Custom hook의 모든 코드는 항상 순수해야 합니다. 컴포넌트의 코드처럼 말이죠.
  • Custom hook으로부터 받은 모든 이벤트 핸들러들을 effect event로 감싸세요.
  • useMount같은 custom hook은 만들지 마세요. 항상 목적을 구체적으로 생각합시다.
  • 코드의 경계선을 어느 정도로 정할지는 항상 당신에게 달렸습니다.
문제에 도전해보세요.

도전 1 : useCounter hook 추출하기​

이 컴포넌트는 매 초마다 증가하는 숫자를 출력하기 위해 상태 변수와 effect를 사용합니다. 이 로직을 useCounter custom hook으로 추출해보세요. 목표는 Counter 컴포넌트가 정확하게 아래와 같이 구현되도록 하는 것입니다.

export default function Counter() {
  const count = useCounter();
  return <h1>Seconds passed: {count}</h1>;
}

Custom hook은 useCounter.js에 작성해야 합니다. 그리고 Counter.js에서 불러오도록 하세요.

코드는 아래처럼 작성해야 합니다.

App.js가 useState와 useEffect를 더 이상 불러오지 않는다는 점을 주목하세요.

도전 2 : 카운터의 딜레이를 조정 가능하도록 만들기​

이 예시에서, delay 상태 변수는 슬라이더를 통해 관리됩니다. 하지만 그 값은 사용되지 않고 있습니다.delay 값을 useCounter custom hook에 전달하고, useCounter hook이 하드코딩된 1000ms가 아닌 전달받은 delay를 이용하도록 수정해보세요.

정답​

useCounter(delay) 같은 방식으로 delay를 hook에 전달합니다. 그리고 hook의 내부에서 delay를 하드코딩된 1000ms 대신 사용합니다. delay는 effect의 의존성으로 추가되어야 할 것입니다. 이는 delay가 바뀌면 interval을 초기화 하는 것을 보장합니다.

도전 3 : useInterval을 useCounter 밖으로 추출하기​

지금 useCounter는 두 가지 일을 하고 있습니다. 하나는 interval을 설정하는 것이고, 다른 하나는 매 interval tick마다 상태 변수를 증가시키는 것입니다. 여기서 interval을 설정하는 로직을 추출해 useInterval로 작성해보세요. useInterval은 onTick 콜백과 delay 두 인자를 받아야 합니다. 이렇게 바꾼 다음엔 useCounter는 아래와 같이 작성되어야 합니다.

export function useCounter(delay) {
  const [count, setCount] = useState(0);
  useInterval(() => {
    setCount(c => c + 1);
  }, delay);
  return count;
}

useInterval은 useInterval.js에 작성하고, useCounter.js에서 불러오도록 하세요.

정답​

useInterval 내부의 로직은 interval을 설정하고 초기화해야 합니다. 다른 일은 할 필요 없습니다.

이 해답엔 약간의 문제가 있지만, 다음 도전에서 해결하게 될 것입니다.

도전 4 : Interval 초기화 코드 고치기​

이 예시엔 두 개의 분리된 interval이 있습니다.

App 컴포넌트는 useCounter를 호출하고, useCounter는 useInterval을 호출해 카운터를 매 초마다 업데이트 합니다. 하지만 App 컴포넌트는 또한 매 2초마다 페이지의 배경색을 랜덤하게 바꾸기 위해 useInterval를 호출하고 있습니다.

어떤 이유로 인해 페이지의 배경색을 바꾸는 콜백은 동작하지 않습니다. useInterval 내부에 로그를 조금 추가해봅시다.

useEffect(() => {
    console.log('✅ delay와 함께 interval을 설정 ', delay)
    const id = setInterval(onTick, delay);
    return () => {
      console.log('❌ delay와 함께 interval을 초기화', delay)
      clearInterval(id);
    };
  }, [onTick, delay]);

로그가 여러분이 예상한 것과 동일하게 찍히고 있나요? 만약 어떤 effect가 불필요하게 다시 동기화되고 있는 것 같다면, 어떤 의존성이 이 문제를 일으키고 있는지 예상이 가나요? Effect로부터 의존성을 제거할 수 있는 방법이 있나요?

이 이슈를 수정한 다음엔, 페이지의 배경색이 매 2초마다 업데이트 되어야 합니다.

힌트​

useInterval hook이 이벤트 리스너를 인자로 받는 것처럼 보입니다. 이벤트 리스너를 감싸서 effect의 의존성으로 넣어줄 필요가 없도록 하는 방법이 있을까요?

정답​

useInterval의 내부에서 이 페이지에서 조금 전에 했던것 처럼 tick 콜백을 effect event로 감쌉니다.

그럼 onTick 콜백을 effect의 의존성에서 제거할 수 있게 됩니다. Effect는 컴포넌트의 모든 리렌더링마다 다시 싱크를 맞추지 않게 되므로 페이지의 배경색에 연결된 interval은 배경색을 바꾸는 동작을 하기도 전에 매 초마다 초기화든 일이 발생하지 않습니다.

이렇게 수정하면 두 interval 모두 기대한것과 같이 동작하고 서로 경쟁하지 않게 됩니다.

도전 5 : 비틀거리는 동작 구현하기​

이 예시에서 usePointerPosition() hook은 현재 포인터의 위치를 추적합니다. 미리보기 영역에서 커서를 움직이거나 손라가을 움직여 빨간 점이 움직임을 따라오는지 확인해보세요. 이 빨간 점의 위치는 pos1 변수에 저장됩니다.

사실은 다섯개(!)의 다른 점이 렌더링 되고 있습니다. 아마 눈치채지 못했읉텐데, 모두 같은 위치에 렌더링 되고 있기 때문입니다. 이 부분은 여러분이 고쳐야 합니다. 그 대신 구현해야 할 것은 "비틀거리는" 동작입니다. 각각의 점은 이전 점의 경로를 "따라"가야 합니다. 예를 들어, 만약 커서를 빠르게 움직이면 첫번째 점은 즉시 따라오지만 두번째 점은 조금의 딜레이를 두고 첫번째 점을 따라가야 합니다. 세번째 점은 두번째 점을 따라가고, 네번째와 다섯번째도 그런 식입니다.

여러분은 useDelayedValue custom hook을 구현해야 합니다. 지금은 인자로 받은 value를 리턴하도록 구현되어 있습니다. 이 대신에 delay 밀리세컨드 이전의 value를 리턴해야 합니다. 아마 state와 effect를 조금 써야 할 수도 있습니다.

useDelayedValue의 구현이 완료되면, 점들이 서로를 따라 다니는걸 볼 수 있어야 합니다.

힌트​

delayedValue를 상태변수로서 custom hook에 저장해야 합니다. value가 변경되면 effect를 호출합니다. 이 effect는 delayedValue를 delay 이후에 업데이트 합니다. setTimeout을 호출하는게 도움이 될 수도 있습니다.

이 effect는 cleanup이 필요할까요? 필요하다면 왜 필요하고, 그렇지 않다면 왜일까요?

정답​

동작하는 버전입니다. delayedValue를 상태 변수로 관리하고 있습니다. value가 업데이트 될때 effect는 delayedValue를 업데이트 하는 timeout을 스케줄링합니다. 이렇게 delayedValue가 항상 실제 value보다 "지연"되도록 할 수 있습니다.

Effect는 cleanup이 필요하지 않은 부분에 주목해보세요. 만약 clearTimeout을 cleanup 함수에서 호출한다면, value가 변하는 각 시간마다 이미 스케줄링된 timeout을 리셋할 것입니다. 이 동작을 계속 유지하고 싶다면, 모든 timeout이 동작해야 합니다.
