탈출구 (Advanced)

여러분의 컴포넌트 중 일부는 React 외부의 시스템과 함께 컨트롤되고 동기화될 필요가 있을 것 입니다. 예를들어 브라우저 API를 활용하여 input에 포커싱할 필요나 React 없이 구현된 비디오 플레이어를 재생/정지하거나, 원격 서버에 연결하여 메시지를 기다리도록 해야합니다. 이 챕터에서 여러분은 React의 "외부로 나가도록"하여 외부 시스템들과 연결해주는 탈출구를 배울 것 입니다. 여러분의 어플리케이션 로직과 데이터 흐름 대부분은 이 기능들에 의존하지 않아야합니다.

In this chapter

  • React에 의해 관리되는 DOM에 접근하는 법
  • 외부 시스템들에 컴포넌트를 동기화하는 법

  • 일부 값들이 Effect를 재트리거 하는 것을 방지하는 법
  • Effect가 덜 재실행되도록 하는 법
  • 컴포넌트간 로직을 공유하는 법

refs로 값들을 참조하는 법​

여러분이 컴포넌트가 일부 정보를 "기억"하게 하고싶지만 정보가 새로운 렌더링을 유도하길 원치 않는다면, ref를 사용할 수 있습니다.

const ref = useRef(0);

상태와 같이, refs는 React에 의해 리렌더링 사이에 보존됩니다. 하지만 상태를 설정하는 것은 컴포넌트를 리렌더링시킵니다. ref를 변경하는 것은 그렇지 않습니다! 여러분은 ref.current 속성으로 현재 ref의 값에 접근할 수 있습니다

ref는 React가 추적하지 않는 컴포넌트의 비밀 저장소입니다. 예를들어 여러분은 timeout IDs나 Dom 요소들, 그리고 컴포넌트의 렌더링 결과에 영향을 주지않는 다른 객체들을 저장하는데 ref를 사용할 수 있습니다.

Ready to learn this topic?

정보를 기억하는데 refs를 어떻게 사용하는지 배우고 싶다면 Refs로 값 참조하기를 읽어보세요

refs로 DOM 조작하기​

React는 여러분의 렌더링 결과와 일치시키기위해 DOM을 자동으로 업데이트하기 때문에, 여러분의 컴포넌트는 자주 이에 대해 조작할 필요가 없습니다. 하지만 가끔 여러분은 React에 조작되는 DOM 요소들에 접근해야할 필요가 있을 수 있습니다 - 예를 들어, node에 포커싱하기 위해, scroll하기 위해, 혹은 사이즈와 위치를 알아내기 위해서입니다. React는 이를 위한 내장 기능이 없기 때문에, 여러분은 DOM 노드의 ref가 필요합니다. 예를들어, 버튼을 클릭하여 ref를 사용하는 input에 포커싱할 수 있습니다.

Effects로 동기화하기​

일부 컴포넌트들은 외부 시스템들과 동기화할 필요가 있습니다. 예를 들어 React 상태로 React기반이 아닌 컴포넌트를 조작하거나, 서버 연결을 설정하거나, 컴포넌트가 화면에 나타났을때 분석 log를 보내는 경우입니다. 여러분이 특정 이벤트들을 처리하는 이벤트 핸들러들과는 다르게, Effects는 여러분이 렌더링 이후에 코드를 실행하도록 합니다. 이를 활용하여 React 외부의 시스템과 컴포넌트를 동기화해보세요.

시작/정지를 여러번 누르고 비디오 플레이어가 isPlaying prop 값에 어떻게 동기화되어있는지 보세요.

대부분의 Effects는 또한 실행 후 "정리(clean up)"가 필요합니다. 예를들어 여러분의 Effect가 채팅서버와의 연결을 설정하였다면, React에게 컴포넌트와 채팅 서버의 연결을 종료하라는 cleanup function을 리턴해야합니다.

개발에서는, React가 즉시 실행 후 Effect를 한번 더 정리합니다. 이는 여러분이 "✅ Connecting..."출력을 두 번 보는 이유입니다. 이로서 여러분은 정리 함수를 작성하는 것을 잊지 않게 됩니다.

Ready to learn this topic?

외부 시스템과 컴포넌트들을 동기화하는 법을 배우려면 Effects로 동기화하기를 읽어보세요

Effect가 필요 없을 수 있습니다.​

Effects는 React 패러다임의 탈출구 입니다. 이는 여러분을 React "밖으로", 그리고 여러분의 컴포넌트를 외부 시스템에 동기화하게 해줍니다. 만약 관련된 외부 시스템이 없다면 (예를 들어, props나 상태가 변경될때 컴포넌트의 상태를 변경하고 싶다면), 여러분은 Effect가 필요없습니다. 불필요한 Effects를 지우는 것은 여러분의 코드를 더 쉽게 팔로업하고, 배우고, 에러를 덜 발생하게 합니다.

여러분이 Effects를 필요로 하지 않는 두가지 흔한 케이스가 있습니다

  • 렌더링을 위한 데이터 변경에 Effect를 필요로 하지 않습니다.
  • 유저 이벤트를 조작하는데 Effect를 필요로 하지 않습니다.

예를 들어, 여러분은 다른 상태를 기반으로 한 어떤 상태를 조작하기 위해 Effect를 필요로 하지 않습니다.

function Form() {
  const [firstName, setFirstName] = useState('Taylor');
  const [lastName, setLastName] = useState('Swift');

  // 🔴 Avoid: redundant state and unnecessary Effect
  const [fullName, setFullName] = useState('');
  useEffect(() => {
    setFullName(firstName + ' ' + lastName);
  }, [firstName, lastName]);
  // ...
}

대신에, 렌더링동안 계산하고 싶은만큼 하면 됩니다.

function Form() {
  const [firstName, setFirstName] = useState('Taylor');
  const [lastName, setLastName] = useState('Swift');
  // ✅ Good: calculated during rendering
  const fullName = firstName + ' ' + lastName;
  // ...
}

하지만, 여러분은 외부 시스템과의 동기화를 위해서는 Effects가 정말로 필요합니다.

Ready to learn this topic?

불필요한 Effects의 제거를 배우고 싶다면 여러분은 Effect가 필요 없을 수 있습니다를 읽어보세요.

활성화된 Effects의 생명주기​

Effects는 컴포넌트들과는 다른 생명주기를 갖고 있습니다. 컴포넌트는 마운트되고, 업데이트되고, 언마운트 됩니다. Effect는 두가지만 합니다: 어떤 것을 동기화하기 시작하는 것, 그리고 이후에 동기화를 그만두는 것입니다. 이 사이클은 여러분의 Effect가 변화하는 props나 상태에 의존하고 있다면 여러번 발생할 수 있습니다.

이 Effect는 roomId prop의 값에 의존합니다. Props는 변화하는 값이고, 리렌더링 중에 변할 수 있다는 의미입니다. Effect가 여러분이 roomId를 업데이트한 후에 재동기화(그리고 서버에 재 연결)함을 보세요.

React는 여러분이 Effect의 의존성들을 잘 명시했는지 체크하는 linter 규칙을 제공합니다. 여러분이 위 예제에서 roomId를 의존성 리스트에서 명시하는 것을 잊었다면, linter가 버그를 자동으로 찾습니다.

Ready to learn this topic?

Effect와 컴포넌트의 생애주기가 어떻게 다른지를 배우려면 변하는 이벤트의 생애주기를 읽어보세요.

Effect에서 이벤트들 분리하기​

Under Constrction

이 섹션은 여러분이 아직 사용할 수 없는 React에 추가되지 않은 실험적 API를 묘사하고 있습니다.

이벤트 핸들러는 여러분이 동일한 인터렉션을 재실행할때만 수행됩니다. 이벤트 핸들러와는 다르게 Effects는 마지막 렌더링에서의 prop 또는 상태값이 달라졌을 때 재 동기화를 수행합니다. 가끔 여러분은 이 두 동작을 섞고 싶을 것 입니다: Effect가 어떤 값에는 반응하여 재실행되고, 다른 것들에는 실행되지 않도록 말입니다.

Effect 내부의 모든 코드는 반응형입니다. 리렌더링때문에 Effect가 읽는 반응형 값이 변경되면 실행될 것 입니다. 예를 들어 이 Effect는 roomId나 theme가 인터렉션 후에 변경되면 재연결 합니다.

이는 이상적이지 않습니다. 여러분은 roomId가 변경되었을 때만 채팅에 재연결하고 싶습니다. theme를 변경하는 것은 채팅을 재연결하면 안됩니다! Effect에서 theme를 읽는 코드를 이벤트 함수로 옮기세요.

이벤트 함수 내부의 코드는 반응적이지 않으므로, theme를 변경하는 것은 Effect가 재연결하게 하지 않습니다.

Effect 종속성 제거하기​

여러분이 Effect를 작성할 때, linter는 여러분이 Effect의 종속성에 속한 모든 반응형 값들(props나 상태와 같은)을 포함하였는지를 확인합니다. 이는 여러분의 Effect가 여러분의 컴포넌트의 prop이나 상태에 동기화된 상태로 유지되도록 보장해줍니다. 불필요한 종속성은 Effect가 너무 자주 동작하도록 하거나, 심지어는 무한루프를 만들기도 합니다. 이를 제거하는 방법은 경우에 따라 다릅니다.

예를 들어, 이 Effect는 여러분이 input을 수정할 때마다 재생성되는 options 객체에 의존합니다.

여러분은 채팅에서 메시지를 쓰기 시작할때마다 채팅을 재연결 하고싶지 않습니다. 이 문제의 해결을 위해서, options객체의 생성을 Effect의 내부로 옮기고 Effect가 roomId 문자열에만 의존하도록 하면 됩니다.

여러분은 options 의존성을 제거하기 위해 의존성 리스트를 편집하며 시작하지 않았습니다. 그것은 잘못됐을 수 있습니다. 대신에, 주변 코드를 수정하여 의존성이 불필요하게 만들었습니다. 여러분은 의존성 리스트를 여러분의 Effect 코드에서 사용되는 반응형 값들의 리스트라고 생각하면 됩니다. 여러분은 의도적으로 그 리스트에 들어갈 것을 정하지 않습니다. 리스트가 여러분의 코드를 묘사하는 것 입니다. 의존성 리스트를 수정하기 위해서는, 코드를 수정하세요.

커스텀 훅으로 로직 재사용하기​

React는 useState, useContext, useEffect와 같은 내부 훅으로 이뤄집니다. 가끔 여러분은 더 구체적인 목적을 가진 훅이 있기를 바랄 것 입니다: 예를 들어 데이터 가져오기, 유저가 접속 중인지 지속적으로 확인하기, 채팅방에 접속하기의 경우입니다. 이를 위해서, 여러분은 어플리케이션의 필요에 따라 각자의 훅을 만들 수 있습니다.

이 예시에서, usePointerPosition 커스텀 훅은 커서의 위치를 추적합니다. 반면에 useDelayedValue 커스텀 훅은 여러분이 넘긴 값에서 특정 밀리초가 "지연된" 값을 반환합니다. 샌드박스 프리뷰 영역에 커서를 옮기고 커서를 따라다니는 점들의 경로이동을 보세요.

여러분은 커스텀 훅을 만들고, 합성하거나 데이터를 교환하거나, 컴포넌트들 간에서 재사용할 수 있습니다. 여러분의 앱이 성장할수록, 이미 작성된 커스텀 훅을 재사용할 수 있기 때문에, 더 적은 Effect를 작성할 것 입니다. 또한 React 커뮤니티에 의해 유지되는 훌륭한 커스텀 훅이 많이 있습니다.
