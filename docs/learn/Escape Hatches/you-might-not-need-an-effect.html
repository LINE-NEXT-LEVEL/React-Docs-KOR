Effect가 필요하지 않을 수 있습니다.

Effect는 React 패러다임의 탈출구입니다. React에서 "벗어나게"하고 여러분의 컴포넌트를 React가 아닌 위젯, 네트워크 또는 브라우저 DOM같은 다른 외부 시스템과 동기화 하도록 합니다. 관련된 외부 시스템이 없다면(예를들어, 어떤 props나 상태가 변경되었을때 컴포넌트의 상태가 변경되길 원한다면), 여러분은 Effect를 사용할 필요가 없습니다. 불필요한 Effects를 제거하는 것은 여러분의 코드를 따라하기 쉽고, 빠르게 배우며, 에러를 덜 만들도록 할 것 입니다.

여러분이 배울 것
  • 컴포넌트에서 불필요한 Effects를 제거하는 이유와 방법
  • Effect없이 비용이 큰 계산을 캐싱 하는 법
  • Effects없이 컴포넌트의 상태를 초기화하거나 조정하는 법
  • 이벤트 핸들러간에 로직을 공유하는 법
  • 어떤 로직이 이벤트 핸들러로 옮겨져야 하는지
  • 부모 컴포넌트에 변경을 알리는 법

불필요한 Effects를 제거하는 방법​

여러분이 Effects를 필요로 하지 않는 두가지 공통 케이스가 있습니다.

  • Effects를 렌더링을 위한 데이터 변경에 사용할 필요가 없습니다. 예를 들어 리스트를 보여주기 전에 필터링 하고싶다고 가정해봅시다. 여러분은 리스트가 변경될때 Effects에서 상태변수를 업데이트하도록 작성하고 싶은 마음을 느낄 것입니다. 하지만 이는 비효율적입니다. 여러분이 컴포넌트의 상태를 업데이트할 때 React는 여러분의 컴포넌트 함수들을 불러 스크린에 보여져야 하는 것을 계산하도록 합니다. 그리도 React는 이 변경을 스크린을 업데이트하는 DOM에 "commit"합니다. React는 여러분의 Effects를 실행합니다. 또 여러분의 Effect가 상태 즉시 업데이트한다면, 이는 전체 과정을 처음부터 재시작합니다! 불필요한 렌더링 과정을 피하기 위해서, 여러분의 컴포넌트 최상단에서 모든 데이터를 변경하세요. 그 코드는 자동으로 여러분의 props나 상태가 변할 때 재실행 될 것입니다.

  • 유저 이벤트를 핸들링하는데 Effects를 사용할 필요 없습니다 예를 들어, 유저가 상품을 구매할 때 /api/buy POST 요청을 보내고 알림을 표시하고 싶다고 가정합시다. 여러분은 구매 버튼 클릭 이벤트 핸들러에서 정확히 무슨 일이 일어났는지 알 수 있습니다. Effect가 실행되는 시점에는, 유저가 무엇을 했는지(예를 들어, 어떤 버튼이 클릭되었는지) 알 수 없습니다. 이는 이벤트 핸들러에서 유저 이벤트를 처리하는 이유입니다.

여러분은 외부 시스템과 동기화하는데에 Effect를 반드시 필요로 합니다. 예를 들어 jQuery위젯과 React 상태가 동기화되도록 유지하는 Effect를 작성할 수 있습니다. 또한 Effect로 데이터를 가져올 수 있습니다: 예를 들어, 현재 검색 쿼리의 검색 결과를 동기화할 수 있습니다. 현대 프레임워크는 여러분의 컴포넌트들에 Effects를 작성하는 것보다 더 효율적인 내장 데이터 호출 메커니즘을 제공함을 생각하세요.

여러분이 올바른 직관을 얻는데 도움이 되는 일반적이고 구체적인 예시들 몇가지를 살펴보겠습니다.

props와 상태를 기반으로 상태를 업데이트하기​

여러분이 두 상태 변수 firstName과 lastName을 가진 컴포넌트를 갖고있다고 가정합시다. 여러분은 그 값들을 합쳐서 fullName을 만들고 싶습니다. 나아가 fullName이 firstName이나 lastName가 변경될 때마다 업데이트 되도록 하고 싶습니다. 여러분의 첫 느낌은 Effect 안에서 fullName 상태 변수를 더하는 것일겁니다.

function Form() {
  const [firstName, setFirstName] = useState('Taylor');
  const [lastName, setLastName] = useState('Swift');

  // 🔴 Avoid: 더 이상 사용하지 않는 상태와 불필요한 Effect
  const [fullName, setFullName] = useState('');
  useEffect(() => {
    setFullName(firstName + ' ' + lastName);
  }, [firstName, lastName]);
  // ...
}

이것은 필요 이상으로 복잡합니다. 또한 효율적이지 않습니다. fullName에 대한 기존 값으로 전체 렌더링을 수행한 다음, 즉시 업데이트된 값으로 재렌더링합니다. 상태 변수와 Effect를 모두 제거하세요.

function Form() {
  const [firstName, setFirstName] = useState('Taylor');
  const [lastName, setLastName] = useState('Swift');
  // ✅ Good: 렌더링동안 계산 됨
  const fullName = firstName + ' ' + lastName;
  // ...
}

기존의 props나 state에서 계산할 수 있는 것은 상태에 넣지 마세요. 대신에, 렌더링 동안 계산하세요. 이렇게 하면 코드가 더 빨라지고 (추가적인 "cascading" 업데이트를 피합니다), 더 간단해지며 (일부 코드를 제거합니다), 그리고 오류를 덜 일으킵니다 (서로 다른 상태 변수가 동기화되지 않아서 발생하는 버그를 피합니다). 이 접근법이 새롭게 느껴지면, React로 생각하기에서 어떤 것이 상태에 들어가야 하는지에 대한 가이드가 있습니다.

비용이 높은 계산들을 캐싱하기​

이 컴포넌트는 props로 받은 todos를 가져와 filter prop에 따라 필터링하여 visibleTodos를 계산합니다. 여러분은 상태 변수에 결과를 저장하고 Effect에서 업데이트 하고싶은 충동을 느낄 것 입니다.

function TodoList({ todos, filter }) {
  const [newTodo, setNewTodo] = useState('');

  // 🔴 Avoid: 더 이상 사용하지 않는 상태와 불필요한 Effect
  const [visibleTodos, setVisibleTodos] = useState([]);
  useEffect(() => {
    setVisibleTodos(getFilteredTodos(todos, filter));
  }, [todos, filter]);

  // ...
}

위의 예제처럼, 이는 불필요하고 비효율적입니다. 먼저, 상태와 Effect를 제거하세요.

function TodoList({ todos, filter }) {
  const [newTodo, setNewTodo] = useState('');
  // ✅ getFilteredTodos()가 느리지 않으면 괜찮습니다.
  const visibleTodos = getFilteredTodos(todos, filter);
  // ...
}

많은 경우에, 이 코드는 괜찬습니다. 하지만 아마도 getFilteredTodos()는 여러분이 많은 todos를 가질 때 느릴겁니다. 그 케이스에서 만약 newTodo과 같은 관련없는 상태 변수가 변경될 때 getFilteredTodos()를 계산하고 싶지 않을 것입니다.

여러분은 비싼 계산을 useMemo 훅으로 감싸서 캐싱할 수 있습니다(또는 "memoize")

import { useMemo, useState } from 'react';

function TodoList({ todos, filter }) {
  const [newTodo, setNewTodo] = useState('');
  const visibleTodos = useMemo(() => {
    // ✅ todos 나 filter가 변경되지 않으면 재실행되지 않습니다.
    return getFilteredTodos(todos, filter);
  }, [todos, filter]);
  // ...
}

혹은 아래처럼 한 줄로 작성할 수도 있습니다.

import { useMemo, useState } from 'react';

function TodoList({ todos, filter }) {
  const [newTodo, setNewTodo] = useState('');
  // ✅  todos 나 filter가 변경되지 않으면 getFilteredTodos()를 재실행하지 않습니다.
  const visibleTodos = useMemo(() => getFilteredTodos(todos, filter), [todos, filter]);
  // ...
}

이는 React에게 todos나 filter가 변경되지 않는 한 내부 함수가 재실행되길 원하지 않는다고 전달하는 것입니다. React는 getFilteredTodos()의 반환 값을 첫 렌더링동안 기억합니다. 다음 렌더링 동안, React는 todos나 filter가 다른지를 확인합니다. 최종적으로 두 값이 같다면, useMemo는 저장해놓았던 마지막 결과값을 리턴합니다. 만약 다르다면, React는 감싸진 함수를 다시 호출합니다(그리고 그 결과를 대신 저장합니다).

여러분이 useMemo로 감싼 함수는 렌더링 동안 실행되며, 순수 계산들에만 동작합니다.

Deep Dive

계산이 비용이 높은지 어떻게 알 수 있을까요?​

일반적으로, 수천 개의 객체를 만들거나 반복하지 않는 한, 대체로 비용은 많이 들지 않습니다. 여러분이 더 확신을 갖고 싶다면, 코드 조각에 소모된 시간을 측정하기 위해 콘솔 로그를 추가할 수 있습니다.

console.time('filter array');
const visibleTodos = getFilteredTodos(todos, filter);
console.timeEnd('filter array');

여러분이 측정중인 상호작용을 실행하세요(예를들어 입력 값을 타이핑하는 것). 그리고 콘솔에서 filter array: 0.15ms와 같은 로그들을 보세요. 전반적인 로깅된 시간이 많은 양에(1ms 또는 이상) 도달한다면, 그 계산을 메모이징하는 게 합리적일 수 있습니다. 실험적으로 여러분은 useMemo로 계산을 감싸 총 로깅된 시간이 상호작용에 따라 줄었는지 아닌지 검증할 수 있습니다.

console.time('filter array');
const visibleTodos = useMemo(() => {
  return getFilteredTodos(todos, filter); // Skipped if todos and filter haven't changed
}, [todos, filter]);
console.timeEnd('filter array');

useMemo는 첫렌더링을 빠르게 하지는 않습니다. 오직 업데이트시의 불필요한 작업을 생략해줍니다.

여러분의 기기가 유저보다 좋을 수 있기 때문에, 인공적으로 느려진 상태에서 성능을 테스트하는 것도 좋은 생각임을 명심하세요. 이를 위해 Chrome은 CPU 쓰로틀링 옵션을 제공합니다.

또한 개발환경에서 성능을 측정하는 것은 정확한 결과를 낼 수 없습니다. (예를 들어 Strict Mode가 켜져있으면, 모든 컴포넌트가 한번이 아니라 두번씩 렌더링 될 것 입니다.) 가장 정확한 타이밍을 얻기 위해서, 여러분의 앱을 프로덕션으로 빌드하고 유저들이 가진 기기로 테스트해보세요.

prop이 변화할때 모든 상태를 리셋하기​

이 ProfilePage 컴포넌트는 userId prop을 받습니다. 페이지는 코멘트 입력을 갖고 있고, comment를 상태 변수로 갖고 있습니다. 그런데 문제가 발생했습니다. 어떤 프로필에서 다른 프로필로 향하면, comment가 리셋되지 않는 것입니다. 결과로, 다른 유저의 프로필에 코멘트를 실수로 작성하기 쉬워졌습니다. 여러분은 이 문제를 해결하기 위해, userId가 변경될 때 comment를 정리하고 싶어합니다.

export default function ProfilePage({ userId }) {
  const [comment, setComment] = useState('');

  // 🔴 Avoid: Effect에서 prop 변경시에 상태를 초기화 하는 것
  useEffect(() => {
    setComment('');
  }, [userId]);
  // ...
}

이는 ProfilePage와 그 자식들이 기존 값으로 첫 렌더링을 진행하고, 다시 렌더링하기 때문에 비효율적입니다. 또한 여러분이 ProfilePage에 있는 상태를 가지는 모든컴포넌트에 수행해야하기 때문에 복잡합니다. 예를 들어, 코멘트 UI가 감싸져있다면, 감싸진 코멘트 상태도 청소하고 싶을 것입니다.

대신에, 여러분은 React에게 분리된 키를 제공해서 각 유저의 프로필은 개념적으로 다른 프로필이라고 알려줄 수 있습니다. 여러분의 컴포넌트를 둘로 나누고, key값을 외부 컴포넌트에서 내부 컴포넌트로 넘겨주세요.

export default function ProfilePage({ userId }) {
  return <Profile userId={userId} key={userId} />;
}

function Profile({ userId }) {
  // ✅ 이 상태와 아래의 상태들은 key 변경시 자동으로 리셋됩니다.
  const [comment, setComment] = useState('');
  // ...
}

일반적으로 , React는 같은 컴포넌트가 같은 장소에 렌더링 될 때 상태를 보존합니다. userId를 key로서 Profile 컴포넌트에 넘김으로서, 여러분은 React에게 다른 userId를 가진 두 Profile 컴포넌트들이 어떤 상태도 공유하면 안되는 다른 컴포넌트들이라고 전달합니다. key(userId로 설정된)가 변경될 때마다, React는 DOM을 재생성하고 Profile컴포넌트와 그 자식들의 [상태를 리셋]https://beta.reactjs.org/learn/preserving-and-resetting-state#option-2-resetting-state-with-a-key)합니다. 결과로 comment 필드는 프로필간에 이동시 자동으로 청소됩니다.

이 예시에서, 외부 ProfilePage컴포넌트만 export되고 프로젝트의 다른 파일들에서 가시적입니다. ProfilePage를 렌더링하는 컴포넌트들은 그 것에 key를 전달하지 않습니다: 그들은 userId를 일반 prop으로 전달합니다. ProfilePage가 내부 Profile 컴포넌트에 key로 전달하는 인자는 세부 구현사항입니다.

특정 상태를 prop이 변경될 때 조정하기​

때때로, 여러분은 prop의 변경시에 모두가 아닌 일부 상태를 조정하거나 리셋하고 싶을 수 있습니다.

이 List컴포넌트는 items 리스트를 prop으로 받고, selection 상태 변수에 선택된 아이템을 저장합니다. 여러분은 items prop이 다른 배열을 받을 때마다 selection을 null로 리셋하고 싶어합니다.

function List({ items }) {
  const [isReverse, setIsReverse] = useState(false);
  const [selection, setSelection] = useState(null);

  // 🔴 Avoid: Effect에서 prop 변경시에 상태를 지정하는 것
  useEffect(() => {
    setSelection(null);
  }, [items]);
  // ...
}

이는, 또, 이상적이지 않습니다. items가 변경될 때마다, List와 그 자식 컴포넌트들은 기존 selection 값으로 첫 렌더링됩니다. 그리고 React는 DOM을 업데이트하고, Effects를 실행합니다. 마지막으로 setSelection(null)은 List와 그 자식 컴포넌트들의 또 다른 재렌더링을 야기하고, 전체 프로세스를 재시작합니다.

Effect를 제거하는 것으로 시작합니다. 대신에, 상태를 렌더링동안 직접 조정합니다.

function List({ items }) {
  const [isReverse, setIsReverse] = useState(false);
  const [selection, setSelection] = useState(null);

  // Better: Adjust the state while rendering
  const [prevItems, setPrevItems] = useState(items);
  if (items !== prevItems) {
    setPrevItems(items);
    setSelection(null);
  }
  // ...
}

이와 같이 이전 렌더링에서 정보를 저장하는 것은 이해하기 어려울 수 있지만, Effect에서 같은 상태를 업데이트 하는 것보다 낫습니다. 위의 예제에서 setSelection은 렌더링동안 직접 호출됩니다. React는 List를 return 구문에서 그 것이 존재하자마자 즉시 리렌더링 합니다. 그 순간, React는 아직 List의 자식을 렌더링하거나 DOM을 업데이트하지 않았고, 이와 같이 List의 자식이 기존 selection 값으로 렌더링하지 않도록 합니다.

여러분이 렌더링동안 컴포넌트를 업데이트하면, React는 리턴된 JSX를 버리고 즉시 렌더링을 재시도합니다. 이를 아주 느린 단계적인 재시도를 피하기위해, React는 여러분이 렌더링동안 같은 컴포넌트의 상태만을 업데이트할 수 있도록합니다. 만약 렌더링동안 다른 상태를 업데이트하면, 여러분은 에러를 보게 될 것입니다. items!==prevItems와 같은 조건은 루프를 피하기위해 필수입니다. 이처럼 상태를 조정하게되지만, 다른 사이드이펙트들(DOM 변경 또는 타임아웃)은 이벤트 핸들러들이나 여러분의 컴포넌트를 예상가능하게 유지하기 위한 Effects에 남아있어야합니다.

이 패턴이 Effect보다 효율적이지만, 대부분의 컴포넌트들은 필요로 하지 않습니다. 여러분이 어떻게 하든, props나 다른 상태로 상태를 조정하는 것은 여러분의 데이터 플로우를 이해하거나 디버깅하기 어렵게합니다. 항상 대신에 모든 상태를 key로 리셋하거나, 렌더링 동안 모든 것을 계산할 수 있다는 것을 체크하세요. 예를들어 선택된 아이템을 저장(리셋)하는 것 대신에 여러분은 선택한 item ID를 저장할 수 있습니다.

function List({ items }) {
  const [isReverse, setIsReverse] = useState(false);
  const [selectedId, setSelectedId] = useState(null);
  // ✅ Best: 렌더링동안 모든 것을 계산하는 것
  const selection = items.find((item) => item.id === selectedId) ?? null;
  // ...
}

이제, 상태를 "조정"할 필요가 없어졌습니다. 만약 리스트 안에 선택된 ID의 아이템이 있다면, 선택된 채로 유지됩니다. 그렇지 않다면, 렌더링동안 계산된 selection은 매칭되는 아이템이 발견되지 않기 때문에 null일 것입니다. 이 동작은 조금 다르지만, items의 대부분의 변경이 selection을 보존하기 때문에 낫습니다. 하지만 여러분은 selection을 모든 로직에서 사용해야 하는데, 왜냐하면 selectedId를 가진 아이템이 존재하지 않을 수 있어서입니다.

이벤트 핸들러 간에 로직 공유하기​

여러분이 둘 다 상품을 구매하도록하는 두 버튼들(구매와 체크아웃)을 가지는 상품 페이지가 있다고 가정합시다. 여러분은 유저가 상품을 장바구니에 넣을 때, 알림을 보여주고 싶어합니다. 두 버튼의 클릭 핸들러에 showNotification() 호출을 추가하는 것은 반복적으로 느껴지기 때문에, 여러분은 로직을 Effect에 놓고 싶을 것입니다.

function ProductPage({ product, addToCart }) {
  // 🔴 Avoid: Event-specific logic inside an Effect
  useEffect(() => {
    if (product.isInCart) {
      showNotification(`Added ${product.name} to the shopping cart!`);
    }
  }, [product]);

  function handleBuyClick() {
    addToCart(product);
  }

  function handleCheckoutClick() {
    addToCart(product);
    navigateTo('/checkout');
  }
  // ...
}

이 Effect는 불필요합니다. 또한 대부분 버그들을 야기합니다. 예를 들어, 여러분의 앱이 페이지 리로드시에 쇼핑 카트를 "기억"한다고 가정해봅시다. 한번 카트에 상품을 추가하고 페이지를 새로고침하면, 알림이 새로 나타날 것입니다. 상품 페이지를 새로고침할때마다 알림이 나타날 것입니다. 이는 product.isInCart가 페이지 로드시에 이미 true이기 때문에, 위의 Effect가 showNotification()을 호출하는 것입니다.

어떤 코드가 Effect나 이벤트 핸들러 중 어디에 있어야 할지 확신이 없다면, 왜 이 코드가 실행되어야 하는지 스스로에게 질문하세요. Effects를 오직 컴포넌트가 유저에게 보여졌기 때문에 실행되는 코드에만 사용하세요. 이 예시에서, 알림은 유저가 페이지가 보여져서가 아니라, 버튼을 눌렀기때문에 나타나야합니다. Effect를 제거하고 공통된 로직을 두 이벤트 핸들러에서 호출하는 함수에 넣으세요.

function ProductPage({ product, addToCart }) {
  // ✅ Good: Event-specific logic is called from event handlers
  function buyProduct() {
    addToCart(product);
    showNotification(`Added ${product.name} to the shopping cart!`);
  }

  function handleBuyClick() {
    buyProduct();
  }

  function handleCheckoutClick() {
    buyProduct();
    navigateTo('/checkout');
  }
  // ...
}

둘 다 불필요한 Effect를 제거했고 버그를 수정하였습니다.

POST 요청 보내기​

이 Form 컴포넌트는 두 종류의 POST 요청을 보냅니다. 그것이 마운트 되었을 때 분석 이벤트를 보냅니다. 여러분이 form을 채우고 제출 버튼을 누르면, /api/register 엔드포인트에 POST요청을 보냅니다.

function Form() {
  const [firstName, setFirstName] = useState('');
  const [lastName, setLastName] = useState('');

  // ✅ Good: This logic should run because the component was displayed
  useEffect(() => {
    post('/analytics/event', { eventName: 'visit_form' });
  }, []);

  // 🔴 Avoid: Event-specific logic inside an Effect
  const [jsonToSubmit, setJsonToSubmit] = useState(null);
  useEffect(() => {
    if (jsonToSubmit !== null) {
      post('/api/register', jsonToSubmit);
    }
  }, [jsonToSubmit]);

  function handleSubmit(e) {
    e.preventDefault();
    setJsonToSubmit({ firstName, lastName });
  }
  // ...
}

위 예시에 같은 조건을 적용해봅시다.

분석 POST 요청은 Effect 내부에 있어야합니다. 분석 이벤트에 보내는 이유가 폼이 보여졌다는 것이기 때문입니다. (개발 환경에서는 두번 실행됩니다, 이에 대한 처리는 이 곳을 보세요)

하지만, /api/register POST 요청은 폼이 보여져서 발생되는 것이 아닙니다. 여러분은 하나의 특정 순간에 요청을 보내고 싶어합니다: 유저가 버튼을 클릭할 때 입니다. 이는 반드시 특정 상호작용에서 발생해야합니다. 두번째 Effect를 제거하고, POST요청을 이벤트 핸들러로 옮기세요.

function Form() {
  const [firstName, setFirstName] = useState('');
  const [lastName, setLastName] = useState('');

  // ✅ Good: This logic runs because the component was displayed
  useEffect(() => {
    post('/analytics/event', { eventName: 'visit_form' });
  }, []);

  function handleSubmit(e) {
    e.preventDefault();
    // ✅ Good: Event-specific logic is in the event handler
    post('/api/register', { firstName, lastName });
  }
  // ...
}

이벤트 핸들러 혹은 Effect에 어떤 로직을 넣을지를 선택할 때, 여러분이 답해야하는 메인 질문은 사용자의 관점에서 어떤 종류의 로직인가 입니다. 로직이 특정 인터렉션으로 발생한다면, 이벤트 핸들러에 유지하세요. 유저가 어떤 컴포넌트를 화면에서 보는 것으로 발생된다면 Effect에 유지하세요.

계산 체인들​

때때로 다른 상태를 기반으로 일부 상태를 조정하는 Effect들을 연결하고 싶은 유혹을 느낄 것입니다.

function Game() {
  const [card, setCard] = useState(null);
  const [goldCardCount, setGoldCardCount] = useState(0);
  const [round, setRound] = useState(1);
  const [isGameOver, setIsGameOver] = useState(false);

  // 🔴 Avoid: Chains of Effects that adjust the state solely to trigger each other
  useEffect(() => {
    if (card !== null && card.gold) {
      setGoldCardCount(c => c + 1);
    }
  }, [card]);

  useEffect(() => {
    if (goldCardCount > 3) {
      setRound(r => r + 1)
      setGoldCardCount(0);
    }
  }, [goldCardCount]);

  useEffect(() => {
    if (round > 5) {
      setIsGameOver(true);
    }
  }, [round]);

  useEffect(() => {
    alert('Good game!');
  }, [isGameOver]);

  function handlePlaceCard(nextCard) {
    if (isGameOver) {
      throw Error('Game already ended.');
    } else {
      setCard(nextCard);
    }
  }

  // ...

이 코드에는 두가지 문제점이 있습니다.

첫번째 문제는 매우 비효율적이라는 것입니다. 컴포넌트(그리고 그 자식들)는 각 set이 체인에서 호출될 때 리렌더링 될 것입니다. 위의 예시에서, 최악의 케이스(setCard->렌더링->setGoldCardCount->렌더링->setRound->렌더링->setIsGameOver->렌더링)에서 트리 아래서 세가지 불필요한 렌더링이 있습니다.

느리지 않더라도, 여러분의 코드가 발전함에 따라, 새로운 요구사항들은 여러분이 작성했던 "체인"이 맞지 않는다는 경우를 마주하게 될 것입니다.``여러분이 게임 이동들의 히스토리를 단계별로 살펴보는 방법을 추가한다고 가정합시다. 여러분은 과거의 값들로 각 상태 변수를 업데이트함으로서 할 수 있습니다. 하지만card` 상태를 과거 값으로 세팅하는 것은 Effect 체인을 트리거해서 여러분이 보여주는 데이터를 변경합니다. 이와 같은 코드는 융통성없고 취약합니다.

이 케이스에서, 렌더링 동안 할 수 있는 계산을 하는 것이 나으며, 상태를 이벤트 핸들러에서 조정하세요.

function Game() {
  const [card, setCard] = useState(null);
  const [goldCardCount, setGoldCardCount] = useState(0);
  const [round, setRound] = useState(1);

  // ✅ Calculate what you can during rendering
  const isGameOver = round > 5;

  function handlePlaceCard(nextCard) {
    if (isGameOver) {
      throw Error('Game already ended.');
    }

    // ✅ Calculate all the next state in the event handler
    setCard(nextCard);
    if (nextCard.gold) {
      if (goldCardCount <= 3) {
        setGoldCardCount(goldCardCount + 1);
      } else {
        setGoldCardCount(0);
        setRound(round + 1);
        if (round === 5) {
          alert('Good game!');
        }
      }
    }
  }

이는 훨씬 더 효율적입니다. 또한 만약 여러분이 게임 히스토리를 보이는 법을 구현한다면, 다른 모든 값을 조정하는 Effect 체인을 호출하지 않고 각 상태 변수를 과거의 움직임으로 설정할 수 있습니다. 만약 여러분이 여러 이벤트 핸들러간에 로직을 재사용해야한다면, 함수를 추출하고 해당 핸들러들에서 호출하세요.

이벤트 핸들러 내부에서는 상태가 스냅샷 처럼 동작함을 기억하세요. 예를 들어 여러분이 setRound(round+1)을 호출한 뒤에, round값은 유저가 버튼을 클릭한 시간의 값을 반영합니다. 하지만 여러분이 계산에서 계산을 위해 다음 값을 사용해야한다면, const nextRound = round + 1과 같이 수동적으로 정의하세요.

어떤 케이스들에서, 여러분은 이벤트 핸들러 내부에서 다음 상태를 직접 계산할 수 없습니다. 예를 들면, 여러 드랍다운을 가진 폼이 있고 다음 드랍다운의 옵션이 이전 드랍다운의 선택에 따라 달라지는 형태를 상상해봅시다. 이 경우, 여러분이 네트워크로 동기화하기 때문에 데이터를 가져오는 Effects 체인이 적당합니다.

애플리케이션 초기화 하기​

일부 로직은 전체 앱이 로드되는 동안 딱 한번만 실행되어야 합니다. 여러분은 아마 최상위 레벨 컴포넌트의 effect에 이를 위치시킬 것입니다.

function App() {
  // 🔴 effect에 딱 한번만 실행되어야 하는 로직을 넣는건 피해야 합니다.
  useEffect(() => {
    loadDataFromLocalStorage();
    checkAuthToken();
  }, []);
  // ...
}

하지만, 여러분은 이 로직이 개발 환경에서 두 번 실행된다는 것을 알아냈을 것입니다. 이 로직은 두 번 호출되는 것을 전제로 만들어지지 않았기 때문에 인증 토큰을 무효화 하는 등의 이슈를 만들 수도 있습니다. 일반적으로, 컴포넌트는 여러번 마운트 되는 것을 전제로 작성됩니다. App 컴포넌트와 같은 최상위 레벨 컴포넌트도 마찬가지입니다. 비록 프로덕션 환경에서는 다시 마운트 되지 않겠지만, 모든 컴포넌트에 같은 제약사항을 부여하는 것은 컴포넌트의 이동과 재사용을 용이하게 만들어 줍니다. 만약 일부 로직이 컴포넌트 마운트당 한 번이 아닌 전체 앱이 로드되는 동안 딱 한번만 실행되어야 한다면, 최상위 레벨에 변수를 하나 추가해 그 로직이 이미 실행되었는지를 판단하고, 다시 실행되는 일을 막을 수 있습니다.

let didInit = false;

function App() {
  useEffect(() => {
    if (!didInit) {
      didInit = true;
      // ✅ 앱 로드 동안 단 한번만 실행합니다.
      loadDataFromLocalStorage();
      checkAuthToken();
    }
  }, []);
  // ...
}

앱 렌더 이전에 모듈이 초기화 되는 단계에서 실행시킬 수도 있습니다.

if (typeof window !== 'undefined') { // 브라우저에서 동작중인지 확인합니다.
   // ✅ 앱 로드 동안 단 한범난 실행합니다.
  checkAuthToken();
  loadDataFromLocalStorage();
}

function App() {
  // ...
}

컴포넌트가 실제로 렌더링 되지 않더라도 컴포넌트가 불려와지는 동안 최상위 레벨의 코드는 딱 한번만 실행됩니다. 임의의 컴포넌트를 불러올 때 앱이 느려지거나 예상하지 못한 동작을 만들고 싶지 않다면, 이 패턴을 남용하지 마세요. 이런 범애플리케이션 단위 로직은 App.js같은 루트 컴포넌트나 애플리케이션의 진입점이 되는 모듈에서만 사용하도록 하세요.

부모 컴포넌트에게 상태 변화를 알려주기​

지금 우리가 Toggle 컴포넌트를 작성중이라고 가정해봅시다. 이 컴포넌트는 true 혹은 false 값을 가질 수 있는isOn을 내부 상태로 갖고 있습니다. 이 변수의 상태를 바꾸는 방법은 클릭이나 드래그 등 여러 방법이 있습니다. 우리가 원하는 것은 Toggle 컴포넌트의 내부 상태가 바뀌었을 때 이를 부모 컴포넌트에게 알려주는 것이므로 우리는 onChange 이벤트를 노출해 effect에서 호출하도록 합니다.

  // ...
}

이전과 같이, 이는 이상적인 구현은 아닙니다. Toggle 컴포넌트는 그가 가진 상태를 제일 먼저 업데이트 하고, React는 스크린을 업데이트 합니다. 그리고 React는 effect를 호출하는데, 여기서 부모 컴포넌트로부터 전달받은 onChange 함수가 호출됩니다. 이제 부모 컴포넌트는 자신의 상태를 업데이트하고, 또 다른 렌더 패스를 시작합니다. 한번에 모든 작업을 끝내는게 더 좋습니다.

Effect를 지우는 대신 같은 이벤트 핸들러에서 두 컴포넌트의 상태를 동시에 업데이트 해봅시다.

function Toggle({ onChange }) {
  const [isOn, setIsOn] = useState(false);

  function updateToggle(nextIsOn) {
    // ✅ 한 이벤트를 처리하며 모든 업데이트를 수행하기 때문에 좋습니다.
    setIsOn(nextIsOn);
    onChange(nextIsOn);
  }

  function handleClick() {
    updateToggle(!isOn);
  }

  function handleDragEnd(e) {
    if (isCloserToRightEdge(e)) {
      updateToggle(true);
    } else {
      updateToggle(false);
    }
  }

  // ...
}

이 접근 방법에서 Toggle 컴포넌트와 그의 부모 컴포넌트는 이벤트를 처리하는 과정에서 그들의 상태를 업데이트 합니다. React는 서로 다른 컴포넌트 변경 사항을 batch 형태로 업데이트 하기 때문에, 결과적으로 하나의 렌더 패스로 처리될 수 있습니다.

아니면 아예 상태를 다 지워버리고, 대신 부모 컴포넌트로부터 isOn을 내려받을 수도 있습니다.

// ✅ 이 방법 또한 컴포넌트가 전적으로 부모 컴포넌트에 의해 제어되기 때문에 좋습니다.
function Toggle({ isOn, onChange }) {
  function handleClick() {
    onChange(!isOn);
  }

  function handleDragEnd(e) {
    if (isCloserToRightEdge(e)) {
      onChange(true);
    } else {
      onChange(false);
    }
  }

  // ...
}

"상태를 위로 올리는 것"은 부모 컴포넌트로 하여금 자신의 상태로 Toggle 컴포넌트를 완전히 제어하게끔 해줍니다. 이는 부모 컴포넌트가 더 많은 로직을 갖게 되지만, 전체적으로 보면 우려되는 상태는 더 적어집니다. 만약 두개의 서로 다른 상태 변수들을 동기화해야 할 일이 있다면, 상태를 위로 올려야 할 수도 있다는 신호입니다.

부모 컴포넌트에게 데이터 전달하기​

Child 컴포넌트는 데이터를 fetch한 다음 effect 내부에서 Parent에게 전달하고 있습니다.

function Parent() {
  const [data, setData] = useState(null);
  // ...
  return <Child onFetched={setData} />;
}

function Child({ onFetched }) {
  const data = useSomeAPI();
  // 🔴 데이터를 effect 내에서 부모 컴포넌트에게 전달하는 것은 피해야 합니다.
  useEffect(() => {
    if (data) {
      onFetched(data);
    }
  }, [onFetched, data]);
  // ...
}

React에서 데이터는 부모로부터 자식으로 전달됩니다. 만약 스크린에서 무언가 잘못된 부분을 찾았다면, 컴포넌트 체인을 따라 올라가 어떤 컴포넌트가 잘못된 prop이나 상태를 전달해주는지 추적하면 됩니다. 자식 컴포넌트가 부모 컴포넌트의 상태를 effect 내부에서 변경하려 한다면 데이터 흐름을 추적하는 것은 매우 어려워질 것입니다. 부모 컴포넌트와 자식 컴포넌트 모두 같은 데이터를 필요로 하는 경우엔 부모 컴포넌트에서 데이터를 fetch하고 이를 자식 컴포넌트에게 내려주는 방식을 선택해야 합니다.

function Parent() {
  const data = useSomeAPI();
  // ...
  // ✅ 데이터를 자식 컴포넌트에게 내려주기 때문에 좋습니다.
  return <Child data={data} />;
}

function Child({ data }) {
  // ...
}

이렇게 하면 데이터 흐름을 더 예상 가능하도록 만들 수 있습니다. 데이터가 부모 컴포넌트로부터 자식 컴포넌트로 전달되도록 말이죠.

외부 저장소 구독하기​

때때로 컴포넌트는 써드 파티 라이브러리나 브라우저의 빌트인 API처럼 React 상태 외부에 있는 데이터를 구독해야 할 수도 있습니다. 이 데이터는 React의 관리로부터 벗어나있기 때문에 컴포넌트가 데이터를 구독하도록 수동으로 설정해줘야 합니다. 보통 이 작업은 effect 내에서 일어납니다.

function useOnlineStatus() {
  // effect 내에서 직접 저장소를 구독하는 방법은 이상적인 방법이 아닙니다.
  const [isOnline, setIsOnline] = useState(true);
  useEffect(() => {
    function updateState() {
      setIsOnline(navigator.onLine);
    }

    updateState();

    window.addEventListener('online', updateState);
    window.addEventListener('offline', updateState);
    return () => {
      window.removeEventListener('online', updateState);
      window.removeEventListener('offline', updateState);
    };
  }, []);
  return isOnline;
}

function ChatIndicator() {
  const isOnline = useOnlineStatus();
  // ...
}

이 컴포넌트는 외부 데이터 저장소를 구독하고 있습니다(이 예시에서는 브라우저의 navigator.onLine API 입니다). 이 API는 서버에 존재하지 않기 때문에(그래서 초기 HTML을 만드는데도 이용할 수 없습니다), 최초 상태는 true입니다. 그리고 브라우저에서 데이터 저장소에 저장된 값이 변경된다면, 컴포넌트도 그 상태를 업데이트 합니다.

이런 상황에서는 effect를 사용하는게 흔한 방법이지만, React는 effect를 쓰는 방법 대신 외부 저장소를 구독하기 위한 빌트인 훅을 제안하고 있습니다. Effect를 제거하고 useSyncExternalStore를 호출하는 방법으로 바꿔봅시다.

function subscribe(callback) {
  window.addEventListener('online', callback);
  window.addEventListener('offline', callback);
  return () => {
    window.removeEventListener('online', callback);
    window.removeEventListener('offline', callback);
  };
}

function useOnlineStatus() {
  // ✅ 내장된 훅으로 외부 저장소를 구독하기 때문에 좋습니다.
  return useSyncExternalStore(
    subscribe, // React는 같은 함수를 전달하는 한 저장소를 새로 구독하지 않습니다.
    () => navigator.onLine, // 클라이언트에서 값을 얻을 수 있는 방법
    () => true // 서버에서 값을 얻을 수 있는 방법
  );
}

function ChatIndicator() {
  const isOnline = useOnlineStatus();
  // ...
}

이 방법은 effect 내에서 수동으로 React state와 동기화 하는 방법보다 에러가 적게 발생합니다. 일반적으로 위처럼 useOnlineStatus() 커스텀 훅을 만들어 사용하면 같은 코드를 여러 컴포넌트에서 반복해서 작성할 필요가 없게 됩니다. 여기서 React 컴포넌트에서 외부 저장소를 구독하는 방법에 대해서 더 읽어볼 수 있습니다.

데이터 fetch 하기​

수많은 앱에서 데이터를 fetch 하는데 effect를 사용하고 있습니다. 이는 꽤 일반적인 데이터 fetch 방법으로, 아래와 같이 사용합니다.

function SearchResults({ query }) {
  const [results, setResults] = useState([]);
  const [page, setPage] = useState(1);

  useEffect(() => {
    // 🔴 cleanup 로직 없이 fetch 하는 것은 피해야 합니다.
    fetchResults(query, page).then(json => {
      setResults(json);
    });
  }, [query, page]);

  function handleNextPageClick() {
    setPage(page + 1);
  }
  // ...
}

Fetch 로직을 이벤트 핸들러로 옮길 필요는 없습니다.

이는 이벤트 핸들러에 로직을 넣어야 했던 이전의 예시들과는 모순되는 것처럼 보일 수도 있습니다. 하지만, 입력 이벤트를 처리할 때 주된 동작이 fetch를 하기 위함이 아님을 고려해야 합니다. 검색창에 입력된 값은 URL을 통해 미리 채워져 있고, 사용자는 아마 검색창을 건드리지 않고 뒤로가기 혹은 앞으로가기를 통해 탐색할 것입니다. page와 query가 어디서 오는 데이터인지는 중요하지 않습니다. 반면에 이 컴포넌트는 현재 사용자에게 보여지고 있고, 여러분은 results가 현재 page와 query에 맞춰 원격 데이터와 동기화되길 바랄 것입니다. 그래서 effect를 사용합니다.

하지만 위 코드는 버그가 있습니다. "hello"를 빠르게 입력한다고 가정해봅시다. 그럼 query는 "h"로부터 "he", "hel", "hell", "hello"로 변할 것입니다. 이는 각각의 fetch를 일으킬 것이지만, 응답이 순서대로 온다는 보장이 없습니다. 예를 들어 "hell"의 응답이 "hello"의 응답보다 늦게 도착한다고 가정해봅시다. 우리는 setResults()를 가장 마지막에 호출하기 때문에 화면에 보여지는 결과는 예상과는 전혀 다르게 될 것입니다. 우리는 이를 "race condition"이라고 부릅니다. 두 개의 서로 다른 요청이 서로 "경쟁"하며 예상과는 다른 순서로 보여질 수 있는 것입니다.

Race condition 문제를 해결하기 위해서는 clean up 함수를 통해 만료된 응답을 무시해주면 됩니다.

function SearchResults({ query }) {
  const [results, setResults] = useState([]);
  const [page, setPage] = useState(1);
  useEffect(() => {
    let ignore = false;
    fetchResults(query, page).then(json => {
      if (!ignore) {
        setResults(json);
      }
    });
    return () => {
      ignore = true;
    };
  }, [query, page]);

  function handleNextPageClick() {
    setPage(page + 1);
  }
  // ...
}

이렇게 하면 effect에서 데이터를 fetch할 때, 마지막 요청을 제외한 모든 요청에 대한 응답은 무시됩니다.

Race condtion을 제어하는 것은 데이터를 fetch할 때만 어려운 것이 아닙니다. 아마 여러분은 응답으로 오는 데이터를 어떻게 캐싱할지(유저가 뒤로가기 했을때 spinner가 보이지 않고 즉시 화면이 보이게 하기 위해), 어떻게 데이터를 서버에서 fetch 할지(서버에서 렌더링한 초기 HTML이 spinner 대신 fetch된 데이터를 보여주게 하기 위해), 그리고 어떻게 네트워크 워터폴을 피할지(자식 컴포넌트에서 데이터를 fetch할 때 부모 컴포넌트의 fetch가 끝나는것을 기다리지 않고도 먼저 fetch를 시작할 수 있도록 하기 위해)에 대해서도 고민할 수 있습니다. 이런 고민들은 React 뿐만이 아니라 다른 어떤 UI 라이브러리에도 적용되어 있습니다. 이 문제를 푸는 것은 쉽지 않고, 모던한 프레임워크들에서 컴포넌트의 effect에 직접 fetch 로직을 작성하는 것보다 더 효과적이고 빌트인된 데이터 fetch 매커니즘들을 소개하고 있습니다.

만약 이런 프레임워크들을 사용하지 않으면서 (그리고 그런걸 스스로 만들지 않으면서) 데이터 fetch 로직을 더 간편하게 만들고 싶다면, 아래 예시와 같이 fetch 로직을 추출해 커스텀 훅으로 만드는 것도 고려해보시기 바랍니다.

function SearchResults({ query }) {
  const [page, setPage] = useState(1);
  const params = new URLSearchParams({ query, page });
  const results = useData(`/api/search?${params}`);

  function handleNextPageClick() {
    setPage(page + 1);
  }
  // ...
}

function useData(url) {
  const [data, setData] = useState(null);
  useEffect(() => {
    let ignore = false;
    fetch(url)
      .then(response => response.json())
      .then(json => {
        if (!ignore) {
          setData(json);
        }
      });
    return () => {
      ignore = true;
    };
  }, [url]);
  return data;
}

아마 여러분은 에러 핸들링이나 컨텐츠의 로딩 상태를 추적하는 로직을 추가하길 원할 수도 있습니다. 그럼 이런 훅을 스스로 만들거나 React 생태계에 이미 존재하는 여러 해법들 중 하나를 골라 사용해볼 수도 있습니다. 이런 방법을 프레임워크의 내장 데이터 fetch 매커니즘을 사용하지 않고 단독으로 사용하면 덜 효과적일 수도 있지만, 데이터 fetch 로직을 커스텀 훅으로 이동시키면 이후에 다른 효과적인 데이터 fetch 전략을 도입하기 수월해질 수 있습니다.

일반적으로, effect를 이용한 코드를 작성해야 하는 경우에는 항상 로직들을 주시하고 추출해 useData같은 더 선언적이고 목적성을 가진 커스텀 훅을 만들어내는 것에 집중해야 합니다. useEffect를 직접 호출하는 횟수가 줄어들수록 애플리케이션을 유지하기 위한 노력이 적게 들 것입니다.

되짚어보기​

  • 만약 렌더링 도중에 계산이 가능하다면, effect를 쓰지 마세요.
  • 비용이 큰 연산을 캐시해야 한다면 useEffect 대신 useMemo를 사용하세요.
  • 컴포넌트 트리 전체를 리셋해야 한다면, 다른 key를 전달하세요.
  • prop 변경에 의존하는 특정 상태를 업데이트하려 한다면, 렌더링 중에 변경하세요.
  • 컴포넌트가 화면에 그려졌기 때문에 실행되어야 하는 코드는 effect에 있어야 하지만, 그 외의 코드는 이벤트 핸들러에 있어야 합니다.
  • 만약 하나의 상태에 대해 여러 컴포넌트가 업데이트 한다면, 하나의 이벤트 핸들러에서 업데이트 하는게 더 낫습니다.
  • 언제든 서로 다른 컴포넌트들에서 상태 변수를 동기화 하려 한다면, 상태를 상위 컴포넌트로 올리는 것을 고려해보세요.
  • 데이터를 effect 내에서 fetch 할 수 있지만, cleanup 함수를 작성해 race condition이 발생하지 않도록 해야 합니다.

이 코드에는 두 조각의 상태가 있습니다. todos 배열과 체크박스가 체크 되었는지를 나타내는 showActive 상태 변수입니다. 다른 모든 상태 변수들은 중복이고 렌더링 동안에 계산이 가능합니다. footer도 마찬가지입니다. 이를 둘러싼 JSX 안으로 이동시킬 수 있습니다.

2. Effect 없이 계산 결과 캐싱하기​

아래 예시에서 할 일들을 필터링 하는 로직은 getVisibleTodos()라는 분리된 함수로 추출되었습니다. 이 함수는 console.log()를 내부에 포함하고 있고, 이를 통해 함수가 호출되었는지 알 수 있습니다. "Show only active todos"를 껐다 켬으로써 getVisibleTodos()가 다시 호출되는 것을 확인해보세요. 이는 체크박스를 켜고 끄는 과정에서 화면에 그려질 할 일들이 변경되기 때문에 충분히 예상 가능합니다.

여러분이 할 일은 visibleTodos를 재계산하는 TodoList 컴포넌트의 effect를 제거하는 것입니다. 하지만 getVisibleTodos()는 입력창에 입력하는 과정에서 재실행되어서는 안됩니다(그리고 어떤 로그도 출력해서는 안됩니다).

힌트​

한가지 방법은 useMemo를 추가해 화면에 노출되는 할 일들을 캐싱하는 것입니다. 이보다 덜 명시적인 방법도 존재합니다.

정답​

상태 변수와 effect를 제거하고, 대신 useMemo를 호출해 getVisibleTodos()의 결과를 캐싱합니다.

이 변화로 인해 getVisibleTodos()는 todos나 showActive가 변경될 때만 다시 호출됩니다. 입력창에 입력하는 것은 text 상태 변수만 바꿀 뿐이고, 따라서 getVisibleTodos()를 다시 호출하지 않습니다.

useMemo를 사용하지 않는 또 다른 방법도 있습니다. text 상태 변수가 할 일들의 배열에 영향을 끼치지 못하기 때문에, 이 form을 추출해 NewTodo라는 분리된 컴포넌트를 만들면 됩니다. 그리고 text 상태 변수를 여기로 이동시킵니다.

이 접근 방식 또한 마찬가지로 요구사항을 만족시킵니다. 입력창에 입력을 하면 text 상태 변수만 업데이트 됩니다. text 상태 변수가 NewTodo 컴포넌트에 있기 때문에, 부모 컴포넌트인 TodoList 컴포넌트는 다시 렌더링되지 않을 것입니다. 이로 인해 getVisibleTodos()도 여러분의 입력에 따라 다시 호출되지 않습니다. (TodoList가 다른 이유로 인해 리렌더링 된다면 호출되긴 합니다.)

3. Effect 없이 상태 초기화 하기​

아래의 EditContact 컴포넌트는 { id, name, email } 형태의 연락처 객체를 savedContact prop으로 전달 받습니다. 이름과 이메일을 수정하고 저장 버튼을 누르면 form이 업데이트되며 수정된 이름으로 바뀝니다. Reset 버튼을 누르면, 작성중이던 모든 form의 변경사항들이 폐기됩니다. 이 UI를 잠시 조작하며 친숙해져봅시다.

상단의 버튼을 통해 연락처를 선택하면 form에는 연락처의 상세 정보가 반영됩니다. 이는 EditContact.js의 effect에서 일어납니다. 이 effect를 제거해봅시다. savedContact.id가 바뀌었을 때 form을 초기화 할 수 있는 다른 방법을 찾아보세요.

힌트​

만약 React에게 savedContact.id가 달라졌음을 알려줄 수 있다면 좋을 것입니다. EditContact의 form은 개념적으로 다른 연락처의 form이 되고, 상태를 유지하지 않게 될 것입니다. 떠오르는 방법이 있지 않나요?

정답​

EditContact 컴포넌트를 둘로 나눕니다. 모든 form의 상태를 내부의 EditForm 컴포넌트로 이동시키십니다. 외부의 EditContact 컴포넌트를 export 하고, EditContact 컴포넌트에 key로 savedContact.id를 전달합니다. 결과적으로 내부의 EditForm 컴포넌트는 다른 연락처를 선택했을 때 모든 form 상태를 초기화 하고 새로운 DOM을 그리게 됩니다.

4. Effect 없이 form 제출하기​

이 Form 컴포넌트는 친구에게 메시지를 보낼 수 있도록 해줍니다. form을 제출하면 showForm 상태 변수가 false로 변경됩니다. 이는 sendMessage(message)를 호출하는 effect를 유도하게 되고, 메시지를 보내게 됩니다(콘솔에서 확인할 수 있습니다). 메시지가 보내지면, "Open chat" 버튼과 함께 "Thank you" 대화창이 나타나고, 다시 form으로 돌아가게 됩니다.

이 앱의 사용자들은 메시지를 너무 많이 보냅니다. 채팅을 약간 더 어렵게 만들기 위해 여러분은 "Thank you" 메시지를 먼저 보여주고 그 다음 form을 보여주려 합니다. showForm 상태 변수를 false로 초기화 하지 않고 true로 초기화 합시다. 이 변경사항이 반영되는 즉시 콘솔은 빈 메시지가 보내졌다고 출력할 것입니다. 이 로직에 무언가 잘못된 부분이 있습니다!

이 문제의 원인은 무엇이고 어떻게 고칠 수 있을까요?

힌트​

메시지가 보내진 것은 유저가 "Thank you" 대화창을 봤기 때문일까요? 아니면 다른 원인이 있기 때문일까요?

정답​

showForm 상태 변수는 form을 보여줄지 "Thank you" 대화창을 보여줄지를 결정합니다. 하지만 메시지가 보내진 것은 "Thank you" 대화창이 보여졌기 때문이 아닙니다. 여러분은 사용자가 form을 제출했기 때문에 메시지가 전송되는 것을 원합니다. 잘못 연결된 effect를 제거하고 sendMessage를 handleSubmit 이벤트 핸들러 내부로 이동시키세요.

이 코드에서 오직 form을 제출하는 것(이벤트)만이 메시지를 보낸다는 것을 확인하세요. 이는 showForm의 초기값이 true일때나 false일때나 상관 없이 모두 잘 동작합니다. (false로 바꿔도 콘솔에 아무 메시지가 찍히지 않음을 확인하세요.)
